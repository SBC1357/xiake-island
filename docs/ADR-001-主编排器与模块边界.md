# ADR-001: 采用主编排器 + 模块化单体作为首阶段架构

> **日期**: 2026-03-13
> **状态**: Accepted
> **适用项目**: `D:\汇度编辑部1\侠客岛`

## Context

`侠客岛` 被定义为一个独立的中文医疗内容能力平台，而不是对现有写作流水线的继续加长。

当前约束和背景如下：

- 现有 `V2` 仍承担一般任务，短期内不能直接停用
- 目标不是只跑通整篇文章链路，而是让模块能力可独立调用、可被工作流编排、可被其他产品复用
- 当前最适合优先产品化的模块是 `opinion` 和 `semantic_review`
- 如果一开始继续沿用单条流水线，模块复用和外部调用仍然会很别扭
- 如果一开始直接拆成多服务，复杂度会过早转移到网络、部署、版本兼容和运维层
- 如果让 `Orchestrator` 同时掌握所有模块的内部 inventory 明细，编排层会迅速膨胀为新的黑箱，重复 `V2` 式的大超市问题

因此，首阶段需要先回答一个核心问题：

`侠客岛` 应该先按什么架构形态落地，才能同时兼顾独立运行、模块复用、可编排性和迁移风险控制。

## Decision

首阶段采用：

**主编排器 + 模块化单体**

作为 `侠客岛` 的基础架构形态。

具体决定如下：

1. `侠客岛` 作为独立项目建设，不在 `V2` 目录内继续扩流水线。
2. 首阶段先保持单体服务形态，不先拆分为多个独立部署服务。
3. 系统高层结构采用：
   - `API Layer`
   - `Orchestrator`
   - `Contracts Registry`
   - `Opinion Module`
   - `Semantic Review Module`
   - `Asset Bridge`
   - `LLM Gateway`
   - `Task Log / Audit Log`（代码层当前由 `runtime_logging` 目录承载）
4. 每个业务模块必须满足两条边界：
   - 能被单独调用
   - 能被主编排器作为工作流节点调用
5. `Orchestrator` 只负责编排，不承载模块内部业务策略。
6. `Contracts Registry` 作为模块结构边界的统一真相源，contract 不再只存在于 Markdown 约定里。
7. `Asset Bridge` 负责只读桥接外部资产，不把旧系统目录结构直接泄漏进核心模块。
8. 过渡期内 `V2` 继续运行；如某项能力正式迁入 `侠客岛`，必须明确唯一真相源归属，禁止双写。
9. 如后续引入 `Agent Teams`，只能作为 `Orchestrator` 上的编排配方，不作为首阶段基础架构。
10. 不是所有预设能力名称都应立即提升为一级模块；模块是否独立，应以复用价值、schema 稳定度、独立调用价值、日志审计价值和 owner 清晰度为准。
11. `Orchestrator` 只掌握能力级目录、contract 边界和模块状态，不掌握各模块内部 inventory 明细；inventory 归各模块 owner 管理。

## Consequences

### Positive

- 模块边界会比流水线式工程更清楚，便于单独调用和后续复用
- 可以先在单体内部把 contract、日志和模块职责做扎实，再决定是否拆服务
- 有利于让 `opinion` 和 `semantic_review` 成为首批标准化能力样板
- 可以在不关闭 `V2` 的前提下渐进迁移，降低切换风险

### Negative

- 首阶段不会立即获得天然的服务隔离和独立扩缩容能力
- 主编排器、Bridge、contract 和日志体系需要先搭出最小基础设施
- 过渡期仍要处理 `V2` 与 `侠客岛` 之间的边界管理成本

### Neutral

- 是否进一步拆分为多服务，不在本 ADR 中一次性锁死
- 首阶段仍允许桥接旧资产，但这只是过渡安排，不等于长期必须保留外部依赖

## Boundary Rules

为避免后续施工时架构回退，首阶段先写死以下边界：

1. 不允许把 `Orchestrator` 变成新的“大黑箱业务模块”。
2. 不允许把模块输入输出只写在提示词或 Markdown 里，而不落 schema。
3. 不允许把 `侠客岛` 变成 `V2` 的目录镜像或第二运行副本。
4. 不允许在尚未跑稳首批模块前，提前扩写为全链路写作平台。
5. 不允许在未证明必要性前，提前拆成多服务架构。
6. 不允许把自由对话式多 agent 协商当成核心运行机制；跨节点传递必须可 schema 校验、可审计。
7. 不允许因为预先列了多个能力名字，就在首阶段一次性把它们全部解耦成正式模块。
8. 不允许把所有规则、模板、证据索引和资产清单重新汇总到 `Orchestrator`，把编排层做成新的大超市。

## Alternatives Considered

### 方案 A：继续扩现有流水线

未采用原因：

- 模块仍会被包在整链里，单独调用和外部复用成本高
- 业务规则继续散落在会话逻辑和文档约定里，不利于产品化
- 无法从结构上解决“模块可独立调用”和“统一 API”这两个目标

### 方案 B：首阶段直接拆成多个独立服务

未采用原因：

- 当前还处于边界澄清阶段，过早服务化会把问题转成部署和协同复杂度
- 模块 contract、日志和错误模型还未稳定，直接拆服务会放大接口漂移风险
- 对首阶段目标来说，收益不如“先做模块化单体”直接

### 方案 C：完整复制 `V2` 再逐步改造

未采用原因：

- 极易制造双真相源
- 会把旧系统耦合和目录结构一起复制进新项目
- 会模糊 `侠客岛` 作为独立项目的边界
