# 侠客岛 SP-6 核心去黑箱与正式交付目标与要求

> **日期**: 2026-03-19
> **文档类型**: 目标与要求 / 新阶段约束文件
> **状态**: 草案，待执行
> **适用时点**: `SP-5` 验收通过后

---

## 一、文档定位

本文档只定义 `SP-6` 的目标、边界和硬要求。

本文档不是：
- 当前 `SP-0 ~ SP-5` 主执行单的替代品
- 现有长线程的回写修订稿
- 逐步实施计划或逐日排期

本文档解决的是：
1. 把 `SP-5` 之后必须补做的核心去黑箱工作讲清楚
2. 把“什么才算真正独立下来”讲清楚
3. 把正式交付物与终版字数门禁讲清楚

`SP-6` 的任务不是继续补外围界面，而是把核心链条中仍然依赖隐式逻辑、模型自述或中间稿假闭环的部分，收回到侠客岛自身。

---

## 二、当前已核实的核心缺口

截至本文件编写时，以下缺口已直接核实：

### 2.1 取证与证据决策链仍未完全脱黑箱

当前系统已经能按产品和领域读取证据事实，但还没有形成显式、可回看的证据决策链。

当前缺少的不是“有没有事实列表”，而是：
- 为什么查这些
- 为什么没查那些
- 候选证据如何排序
- 重复和冲突如何处理
- 证据是否足以支撑当前稿件
- 最终为什么选中这一批证据

换句话说，当前更像“能取到事实”，还不是“有自己的显式取证编排器”。

### 2.2 规则库仍主要停留在知识文件层，而非执行层

当前系统已经具备规则资产、语义审核输出结构和部分质量门禁，但规则尚未全面转化为程序化执行结果。

当前最核心的问题是：
- 规则文件存在，不等于规则已经在主链上执行
- 模型输出 `matched_rules` / `unmatched_rules`，不等于程序真的执行过这些规则
- 没有 rule-level trace，就不能宣称语义规则真正接入

### 2.3 写作主链仍偏向“提示包可见”，不是“正文约束可见”

当前写作链已经把规划转成写作输入，但正文约束还没有完全显式化。

当前仍不够透明的部分包括：
- 段落如何绑定证据
- 哪些句式和风格约束真的进了正文生产
- 哪些问题来自确定性规则，哪些来自模型判断
- 改写后哪些门禁必须重跑

### 2.4 正式交付物与最终字数门禁尚未闭环

当前交付链已经能生成 Markdown，但 Markdown 只能算中间稿，不是最终正式交付物。

当前仍未闭合的部分包括：
- 还没有 `docx` 终交付链
- 还没有基于 Python 的 `docx` 生成能力
- 还没有以后置 `docx` 正文为准的最终字数门禁
- 当前 Markdown 阶段的长度检查不能代表正式稿达标

---

## 三、SP-6 总目标

`SP-6` 完成后，侠客岛必须从“外围较完整、核心仍半黑箱”的状态，推进到“核心链条可解释、可追踪、可执行、可正式交付”的状态。

`SP-6` 要达成的不是更多页面或更多包装，而是以下 4 件核心事实：

1. 取证链必须可解释
2. 规则链必须可执行
3. 写作与审核链必须可追踪
4. 正式交付链必须以 `docx` 和终版字数门禁为准

在 `SP-6` 完成前，不应宣称：
- 侠客岛已经真正摆脱核心黑箱
- 中文语义规则已经真正接入主链
- 交付已经符合正式编辑要求
- 字数门禁已经真实有效

---

## 四、SP-6 硬要求

### 4.1 显式取证与数据编排要求

`SP-6` 必须把取证链从“事实读取”提升为“显式证据决策”。

最低要求如下：

1. 必须显式记录查询输入
   - 产品
   - 领域
   - 受众
   - register
   - 项目目标
   - 交付类型

2. 必须显式记录候选召回结果
   - 候选事实
   - 候选来源
   - 候选数量
   - 候选筛选前后变化

3. 必须显式记录决策链
   - 排序规则
   - 去重规则
   - 冲突消解规则
   - 证据充分性判断
   - 最终入选原因

4. 必须产出 retrieval/query trace
   - 不能只保留最终事实列表
   - 必须能回看为什么选了这批证据

5. 不接受以下假完成：
   - 只按 `product_id` / `domain` 过滤事实文件
   - 只把“查到了哪些事实”打印出来
   - 只有结果，没有决策过程

### 4.2 规则执行引擎要求

`SP-6` 必须把规则从知识文件转成程序可执行结果。

最低要求如下：

1. 规则必须可加载、可解析、可执行
2. 规则执行结果必须落到程序输出，而不是只存在模型口头描述
3. 必须产出 rule-level trace，至少包含：
   - matched rule ids
   - failed rule ids
   - skipped rule ids
   - failure reason
4. LLM 只能承担：
   - 补充发现
   - 解释性说明
   - 改写建议
5. LLM 不能承担：
   - 规则是否命中这一事实的最终裁决
   - 规则是否通过的唯一依据
   - 合规阻断的唯一判定

### 4.3 规则族接入要求

以下规则族在 `SP-6` 中必须有明确作用场景，不得只作为目录存在：

| 规则族 | 必须作用阶段 | 最低要求 |
|--------|--------------|----------|
| `register_levels` | 规划、写作、审核 | 不同 register 必须对应不同强度的语体约束 |
| `native_syntax_rules` | 写作后审核、改写再审 | 通用中文语法戒律必须可执行，不得只交给模型自由发挥 |
| `expression_base` | 写作前约束、写作后反查 | 通用过渡表达和禁用表达必须进入检查链 |
| `style_dna_schema` | 规划、写作 | 文风结构必须能约束正文生成，不得只保留 schema |
| `m0_persona_templates` | 规划、写作 | 人格模板必须影响论证视角和关注顺序 |
| `medical_expression_profiles` | 规划、写作、审核 | 不同受众的医学表达模板必须进入正文约束 |
| `medical_syntax_rules` | 审核、终审阻断 | 医学术语、具名归因、临床落地结尾必须程序化检查 |
| `sentence_patterns` | 写作 | 句式模板必须参与正文结构或段落生产，不得闲置 |
| `m5_compliance_schema` | 质量门禁、交付前阻断 | 合规锚点必须可阻断，不得只作为参考文档 |
| `thesis_derivation_rules` 与产品级 overlay | 规划、写作、合规 | 论点生成和产品级风格/合规约束必须显式落地 |

### 4.4 写作主链去黑箱要求

`SP-6` 不接受“写作层只有提示包可见、正文规则不可见”的状态。

最低要求如下：

1. 规划结果必须显式约束正文
   - 核心论点
   - 章节结构
   - 目标受众
   - register
   - 风格约束
   - 目标字数

2. 正文必须显式绑定证据
   - 段落与证据锚点的关系必须可追踪
   - 不能只看到最终文案，看不到证据绑定

3. 写作阶段必须能说明：
   - 用了哪些规则
   - 用了哪些风格模板
   - 用了哪些证据锚点
   - 哪些约束属于硬性，哪些属于建议

4. 不接受以下假完成：
   - 只生成更长的 prompt
   - 只增加更多 metadata
   - 只在结果里追加一句“已参考规则库”

### 4.5 审核主链去黑箱要求

`SP-6` 必须把审核链拆成三个明确层次：

1. 确定性规则层
   - 用于执行可程序化的语法、风格、合规和锚点规则

2. 模型审校层
   - 用于发现规则层难以完全覆盖的问题
   - 只能在规则层之后运行

3. 改写建议层
   - 用于提出修订目标和修订方向

同时必须明确：
- 哪些问题属于可自动改写
- 哪些问题属于必须阻断
- 哪些问题改写后必须重跑全部 gate
- 哪些问题只需重跑局部 gate

没有这套分层，就不能宣称审核链真正拆箱。

### 4.6 正式交付要求

`SP-6` 必须把正式交付物定义为 `docx`，而不是 `md`。

最低要求如下：

1. `md` 只作为中间审核稿存在
2. 审核通过后，必须使用 Python 生成 `docx`
3. 最终对外交付物必须是 `docx`
4. 不接受以下假完成：
   - 只有 `md`
   - `docx` 只是人工另存
   - `docx` 不是由主链生成

### 4.7 最终字数门禁要求

`SP-6` 必须把最终字数门禁以后置 `docx` 正文为准。

字数门禁的默认口径在 `SP-6` 中必须明确为：

**计入最终字数：**
- 导语
- 正文段落
- 正文内小标题
- 正文内项目列表或编号列表的文本

**不计入最终字数：**
- 封面标题
- 目录
- 页眉页脚
- 表格单元格文本
- 图片说明
- 脚注与尾注
- 附录
- 参考文献清单
- 系统元数据

同时必须明确：

1. Markdown 阶段的长度检查只能算预检
2. 只有 `docx` 正文字数达标，才能算最终达标
3. 如果 `docx` 字数不达标：
   - 不能因为 Markdown 通过就放行
   - 必须回写 fix batch
   - 必须重新生成 `docx`
   - 必须重跑最终字数门禁

### 4.8 不得偷换的结论

在 `SP-6` 中，以下结论一律禁止偷换：

| 错误说法 | 正确要求 |
|----------|----------|
| 规则已接入 | 规则已程序执行并产出 rule-level trace |
| 语义审核已使用规则库 | 审核链中有确定性规则执行结果，且能区分规则层与模型层 |
| 取证已透明 | 有完整 retrieval/query trace 和证据决策链 |
| 交付已完成 | 已生成并验收最终 `docx` 产物 |
| 字数达标 | 已按最终 `docx` 正文口径验收通过 |
| 侠客岛已真正独立 | 核心取证、规则、正文、终交付都已脱离隐式黑箱 |

---

## 五、SP-6 完成态定义

只有在以下条件同时满足时，`SP-6` 才能被描述为“闭合”：

1. 取证链有显式决策 trace
2. 规则链有 rule-level trace
3. 写作链能说明正文如何受规划、证据和规则共同约束
4. 审核链能区分规则层、模型层、改写层
5. 最终交付物为 `docx`
6. 最终字数门禁以后置 `docx` 正文为准
7. 不再依赖“模型自称参考了规则”来替代规则执行证据

若只完成其中一部分，应表述为：
- `已实现`
- `已验证`
- `部分闭合`

不得直接表述为：
- `已完全拆箱`
- `已真正独立`
- `已正式交付闭环`

## 六、Evidence Summary

1. 已直接核对当前语义审核实现：现有审核链会把内容、受众、原型提示、register 组装为模型提示，再解析模型返回的 JSON；当前代码路径未体现对规则资产的程序化读取与执行。
2. 已直接核对当前交付与依赖状态：现有交付链输出为 Markdown 文件，当前项目依赖中未发现 `python-docx` 或其他现成 `docx` 生成依赖，因此正式 `docx` 交付链尚未落地。
3. 已直接核对当前质量与取证实现：现有内容检查仍以基础检查和长度检查为主，现有证据查询逻辑仍以按产品、领域、键值过滤事实为主，尚未形成显式 retrieval/query trace 与证据决策链。
