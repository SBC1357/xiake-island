## 0. 文档定位

### 0.1 长线程定位

本文档是 **侠客岛主线长线程执行单**，不是短期一次性交付单。

**目标不是**：只完成工作台和质量闭环就结束。

**目标是**：把侠客岛逐步建设成 **"编辑部可用的一期产品"**。

**关键区分**：
- "知识底座已接通" ≠ "V2 全能力已接入"
- "模块代码存在" ≠ "产品入口已落地"
- "能跑通 seam" ≠ "真正接入完成"

### 0.2 三条主线

本文档按三条主线推进，每条主线分 batch 执行：

| 主线 | 代号 | 核心目标 | 当前阶段 |
|------|------|----------|----------|
| **A线：编辑部工作台与质量闭环** | A | 任务历史、详情、再生成、审查反馈、日志契约、持久化、审计链 | ✅ A-0~A-8 全部完成 |

### 0.3 状态分类标准

## 4. 长线程执行规则

### 4.1 Batch 执行规则

| 规则 | 内容 |
|------|------|
| **R1** | 每完成一个 batch，必须先写 checkpoint |
| **R2** | 每个 batch 后必须双轮自审 |
| **R3** | 有未修复 P1/P2 时，不得进入下一 batch |
| **R4** | 只有当前 batch 收口后，才能继续 |
| **R5** | 不允许自由扩 scope |
| **R6** | 不允许把"知识底座已接通"表述成"V2 全能力已接入" |
| **R7** | 不允许把"模块存在"表述成"产品入口已落地" |

### 4.2 启动/停止规则

**允许启动下一 batch 的条件**：
1. 当前 batch checkpoint 已写
2. 双轮自审无 P1/P2
3. 验证命令全部通过
4. 用户显式授权或上游方案通过

**必须停止并汇报的情况**：
1. 发现未修复 P1
2. 执行单与仓库现状冲突
3. 验证无法成立
4. 发现遗漏的前置依赖

**视为 batch 完成的条件**：
1. 所有"必须完成"项已实现
2. 验证命令通过
3. 测试覆盖
4. checkpoint 已写
5. 双轮自审通过

**只能算"实现 seam"，不能算"真正接入完成"的情况**：
1. 后端 API 存在但前端无入口
2. 模块代码存在但未接入业务流程
3. 能跑通但编辑人员无法理解操作
