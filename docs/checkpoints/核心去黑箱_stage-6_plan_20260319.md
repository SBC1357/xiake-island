## 1. 已核实缺口

### 1.1 取证链仍是“过滤事实”，不是“显式决策”

- `src/modules/evidence/service.py` 当前只做：
  - 加载证据文件
  - 按 `product_id / domain / fact_keys / status` 过滤
  - 直接返回 `FactRecord`
- 当前没有：
  - 候选召回前后数量
  - 排序 / 去重 / 冲突消解过程
  - 证据充分性判断
  - retrieval/query trace

### 1.2 规则执行层尚未落地

- 当前仓库中可直接命中的规则资产只有 `src/contracts/m5_compliance_schema.py`
- 本地检索未发现 `register_levels / expression_base / style_dna_schema / sentence_patterns / thesis_derivation_rules` 等可执行规则资产文件
- `src/modules/semantic_review/reviewer.py` 当前仍以 LLM JSON 返回的 `matched_rules / unmatched_rules` 作为输出来源，不是程序执行 trace

### 1.3 写作主链仍以 prompt 可见为主

- `src/modules/writing/service.py` 当前只把 thesis / outline / style_notes / evidence facts 拼到 prompt
- 没有正文级证据锚点绑定结果
- 没有写作时使用了哪些规则 / 风格模板 / 约束层级的显式 trace

### 1.4 交付仍只有 Markdown

- `src/modules/delivery/service.py` 当前只有 `MarkdownWriter`
- `pyproject.toml` 当前依赖中没有 `python-docx`
- 当前没有 `docx` 终交付，也没有基于 `docx` 正文的最终字数门禁

## 2. SP-6 分批实施方案

### Batch 6A：显式取证 / 数据编排去黑箱

目标：
- 把 Evidence 从“事实过滤”升级为“可回看决策链”

实施范围：
- 新增 retrieval/query trace 数据模型
- 为候选召回、筛选、排序、去重、最终入选原因建立结构化 trace
- 在标准五段链的 evidence / planning 结果中回传 trace

最低验收：
- 有 focused tests 证明 trace 会记录：
  - 查询输入
  - 候选数
  - 筛选后数量
  - 最终入选原因
- 标准链结果或子任务结果中能回读 retrieval/query trace

### Batch 6B：规则执行引擎落地

目标：
- 把规则从知识描述转成确定性程序执行结果

实施范围：
- 新增规则资产加载与执行引擎
- 先覆盖 `register_levels / expression_base / medical_syntax_rules / m5_compliance_schema / thesis_derivation_rules`
- 产出 rule-level trace：matched / failed / skipped / reason

最低验收：
- focused tests 证明规则可加载、可执行、可输出 rule-level trace
- 语义审核链能区分规则层和模型层

### Batch 6C：写作 / 审核主链去黑箱

目标：
- 正文生产和审核结果能回溯到规划、证据和规则

实施范围：
- 写作结果补充段落级证据锚点 / 约束 trace
- 审核链拆成：
  - 确定性规则层
  - 模型审校层
  - 改写建议层
- 明确哪些问题必须全量重跑 gate，哪些只需局部重跑

最低验收：
- focused tests 证明正文 trace 可回读
- 审核输出可区分规则层与模型层

### Batch 6D：`docx` 正式交付与终版字数门禁

目标：
- 让正式交付物从 Markdown 升级为主链生成的 `docx`

实施范围：
- 引入 `python-docx`
- 新增 `docx` writer
- 新增最终正文提取与字数统计逻辑
- 让最终 gate 以后置 `docx` 正文为准

最低验收：
- focused tests 证明：
  - 主链能产出 `docx`
  - 能按 `docx` 正文口径统计字数
  - `docx` 不达标会阻断交付

## 3. 执行顺序

1. `Batch 6A`：先把 retrieval/query trace 和显式决策链立住
2. `Batch 6B`：再把规则执行引擎接进主链
3. `Batch 6C`：再做写作 / 审核层拆箱
4. `Batch 6D`：最后接 `docx` 与终版字数门禁

## 4. 第一批立即执行范围

本轮紧接规划后，立即执行 `Batch 6A`：

- 只围绕“显式取证 / retrieval/query trace”
- 不提前扩到 `docx`
- 不把规则执行、写作 trace、`docx` 同时混做
- `Batch 6A` 验收后再进入 `Batch 6B`
