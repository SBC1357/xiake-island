# 侠客岛介绍

- 文档类型：产品介绍 / 前端设计对齐文档
- 适用对象：前端设计、产品讨论、前后端对接
- 当前状态：用于设计对齐
- 形成日期：2026-03-14

## 0. 文档目的

本文不是接口文档，也不是施工图纸。

本文的目的，是在进入前端设计时，把下面几件事讲清楚：

1. `侠客岛` 到底是什么。
2. `侠客岛` 当前阶段到底做到哪里。
3. `侠客岛` 面向谁。
4. 为什么“开发者界面”和“使用者界面”不应该混在一起。
5. 使用者界面应该按什么逻辑组织。
6. 开发者界面应该按什么逻辑组织。

如果这几个问题不先讲清楚，前端设计很容易一上来就把它画成“一个什么都有的后台”，最终同时误伤产品边界、交互复杂度和后端接口范围。

- 已核对 `src/api/app.py`、`src/api/routes/opinion.py`、`src/api/routes/semantic_review.py`、`src/api/routes/workflow.py`，当前对外暴露的核心执行入口只有 3 个：`/v1/opinion/generate`、`/v1/review/semantic`、`/v1/workflow/article`。
- 已核对 `src/runtime_logging/task_logger.py` 与 `src/runtime_logging/memory_store.py`，当前有任务追踪能力，但没有独立任务查询 HTTP 路由，也没有前端配置中心能力；因此本文不会把 `侠客岛` 描述成完整后台平台。

## 1. 一句话理解侠客岛

`侠客岛` 是一个面向中文医疗内容生产的能力平台。

它不是单篇文章生成器，也不是一个只能靠对话跑通的 AI 工作流壳。

更准确地说，它是：

**模块化内容能力底座 + 主编排器 + 对外 API 能力。**

它当前最重要的事情，不是把所有事情都做完，而是先把几项高价值能力产品化、模块化、可调用化。

## 2. 侠客岛是什么

从系统角度看，`侠客岛` 当前是一个独立的中文医疗内容能力平台，目标是把内容生产过程中的关键能力拆成可以独立调用、可以被编排、可以被审计的模块。

它希望解决的问题不是“怎么再多生成一点字”，而是：

1. 把黑箱式整链流程拆开。
2. 让单模块可以单独被别的产品调用。
3. 让工作流编排不再等于把所有逻辑都塞进一个大 prompt。
4. 让输入、输出、日志、错误、追踪都能结构化。
5. 让这套能力未来能被别的系统复用，而不是永远只能在一个单一写作流程里使用。

所以，`侠客岛` 的本质不是“一个页面”，而是“一个能力平台”。前端只是这个平台的某种入口，不是平台本身。

## 3. 侠客岛不是什么

为了防止前端设计偏航，下面这些误解需要直接排除：

### 3.1 它不是单一写作工具

`侠客岛` 不是“输入主题 -> 输出整篇文章”的单一成稿工具。

它当前至少已经明确有三种能力形态：

- 单模块观点生成
- 单模块语义审核
- 最小文章工作流

也就是说，它更像“能力中心”，而不是“单按钮写稿器”。

### 3.2 它不是完整自动化平台

当前 `侠客岛` 没有独立的触发配置中心、规则引擎、定时调度系统、任务大厅或运营后台。

因此它不应该被前端画成：

- 调度后台
- 规则中心
- 低代码流程台
- 运营策略总控台

### 3.3 它不是给开发者看的 Python 工程外壳

虽然当前仓库里已经有 `src/modules`、`src/orchestrator`、`src/runtime_logging` 等目录，但前端不能把这些内部结构原样抬上界面。

对使用者来说，`侠客岛` 不应该长得像“Python 包导航器”。

## 4. 当前阶段的真实范围

理解 `侠客岛` 前端，必须接受一个现实：当前它是“能力平台的首阶段”，不是完整平台终态。

当前已经明确的能力边界是：

1. `opinion`
2. `semantic_review`
3. `article` 最小工作流

当前已经明确的系统结构是：

1. API Layer
2. Orchestrator
3. Opinion Module
4. Semantic Review Module
5. Asset Bridge
6. LLM Gateway
7. Task Log / Audit Log 的最小追踪基础

这意味着前端设计必须接受两个现实：

1. 使用者入口可以成立，因为已经有执行 API。
2. 完整开发者后台还不能被描述成“已具备全部平台管理能力”，因为很多管理型能力并不存在。

## 5. 谁在使用侠客岛

当前可以把 `侠客岛` 的核心角色拆成两类：

### 5.1 使用者

这里的“使用者”，不是泛指所有接触系统的人，而是直接来完成内容任务的人。

典型包括：

- 内容编辑
- 医学编辑
- 内容运营人员
- 需要手动发起某个内容能力的人

他们最关心的问题是：

- 我现在要做什么任务
- 该填什么参数
- 点哪里开始
- 结果怎么看
- 失败了我该怎么办

他们不关心：

- Pydantic 模型怎么定义
- Orchestrator 内部怎么拼接
- Python 模块目录怎么组织
- task logger 怎么存的

### 5.2 开发者

这里的“开发者”，是负责接入、维护、扩展、调试、联调或平台建设的人。

典型包括：

- 前端开发
- 后端开发
- 工具接入开发者
- 平台维护人员
- 做联调和排障的人

他们最关心的问题是：

- 有哪些接口
- 请求响应契约是什么
- 哪些字段是正式字段
- 错误怎么返回
- 哪个任务为什么失败
- 当前环境怎么配
- 哪些能力是一期、哪些是后续

他们不需要一个“傻瓜式任务入口”，他们需要的是“可理解、可调试、可对接”的系统视图。

## 6. 为什么开发者和使用者必须是不同的界面

这是前端设计最需要先写死的一条原则：

**开发者界面和使用者界面，不是权限不同，而是心智模型不同。**

两类人的任务完全不一样：

- 使用者的目标是完成任务
- 开发者的目标是理解系统、接入系统、调试系统

如果把两类逻辑混在一个界面里，会出现四个问题：

1. 使用者会看到不该看到的技术细节，页面会变重。
2. 开发者会拿不到真正需要的调试与契约信息。
3. 产品设计会在“业务语言”和“技术语言”之间来回摇摆。
4. 一期很容易被误画成“一个什么都有的大后台”。

所以，这里推荐的不是“同一个页面按权限隐藏”，而是“同一个产品下的两种界面逻辑”。

## 7. 三种界面组织方案对比

### 7.1 方案 A：一套混合界面，所有人共用

不推荐。

这种方案看上去省事，但很快会出现：

- 使用者嫌复杂
- 开发者嫌不够深
- 产品边界被后台感吞掉

### 7.2 方案 B：同一产品下，分成使用者界面和开发者界面

推荐。

这个方案的好处是：

- 品牌还是同一个 `侠客岛`
- 后端能力还是同一套 API 基础
- 但入口、导航、术语、信息密度完全分开

这最适合当前阶段。

### 7.3 方案 C：完全拆成两个独立前端产品

可以作为后续工程组织方式，但不必在产品定义上提前锁死。

也就是说：

- 逻辑上要分开
- 代码仓、构建、部署是否分开，可以后定

这和之前“一期允许独立分包建设”的边界是一致的。

## 8. 推荐结论：一个平台，两套界面逻辑

对前端设计来说，最稳妥的结论是：

`侠客岛` 应该被理解为“一个平台，两套界面逻辑”。

具体是：

1. 使用者界面：任务完成型界面
2. 开发者界面：平台理解与对接型界面

这两套界面可以共享品牌、共享后端、共享一些状态定义，但不应该共享同一套导航结构和同一套交互主线。

## 9. 使用者界面应该是什么逻辑

### 9.1 使用者界面的目标

使用者界面的唯一目标，是让人顺利完成一次内容任务。

因此它的主线应该是：

**我要做什么 -> 我填什么 -> 我点执行 -> 我看结果。**

而不是：

**这个系统有哪些模块 -> 这些模块怎么组织 -> 这个接口底层是什么。**

### 9.2 使用者界面的对象模型

使用者界面里最核心的对象不应该是“模块”，而应该是“任务入口”。

一期中，这些任务入口可以直接表现为三个可执行目标：

1. 观点生成
2. 语义审核
3. 文章工作流

对使用者来说，他们看到的是“我要做哪类任务”，而不是“我要调用哪个 Python 模块”。

### 9.3 使用者界面的信息架构

使用者界面建议按下面的结构组织：

1. 任务入口页
2. 目标选择
3. 参数填写
4. 执行按钮
5. 结果展示
6. 本页最近执行

这就是典型的“执行工作台”逻辑。

它不需要先出现：

- API 文档
- 调试参数总览
- schema 列表
- 日志查询中心
- 环境配置中心

### 9.4 使用者界面的语言系统

使用者界面必须使用业务语言，而不是代码语言。

建议：

- 用“观点生成”代替 `opinion`
- 用“语义审核”代替 `semantic_review`
- 用“文章工作流”代替 `workflow/article`

底层标识可以保留给系统内部，但主界面文案不应该让使用者对着路由名思考。

### 9.5 使用者界面的判断逻辑

使用者界面的判断逻辑应该围绕任务进行：

1. 当前选择了什么任务
2. 这个任务需要哪些输入
3. 当前输入是否足够执行
4. 执行后应该展示什么结果
5. 用户下一步最可能做什么

例如：

- 观点生成后，下一步是查看观点、支撑点和置信说明
- 语义审核后，下一步是查看是否通过、问题列表和改写建议
- 文章工作流后，下一步是查看整体状态和子任务结果

### 9.6 使用者界面不应该承载什么

使用者界面不应该承载：

- 开发调试入口
- 接口样例构造器
- 原始错误栈
- 内部目录结构
- 运行时日志细节
- 鉴权和环境配置说明

换句话说，使用者界面是“完成任务的地方”，不是“理解系统的地方”。

## 10. 开发者界面应该是什么逻辑

### 10.1 开发者界面的目标

开发者界面的目标不是让人做任务，而是让人理解和接入这个平台。

因此它的主线应该是：

**这个平台提供什么能力 -> 我怎么调用 -> 契约是什么 -> 出错怎么查。**

### 10.2 开发者界面的对象模型

开发者界面中最核心的对象应该是：

1. 能力
2. API
3. 契约
4. 环境
5. 调试与追踪

这套对象模型和使用者界面的“任务入口 + 表单 + 结果”是完全不同的。

### 10.3 开发者界面的信息架构

开发者界面建议按下面逻辑组织：

1. 平台概览
2. 能力清单
3. API 入口说明
4. 请求/响应契约说明
5. 错误响应说明
6. 环境与接入说明
7. 调试与任务追踪入口

如果以后要做开发者控制台，这个控制台更像：

- API 控制台
- 调试控制台
- 对接说明台

而不是运营工作台。

### 10.4 开发者界面当前能说到哪里

必须注意，当前仓库真实存在的开发者能力，主要还是这些：

1. 已有 3 个核心执行路由
2. 已有最小编排能力
3. 已有任务日志基础
4. 已有对应测试文件

当前还不能默认说成已经具备：

- 完整 schema registry 前台
- 可浏览的任务中心
- 完整配置中心
- 开发者专用聚合接口
- 完整权限与组织体系

所以开发者界面一期如果要出现，应该以“文档 + API 说明 + 调试辅助”为主，而不是先画成成熟控制台。

### 10.5 开发者界面的判断逻辑

开发者界面的判断逻辑应该围绕“对接与排错”展开：

1. 当前要接哪个能力
2. 该能力的正式接口是什么
3. 哪些字段必须传
4. 哪些字段只是前端辅助字段
5. 错误会以什么格式返回
6. 任务如何追踪
7. 这个需求是否已经超出一期边界

### 10.6 开发者界面不应该承载什么

开发者界面不应该被误做成：

- 使用者任务入口页
- 内容编辑台
- 运营配置中心
- 一键自动化策略后台

开发者界面是“平台控制与理解层”，不是“业务使用层”。

## 11. 使用者界面与开发者界面的共享和隔离

### 11.1 可以共享的东西

两类界面可以共享：

- 同一个 `侠客岛` 品牌
- 同一套后端 API 基础
- 同一套能力命名真相源
- 同一套错误状态基础语义

### 11.2 必须隔离的东西

两类界面必须隔离：

- 主导航
- 页面主目标
- 文案层级
- 信息密度
- 默认可见信息
- 入口路径

### 11.3 最重要的隔离原则

最重要的一条是：

**不要让使用者界面承担“理解平台”的责任，也不要让开发者界面承担“完成内容任务”的责任。**

## 12. 一期前端应该先做哪一套

如果一期前端只能优先做一套，那么应该优先做：

**使用者界面。**

原因很简单：

1. 当前已有执行 API，最适合先形成“可配置的执行入口页”。
2. 当前产品价值最直接的体现，是把三个触发目标做成可用入口。
3. 完整开发者控制台会天然要求更多接口、追踪能力和配置能力，而这些不属于当前一期主目标。

所以一期最合理的节奏是：

1. 先做使用者界面
2. 开发者侧先以文档、接口说明、轻量调试说明承接
3. 等后端补齐更多平台能力后，再考虑正式开发者控制台

## 13. 对前端设计的直接指导

如果你现在要开始画前端，应该先接受下面 6 条设计指导：

1. 你在画的不是“后台总控台”，而是“能力平台的入口界面”。
2. 一期主界面优先服务使用者，不优先服务开发者。
3. 使用者界面应围绕任务入口组织，不围绕 Python 模块目录组织。
4. 开发者界面应围绕 API、契约和调试组织，不围绕内容任务组织。
5. 两类界面可以共属于 `侠客岛`，但不能共用一套信息架构。
6. 一期不要提前把未来的调度、规则、平台治理能力画进主界面。

## 14. 最终结论

`侠客岛` 不是一个普通前端页面项目，也不是一个单一写稿工具。

它是一个正在首阶段建设中的中文医疗内容能力平台。当前最真实、最重要的产品表达方式，是把它理解成：

**一个有明确能力边界的平台，至少应该有两套不同的前端界面逻辑：**

1. 面向任务完成的使用者界面
2. 面向对接与调试的开发者界面

其中，一期前端应优先落地前者，后者先保持为平台说明与对接视角，而不是提前膨胀成完整后台。
