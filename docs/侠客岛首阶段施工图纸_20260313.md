## 3. 模块设计

### 3.1 API Layer

职责：

- 暴露统一 HTTP 接口
- 做请求鉴权、限流、参数校验
- 将外部请求转换为内部标准调用对象

首批接口草案：

- `POST /v1/opinion/generate`
- `POST /v1/review/semantic`
- `POST /v1/workflow/article`

首阶段鉴权立场：

- 本地开发阶段默认允许无鉴权运行，但仅限受控开发环境
- 一旦进入跨进程、跨团队或对外网络调用，至少升级为 API Key 鉴权
- 更复杂的鉴权方案进入后续 ADR

### 3.2 Orchestrator

职责：

- 接收标准化任务
- 判断是单模块调用还是工作流调用
- 组织模块顺序
- 传递 payload
- 汇总结果、状态与日志

边界：

- 只读取 capability metadata、contract 兼容性、模块状态和执行结果
- 不维护各模块内部规则、模板、证据索引或资产 inventory 明细
- 不因为某模块资料膨胀而同步膨胀编排层上下文

### 3.3 Opinion Module

输入最小集合：

- `source_packet` 或 `evidence_bundle`
- `audience`
- `thesis_hint`
- `context_metadata`

输出最小集合：

- `thesis`
- `support_points`
- `boundaries`
- `evidence_mapping`
- `confidence_notes`

### 3.4 Semantic Review Module

输入最小集合：

- `content`
- `prototype_hint`
- `register`
- `audience`
- `context_metadata`

输出最小集合：

- `passed`
- `severity_summary`
- `findings`
- `rewrite_target`
- `prototype_alignment`

### 3.5 Contracts Registry

Registry 组件职责：

- 管理模块输入输出 schema
- 管理 handoff 结构
- 管理 contract 版本
- 为测试和校验提供统一真相源
- 管理能力级目录元数据，使编排层知道“这个模块能提供什么类型的能力”

明确不负责：

- 不管理模块内部 inventory 明细
- 不替模块 owner 维护规则清单、模板清单或证据索引全文

首批 contract：

- `opinion_to_write`
- `review_then_rewrite`
- `semantic_review_result`
- `module_error`
- `task_trace`

命名约定：

1. handoff 类 contract 使用 `{source}_to_{target}` 风格
2. 模块结果类 contract 使用 `{module}_{result}` 风格
3. 基础设施类 contract 使用 `{domain}_{artifact}` 风格

contract 治理规则：

1. 新增、修改或废弃 contract，必须同时识别生产方与消费方
2. 任何 breaking change 只能通过 major version 生效
3. 非 breaking change 也必须同步更新 schema、测试和变更日志
4. contract 版本号独立于模块版本号，但必须建立兼容映射
5. contract 生效前，至少需要一次设计评审或 ADR 记录

### 3.6 Asset Bridge

职责：

- 向核心模块提供统一资产读取接口
- 隔离外部目录结构
- 屏蔽历史系统差异

补充边界：

- Asset Bridge 面向模块提供读取能力，不向 Orchestrator 暴露全量资产清单
- 是否读取哪些资产、如何组织内部 inventory，由模块 owner 在模块内部决定

首批桥接资产类别：

- rules
- editorial
- evidence
- structured

当前本地默认来源：

- `D:\汇度编辑部1\写作知识库\medical_kb_system_v2\rules\`
- `D:\汇度编辑部1\写作知识库\medical_kb_system_v2\editorial\`
- `D:\汇度编辑部1\写作知识库\medical_kb_system_v2\evidence\`
- `D:\汇度编辑部1\写作知识库\medical_kb_system_v2\structured\`

后续要求：

- 这些路径不能硬编码进核心模块
- 必须通过配置层注入

### 3.7 LLM Gateway

职责：

- 统一封装模型调用
- 管理模型配置
- 管理超时、重试、错误归因
- 为不同模块提供一致的模型执行接口

首阶段策略：

1. 默认采用单 provider、单主模型策略
2. 所有 prompt 模板与模型参数必须版本化
3. 必须记录模型名、耗时、失败原因和基础成本信息
4. 复杂多模型路由与回退策略后移到独立 ADR

## 4. 运行模式与失败策略

### 4.1 阶段与模式映射

- `Phase A / P0`：只要求单模块模式可用，首个目标是跑通 `opinion` 单模块调用
- `Phase B / P1`：补齐 `semantic_review` 单模块模式，并引入最小工作流模式
- 平台集成模式是否开放，取决于 API、日志、鉴权和部署边界是否达到可控状态

### 4.2 失败模式基本立场

1. contract 校验失败：直接拒绝请求，不进入模块执行
2. LLM 调用失败或超时：允许有限重试；若仍失败，返回结构化错误
3. Asset Bridge 读取失败：中止当前模块运行，并返回 `source_unavailable` 类错误
4. 工作流模式默认采用 fail-fast；只有被显式标记为可选的模块，才允许降级继续
5. 所有失败都必须返回统一 `module_error` 结构，并落任务日志
