## 肆、藏经阁储存模式建议

### 建议采用的储存结构

```
藏经阁/
├── L1/基础层/
│   ├── actions/           # 动作库
│   │   ├── core/          # 核心动作（稳定可迁移）
│   │   └── style/         # 风格动作（作者特色）
│   ├── components/        # 构件库
│   └── rules/             # 规则库
├── L2/样板层/
│   └── [作者名]/          # 样板作为上层聚合视图
│       └── [样板名].json  # 包含动作组合、规则调用、风格参数
└── L3/成品层/
    └── [文章ID]/          # 成品文章及其拆解
```

### 储存原则

1. **动作是底层主存储**，样板只是聚合视图
2. **稳定动作进core/**，风格动作进style/
3. **规则与动作绑定**，包含触发条件和失败降级
4. **隐喻作为独立资产**，标记适用领域和边界

### 不应采用的结构

- **整篇样板包作为底层存储**（对照组验证：方案A迁移成功率仅15%，强作者绑定、强领域绑定）
- **纯碎片化存储**（对照组验证：方案B迁移成功率仅25%，丢失叙事逻辑和整体风格）

## 伍、不完备之处

### 对象层面

1. **触发条件不完整**
   - 隐喻承重的触发条件（什么情况下用隐喻vs直接分析）
   - 级联vs平行的选择条件

2. **失败降级机制缺失**
   - 当对象无法迁移时的替代方案
   - 当新文章超出对象覆盖范围时的处理

3. **领域边界不清晰**
   - 医学稿vs科技稿vs商业稿的对象差异
   - 突发新闻vs常规分析的边界

### 规则层面

1. **组合规则不完整**
   - 动作组合的合法顺序
   - 构件嵌套的约束条件

2. **风格参数化不足**
   - 如何用同一套动作生成不同风格
   - 风格特色词与动作的关系

## 捌、最终结论

**当前候选储存模式方向正确，且已验证显著优于对照组，但规则层仍明显不完备。**

**依据**：

1. **方向正确**：动作/构件/规则的框架能解释多篇验证文章的结构
2. **对象稳定**：9个动作100%迁移成功，覆盖主要文章类型
3. **可生成性**：跨域重组证明了对象可用于生成而非仅分析
4. **显著优于对照组**：方案C（70%迁移成功率）vs 方案A（15%）vs 方案B（25%）
5. **IB期验证通过**：人物向80%、张佳伟90%构件稳定存在
6. **不完备**：触发条件、失败降级、领域边界等维度缺失
7. **规则层必须补硬**：否则生成可用性和失败定位能力受限

**建议**：继续沿动作/构件/规则方向深化，**优先补硬规则层（触发条件、失败降级）**，再做最终判断。
