# 阶段过渡操作规则

状态：Active  
日期：2026-04-03  
用途：规定多 Agent 藏经阁实验在一个正式阶段结束、下一个正式阶段开始前，如何单独存放讨论过程、阶段决议和补充执行文档

## 证据摘要

- 已直接核对当前实验正式目录长期按 `00_实验中心 / 01_实验方案 / 02_阶段结果 / 98_历史归档 / 99_待清理` 运行，但此前缺少一类专门存放“阶段过渡讨论与补充执行文档”的固定位置。
- 当前线程已明确要求：`IIA -> IIB` 之间的讨论过程、讨论结果、补充执行文档应单独落在实验目录内，而不是继续散落在总纲、方案或聊天上下文里。
- 已直接核对当前 `IIA` 已经产生需要过渡收纳的正式内容：人格命名统一、条件性对象表单列、IIB 候选样本筛选方式、阶段中间总结。

## 一、这条规则解决什么问题

阶段结束后，实验往往会出现一批不适合直接塞进：

1. `00_实验中心`
2. `01_实验方案`
3. `02_阶段结果`

但又必须长期保留的文档，例如：

1. 讨论过程纪要
2. 阶段决议
3. 补充执行指令
4. 待执行事项

这些内容如果没有单独落点，就会出现两个问题：

1. 正式方案和结果文档被过渡性讨论污染
2. 讨论结论只留在聊天里，后续无法回查

## 二、固定目录规则

从现在开始，实验目录固定新增一层：

`03_阶段过渡`

每当一个正式阶段结束、下一个正式阶段尚未正式开跑时，都在其下建立一个过渡子目录：

`<前阶段>-<后阶段>中间环节`

当前首个实例为：

`03_阶段过渡/IIA-IIB中间环节`

## 三、固定文档类型

每个阶段过渡目录，默认至少包含下面 4 类文档：

1. `00_目录说明.md`
   - 说明这个过渡目录收什么、不收什么
2. `01_讨论过程纪要.md`
   - 记录关键讨论路径和问题收敛过程
3. `02_讨论结果与阶段决议.md`
   - 记录已经被确认的阶段决议
4. `03_补充执行文档.md`
   - 记录给各中心或总协调方的额外执行指令

如有必要，可补：

5. `04_待执行事项.md`
   - 用于阶段过渡期间的明确待办

## 四、执行文档强制规则

只要某份文档属于“要发给别人执行”的文档，就必须同时写清两件事：

1. 交什么
2. 交到哪里

这里的“交到哪里”必须是：

**精确提交路径**

不能只写：

1. 文件名
2. “你自己的目录”
3. “当前阶段目录”

如果是多中心并行任务，还必须额外满足：

1. 先预建每个中心自己的子目录
2. 在执行文档里写死每个中心对应的目标路径
3. 过程文档和结果文档默认与主交付同目录存放，除非另有明确约定

这条规则的目的，是避免再次出现：

1. 知道要交什么，但不知道交到哪
2. 各中心各自理解“自己的目录”
3. 后续汇总时找不到统一落点

## 五、这类目录收什么，不收什么

### 收纳

1. 讨论过程
2. 讨论结果
3. 过渡阶段指令
4. 阶段间补充规则
5. 尚未上升为正式方案但必须留痕的执行文档

### 不收纳

1. 三中心独立实验结果
2. 正式冻结版方案
3. 正式冻结版对象表
4. 正式冻结版终点评估表

这些仍应分别进入：

1. `00_实验中心`
2. `01_实验方案`
3. `02_阶段结果`

## 六、创建时机

一个阶段过渡目录应在下面两个条件同时满足时创建：

1. 上一阶段的主体执行已完成
2. 下一阶段的正式实验方案尚未冻结

也就是说，它的用途是：

**承接上阶段的中间总结，并给下阶段开工做准备。**

## 七、当前执行决定

本规则从 `IIA -> IIB` 开始执行。  
后续所有正式阶段结束后，均按同样方式建立新的过渡目录。
