## 一句话口径

`III期` 的模块层补丁，按下面这条链执行：

`需求对齐回执层识别主线与挂载模块 -> 输入冻结层按模块冻结材料 -> 写作合同与结构编译层调度模块并写清模块写法边界 -> 门禁层检查模块是否抢主线、是否把骨架压坏`

其中，《仑卡奈单抗》改稿复盘这条经验本身，不新增回执层字段，也不新增冻结前置。
模块层只负责识别它命中了哪类模块家族；真正吸收到系统里的位置，是写作合同层先写清好稿该长什么样、门禁层按同一口径复核、定向返修层按同一口径回修。

## 一、补丁落点

| 层级                 | 当前职责                                         | 本次补丁新增职责                       |
| -------------------- | ------------------------------------------------ | -------------------------------------- |
| 需求对齐回执层       | 锁真实写作场景、一级稿型、主二级稿型、主样稿母本 | 锁主线模块、挂载模块                   |
| 输入冻结层           | 冻结整篇稿件输入                                 | 冻结各模块取材边界                     |
| 写作合同与结构编译层 | 编译整篇写作合同                                 | 编译模块顺序、位置、篇幅和禁止动作     |
| 质量与语义门禁层     | 检查事实、风格、结构、合规                       | 检查模块是否抢主线、越界展开、回错收口 |

## 二、系统新增对象

### 1. 主线

主线由下面 `3` 项共同决定：

1. 真实写作场景
2. 一级稿型
3. 主二级稿型

主线负责回答：

1. 这篇稿到底在写什么
2. 全文最大承重是什么
3. 最终收口回哪里

### 2. 挂载模块

挂载模块是围绕主线插入的内容块。
模块不等于稿型。

模块负责回答：

1. 还要补哪些内容块
2. 每个内容块承担什么功能
3. 它们出现在哪一段
4. 它们最多占多大篇幅

### 3. 模块卡

模块卡是模块层的正式对象。
后续所有混合需求稿件都必须落到模块卡。

## 三、需求对齐回执层新增输出

通过 `R-01` 后，回执层除原有输出外，新增 `2` 项：

1. `主线模块`
2. `挂载模块`

这里的 `主线模块 / 挂载模块` 是模块层通用对象，不等于《仑卡奈单抗》这条经验的直接吸收位。
这条经验只借它定位：当前命中的是开头位模块、研究/证据型模块，还是回收型模块。

最小回执格式更新为：

```text
- 真实写作场景：
- 一级稿型：
- 主二级稿型：
- 主样稿母本：
- 主线模块：
- 挂载模块：
- 交付物：
- 受众 / 平台 / 篇幅：
- 主角度 / 主问题：
- 主证据链 / 主信息链：
- 必写点：
- 禁区：
```

## 四、输入冻结层新增动作

冻结层不再只冻结整篇输入。
冻结层必须新增“模块取材冻结”。

模块取材冻结只负责保证模块吃对材料，不承载《仑卡奈单抗》这条经验本身的压缩规则。
这条经验的前置约束，不在冻结层生效，而在后面的写作合同层生效。

每个挂载模块至少冻结 `3` 类信息：

1. 该模块必吃材料
2. 该模块不该吃的材料
3. 该模块允许引用的样稿边界

示例：

| 模块           | 必吃材料                   | 禁止缺失               |
| -------------- | -------------------------- | ---------------------- |
| 单产品机制深挖 | 主文献、机制图、关键机制链 | 缺主文献时不得成立     |
| 科学家人格背书 | 访谈原话、履历、关键判断   | 只有宣传口号不得成立   |
| 公司价值回收   | 管线、平台、商业位置       | 没有公司层证据不得拔高 |

## 五、写作合同与结构编译层新增动作

模块在这一层正式挂载。

《仑卡奈单抗》改稿复盘命中时，真正的写作前置约束就在这一层落地，而不是上提到回执层或冻结层。
这条经验不只管“压缩字数”，而是先把好稿的基本样子写清：不堆修辞、不散主线、不断逻辑、不让证据和结论脱节。

这一层必须给每个模块写清 `5` 件事：

1. `位置`
   - 开头 / 前中段 / 中段 / 结尾
2. `优先级`
   - 主挂载 / 次挂载 / 回收模块
3. `篇幅上限`
   - 不允许无限展开
4. `禁止动作`
   - 不得抢主线 / 不得提前收口 / 不得改写全文主问题
5. `写法边界`
   - 不过度修辞 / 不重点发散 / 不逻辑断裂 / 不让证据和结论脱节；开头位模块不得改写进入方式 / 研究型模块不得压没动作链 / 回收模块不得压成抽象总结

## 六、模块卡标准字段

每张模块卡最少包含下面 `6` 项：

1. `模块名`
2. `模块职责`
3. `允许位置`
4. `必吃材料`
5. `篇幅上限`
6. `禁止动作`

推荐字段：

7. `可挂在哪些主二级稿型下`
8. `不可共存模块`
9. `对应样稿参考`

## 七、模块调度规则

### 规则 1

一篇稿只能有 `1` 条主线。

### 规则 2

挂载模块可以有多个，但必须有主次。最小排序：

1. 主线模块
2. 主挂载模块
3. 次挂载模块
4. 结尾回收模块

### 规则 3

挂载模块默认不能抢主线。

必须显式检查：

1. 机制模块是否把产业故事写没了
2. 人物背书模块是否把产品线写没了
3. 价值回收模块是否提前抢结论

### 规则 4

模块挂载必须受主二级稿型约束。

示例：

1. `产业故事` 可挂 `单产品机制深挖`、`科学家人格背书`、`公司价值回收`
2. `研究/文献深度解读` 可挂 `产品价值回收`
3. `患者向-AE与管理` 不挂 `产业故事`

### 规则 5

模块一旦改动，门禁层必须回看主线是否被抢。

### 规则 6

模块层不只管“挂什么”，也管“怎么写、怎么审、怎么改”。

但这里要分清节点职责：

1. 模块层只负责识别命中家族
2. 写作合同层负责把约束写成正式边界
3. 门禁层负责复核有没有压坏骨架
4. 定向返修层负责按同一套边界回修被点名问题，不把改稿理解成单纯删字

这条经验不生成新模块，也不新增上游前置。它不是只管压缩字数，而是管首稿生成、审核和改稿都不能把稿子写坏，只按下面 `3` 类模块写法生效：

1. 开头位模块
   - 在写作合同层写成：只准建立入口，不准改写全文进入方式
   - 在门禁层复核：有没有把全文入口换轨、把主问题落点压没
2. 研究/证据型模块
   - 在写作合同层写成：必须保留最小动作链和证据桥
   - 在门禁层复核：有没有只剩结果名词、指标名或结论标签，是否从指标直接跳抽象判断
3. 回收模块
   - 在写作合同层写成：必须回收前文主证据链，不得压成抽象总结或广告话
   - 在门禁层复核：有没有失去回收功能，只剩抽象总结、拔高句或广告话

定向返修层命中这条经验时，只按门禁点名的问题回修入口换轨、动作链缺失、回收失效、过度修辞、重点发散、逻辑断裂和证据脱节，不做泛修。

### 规则 7

本轮新增经验说明：影响返修质量的，不只是“提示词写得好不好”，而是上下文工程本身。

这里至少新增 `4` 条硬认识：

1. 输入标准化的目标，不只是“整理一下”，而是 `不丢失 + 降噪 + 不多不少`
2. 写作质量最稳的场景，不是“从头空写”，而是上下文定死后的局部补写
3. 大纲不是附属产物，而是独立过闸点；它要先成型，再快速确认
4. 生成全稿和局部修改不是同一个 Agent 任务，内置约束应该分开

上下文粒度的优先级，当前可先写成：

1. `给上文 + 给下文 + 补中间一句`，质量最高
2. `给上一句 + 接下一句`，质量次一级
3. `只有提纲和逻辑，让它从头写开头`，质量最低

这也解释了为什么开头和结尾最难：它们天然缺一侧上下文，默认不能按中段同等质量预期处理。

### 规则 8

返修层最容易失效的，不是“不会改”，而是上下文过大后开始反复横跳。

当前真实风险不是单点删改，而是：

1. 写作材料和上下文一旦爆炸，AI 很容易记不稳全文约束
2. 医药稿很多改动不是“换一个词”这么简单，而是逐字逐句牵动上下文
3. 一个比喻、一个论断、一个承接句，背后可能牵着上万字的信息链
4. 如果返修还继续沿用又长又散的原上下文，AI 常见表现不是“越改越准”，而是来回补偿
5. 旧会话里已经残留了前几轮写稿、审稿、返修的历史偏差；继续在旧会话里补，会把这些偏差一起带下去

最典型的横跳表现就是：

1. 改了逻辑，修辞炸掉
2. 收了修辞，逻辑断掉
3. 压了字数，信息掉了
4. 扩了信息，水字数了
5. 修了一个局部比喻，整段主线被带歪

所以，门禁后的返修不能默认理解成：

1. 继续沿用原写稿上下文
2. 继续让同一轮长对话硬扛
3. 在旧 session 里来回补几轮
4. 把所有问题一次性丢给一个 Agent 整篇重写

更合理的理解是：

门禁层返回的不是“继续聊着改”，而是“生成一份更小、更干净的改稿任务”。

这里要写死一条更强约束：

1. 每一轮修改都必须新开 `1` 个全新 session
2. 新 session 只能吃本轮标准上下文和干净约束，不继续继承旧会话里的长链聊天噪声
3. 旧 session 只负责提供原始证据，不负责继续执行修改
4. 这条不分整稿改还是局部改，统一生效

这里说的“标准上下文”，至少包括：

1. 当前待改的原文范围
2. 上下文定死的前后句 / 前后段
3. 本轮只改什么
4. 不允许动的骨架
5. 门禁点名问题
6. 本轮验收标准

但这个“重开改稿任务”不默认一律打回需求层，而是按问题级别分流：

1. `局部问题`
   - 例如 `3` 句话内的逻辑断裂、某一段修辞过满、某个结论和证据没接上、某个比喻不稳
   - 这类问题留在定向返修层处理，但也必须新开干净 session
2. `结构性问题`
   - 例如故事线错了、逻辑大纲错了、主架构没立住，或多个局部问题彼此牵连
   - 这类问题应回到写作合同与结构编译层，重开一个整稿改任务，并新开 session
3. `任务理解问题`
   - 例如受众错了、稿型错了、主问题错了、交付目标错了
   - 这类问题才回到需求对齐回执层

因此，`门禁 -> 返修` 的真实分流不该写死成单一路径，而应是：

`门禁层判问题级别 -> 新开 session -> 小问题走局部改 / 中问题回写作合同层重开整稿改 / 大问题回需求层重开`

### 规则 9

严格审核后的返修，理想形态不是“整篇稿再给一个大 Agent 统改一轮”，而是先拆小，再串改。

原因很简单：

1. 小问题本来就是局部问题，不值得让整篇上下文再次爆炸
2. 很多修改意见天然只作用于一个很小的范围，例如某 `3` 句话、某 `1` 段、某 `1` 个比喻、某 `1` 条证据桥
3. 如果把这些小问题和全文其它问题一起交给同一个长上下文，最容易出现补这头坏那头
4. 如果多个相邻句子同时并改，最容易出现“每句都像改好了，但拼回去串味”

所以更合理的返修逻辑是：

1. 先严格审核，把问题拆成尽量小的修改单
2. 每张修改单只负责 `1` 个点，不顺手泛改
3. 每轮默认只改 `1` 个句位 / `1` 个信息点 / `1` 个空缺位
4. 每一张修改单都要新开 `1` 个干净 session
5. 每个 Agent 只吃：
   - 当前小段原文
   - 点名问题
   - 不允许动的骨架
   - 允许重写的范围
   - 本次只优先修什么
6. 当前轮改完后，再把新结果作为下一轮的定死上下文
7. 合并后的全文必须再过一轮总审

这里必须强调：

1. 默认是串改，不是并改
2. 相邻句、同一段、共享同一承重逻辑的问题，禁止并改
3. 只有明确证明彼此完全独立的空缺位，才允许极少量并发
4. 开头、结尾、过渡句、证据桥这类高耦合位置，默认禁止并发

这也意味着：局部修改的最佳粒度不是改字，也不是一整段重写，而更接近：

1. `1` 句话
2. `1` 个信息点
3. `1` 个空缺位

再大，容易串味；再小，又会重新发散。

如果系统当前还做不到稳定多 Agent 并发，也不等于这条逻辑失效。
最小可执行版本仍然成立：

1. 审核先吐出小修改单
2. 每张单新开一个干净 session，逐单串改
3. 每次只让 Agent 吃当前小任务，不继续背整篇上下文
4. 最后统一合并并总审

也就是说，即使暂时做不到真正并发，也要先把返修口径改成“拆小任务 + 新开 session + 串改”，而不是“带着原上下文继续聊着改”。

## 八、门禁层新增检查项

门禁层对混合需求稿件新增 `4` 个问题类型：

1. `主线被抢`
2. `模块越界展开`
3. `回收位置错误`
4. `模块占比失控`

门禁层输出的问题单，必须能明确指出：

1. 哪个模块有问题
2. 问题类型是什么
3. 要缩、移、删还是回收

首稿、压缩稿、改稿后的混合稿件，还必须额外回看下面 `3` 条：

门禁命中这条经验时，不只看字数，还要回看有没有过度修辞、重点不清、逻辑断裂、思路发散、证据和结论脱节。
1. 开头模块有没有把全文入口换轨
2. 研究/证据型模块有没有把动作链或证据桥压没
3. 回收模块有没有把前文证据链压成抽象总结

## 九、模块层与 8 公式的衔接

`8公式` 不新增编号，按下面方式接模块层：

1. `公式1-2`
   - 继续锁整篇主任务、主范围
2. `公式3-4`
   - 显式接模块，把模块视为成组的内容单元 / 逻辑单元
3. `公式5-8`
   - 负责模块组合、顺序、篇幅、收口和有效大纲

当前缺口可以直接写成一句：

`8公式当前主要覆盖整稿级主线编译，尚未覆盖模块级调度。`

## 十、III期后续测试点

模块层补丁后续至少测下面 `3` 项：

1. 混合需求稿件的主线是否更稳
2. “每条线都写一点、没有主轴”的情况是否下降
3. 加入模块卡后，`8公式` 是否更能稳定编排混合稿件
4. 压缩后的混合稿件，开头入口、研究动作链、结尾回收是否仍完整

## 十一、当前执行口径

冷启动 Agent 处理混合需求稿件时，先按下面顺序执行：

1. 锁真实写作场景
2. 锁一级稿型
3. 锁主二级稿型
4. 锁主样稿母本
5. 锁主线模块
