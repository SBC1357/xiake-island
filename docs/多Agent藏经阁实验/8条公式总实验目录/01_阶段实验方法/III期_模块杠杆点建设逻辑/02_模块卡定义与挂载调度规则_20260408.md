# 模块卡定义与挂载调度规则

状态：III期模块层硬规格 / 已写入 / 待验证
日期：2026-04-08
用途：冷启动 Agent 直接执行——模块卡怎么填、模块怎么挂、哪些稿型能挂哪些模块、门禁怎么判"抢主线"
前置依赖：必须先读完 [01_模块层补丁建设逻辑_20260408.md](01_模块层补丁建设逻辑_20260408.md)

---

## 执行口径

1. 一篇稿只有 `1` 条主线。主线由 `真实写作场景 × 一级稿型 × 主二级稿型` 共同决定。
2. 挂载模块是围绕主线插入的内容块，不等于稿型，不替代主线，不改写全文主问题。
3. 每个挂载模块必须写成标准模块卡，模块卡不完整不准挂载。
4. 模块挂载受主二级稿型约束——映射表里没有的组合，默认禁止。
5. 门禁层必须检查模块是否抢主线；发现就判 `MODULE_OVERREACH`。

---

## 一、模块卡标准字段

每张模块卡必须包含以下 `9` 项。前 `6` 项为必填，后 `3` 项为必填（非空即有效）。

| 序号 | 字段名 | 定义 | 取值规则 |
| --- | --- | --- | --- |
| 1 | `module_name` | 模块名 | 从下方模块库中选取，不允许自造 |
| 2 | `module_role` | 模块职责 | 一句话写清这个模块在本篇稿里干什么 |
| 3 | `position` | 允许位置 | 只允许填：`开头` / `前中段` / `中段` / `后中段` / `结尾` |
| 4 | `priority` | 挂载优先级 | 只允许填：`主挂载` / `次挂载` / `回收模块` |
| 5 | `source_materials` | 必吃材料 | 列出本模块必须消费的材料类型；缺必吃材料时本模块不得成立 |
| 6 | `max_share` | 篇幅上限 | 占全文比例上限，格式为百分比，如 `≤20%` |
| 7 | `forbidden_actions` | 禁止动作 | 本模块绝对不能做的事，至少 1 条 |
| 8 | `incompatible_modules` | 不可共存模块 | 同一篇稿中不允许同时挂载的模块名 |
| 9 | `sample_ref` | 对应样稿参考 | 可引用的样稿母本或具体样稿片段 |

### 字段硬约束

1. `module_name` 只能从"二、模块库"里选。如果冷启动 Agent 认为需要新模块，走 `HARD_BLOCK_MINIMAL`，不允许当场自造。
2. `position` 一张卡只能选 `1-2` 个位置。选 `2` 个时必须相邻（如 `中段` + `后中段`）。
3. `priority` 一篇稿最多 `1` 个 `主挂载`、`2` 个 `次挂载`、`1` 个 `回收模块`。
4. `max_share` 所有挂载模块合计不得超过 `45%`。主线内容必须占 `≥55%`。
5. `source_materials` 缺必吃材料时，该模块降级为 `不挂载`，不允许用推测或泛论填充。

### 模块写法补丁（接入《仑卡奈单抗》改稿复盘）

这条经验不新增模块编号，也不生成新模块；它按模块家族补成写法约束。
模块层在这里也只负责识别命中的模块家族；真正写入系统的位置，是写作合同层把边界写死，门禁层按同一套边界复核。
它不是只管“压缩字数”，而是同时约束首稿生成、审核和改稿，防止 AI 把稿子写成过度修辞、重点不清、逻辑断裂、思维发散、证据和结论脱节。

#### 1. 开头位模块

适用：`M-05 格局定位`、`M-07 患者人文`、`M-10 未满足需求`

硬约束：

1. 只准建立入口，不准改写全文进入方式。
2. 可以压背景解释，但不能把主问题落点压没。
3. 不得把模块入口写成独立赛道，导致读者先进模块、后进主线。

#### 2. 研究/证据型模块

适用：`M-01 机制锚点`、`M-02 数据锚点`、`M-08 安全性管理`、`M-09 监管背景`

硬约束：

1. 必须保留最小动作链或证据桥。
2. 不得只剩结果名词、指标名或结论标签。
3. 不得从指标/数据/监管动作直接跳到品牌化结论或抽象价值判断。

#### 3. 回收型模块

适用：`M-04 价值回收`；`M-07 患者人文` 在 `回收模块` 用法下

硬约束：

1. 可以变短，但必须回收前文主证据链。
2. 不得只剩抽象总结、拔高句或广告话。
3. 不得把“结尾变短”偷换成“结尾失去回收功能”。

### 映射表硬约束

1. 一篇稿最多挂载 `3` 个模块（含主挂载 + 次挂载 + 回收模块）。
2. `回收模块` 全篇最多 `1` 个，且只能出现在 `后中段` 或 `结尾`。
3. 标记 `×` 的组合不允许在任何条件下挂载。
4. 标记 `○` 的组合挂载时，模块卡的 `module_role` 必须写清"在这篇稿里具体干什么"，不允许只写默认值。
5. 如果同一篇稿要挂 `2` 个 `●` 推荐模块，两者 `position` 不得相同。

## 四、模块挂载顺序与优先级规则

### 规则 1：主线永远先写

全文结构编译时，主线承重段先锁、先写、先占位。模块只能在主线骨架确定后再插入。

### 规则 2：挂载优先级排序

```
主线 > 主挂载模块 > 次挂载模块 > 回收模块
```

篇幅冲突时，按此顺序砍。先砍回收模块篇幅，再砍次挂载，最后才考虑砍主挂载。主线不砍。

### 规则 3：位置排序规则

```
开头区: 未满足需求 > 格局定位 > 患者人文
前中段: 格局定位 > 机制锚点 > 监管背景 > 数据锚点
中段:   机制锚点 > 数据锚点 > 竞品对照 > 人物背书 > 监管背景
后中段: 安全性管理 > 竞品对照 > 人物背书 > 价值回收
结尾区: 价值回收 > 患者人文
```

同一位置出现 2 个模块时，按上述顺序排前后。如果排不下，降级后面那个模块到下一个位置。

### 规则 4：篇幅硬上限

| 约束 | 值 |
| --- | --- |
| 所有模块合计篇幅上限 | ≤45% |
| 单个模块篇幅上限 | 见模块卡 `max_share` |
| 主线内容最低占比 | ≥55% |

### 规则 5：模块写作、压缩、返修都不准压坏骨架

这条规则不只在压缩时生效；首稿生成、删改提纯、返修回写都按同一口径执行。
模块需要压缩时，默认只压重复强调、背景详解、次级解释；承担逻辑功能的骨架句不准直接压没。
开头位模块、研究/证据型模块、回收型模块的具体硬约束，以“模块写法补丁（接入《仑卡奈单抗》改稿复盘）”为唯一来源，不在这里重复展开。
需要压缩时，优先砍重复强调，再砍背景详解，再砍次级解释，最后才微调主线和模块骨架句。

### 规则 6：模块不得改写主线

以下动作构成"改写主线"，一律禁止：

1. 模块段落的论证结论替代了全文主问题的回答
2. 模块占比超过主线，读者先被模块观点说服
3. 模块提前给出全文收口判断
4. 模块引入了主线材料以外的核心论断

### 规则 7：模块降级规则

遇到以下情况，模块必须降级或移除：

| 情况 | 动作 |
| --- | --- |
| 必吃材料不足 | 直接移除该模块 |
| 篇幅超限 | 先压缩到上限，压不下则降级优先级 |
| 与不可共存模块冲突 | 保留优先级更高的，移除另一个 |
| 门禁判定 `MODULE_OVERREACH` | 压缩或移除，重跑门禁 |

### 规则 8：上下文工程优先于空写，大纲是独立过闸点

本轮新增经验，不只在“怎么返修”上，而是进一步说明了“为什么会写坏”。

当前可直接写死的最小口径是：

1. 输入标准化的目标，不只是“整理输入”，而是 `不丢失 + 降噪 + 不多不少`
2. 写作质量最稳的场景，是上下文定死后的局部补写
3. `给上文 + 给下文 + 补中间一句`，默认优于 `只给上文接下一句`
4. 开头和结尾天然缺一侧上下文，默认比中段更难写
5. 大纲不是附属物，而是独立过闸点；它先成型，再快速确认
6. 生成全稿 Agent 和局部修改 Agent，不应共用一套内置 prompt

### 规则 9：返修不默认沿用原上下文，先按问题级别分流

门禁后的返修，默认不是“继续沿用原写稿上下文再聊一轮”。

原因不是形式问题，而是当前稿件返修经常存在下面这类高风险：

1. 上下文太长，材料太多，Agent 容易记忆漂移
2. 改动粒度很深，经常是逐字逐句改，不是简单替换
3. 单个比喻、承接句、证据桥背后牵连整段甚至全文
4. 多个问题一起丢回去，最容易来回补偿，最后反复横跳
5. 旧 session 会把前几轮的误判、跑偏和聊天噪声继续带下去

典型表现：

1. 改逻辑，修辞炸了
2. 改修辞，逻辑断了
3. 压字数，信息少了
4. 补信息，水字数了

所以返修的默认动作不是“沿用旧会话继续修”，而是：

`每一轮修改都新开 1 个全新 session，让它只吃本轮标准上下文和干净约束。`

这里的“标准上下文”至少要包含：

1. 当前待改的原文范围
2. 定死的上下文前后句 / 前后段
3. 本轮只改什么
4. 不允许动的骨架
5. 门禁点名问题
6. 本轮验收标准

所以返修必须先分级：

1. `局部问题`
   - 限于 `1` 处小段、`1` 组三句话、`1` 个比喻、`1` 条证据桥
   - 留在定向返修层处理，但也必须新开干净 session
2. `结构性问题`
   - 故事线、逻辑大纲、主架构没立住，或多个局部问题彼此牵连
   - 回写作合同与结构编译层，重开一个整稿改任务，并新开 session
3. `任务理解问题`
   - 受众、稿型、主问题、交付目标判断错了
   - 回需求对齐回执层，重开任务理解

当前系统里的默认分流应理解为：

`门禁层 -> 先判问题级别 -> 新开 session -> 局部问题走局部改 / 结构性问题回合同层 / 任务理解问题回需求层`

### 规则 10：严格审核后的小问题，优先拆成小修改单

如果门禁给出的是否定局部问题，而不是全文级错误，则返修默认先拆单，不默认整篇统改。

每张小修改单只允许包含：

1. 当前小段原文
2. 门禁点名问题
3. 不允许动的骨架
4. 允许重写的范围
5. 本次只优先修什么
6. 定死的上下文前后句 / 前后段

一张单里不允许同时混入：

1. 修辞
2. 逻辑
3. 结构
4. 信息补充

这 `4` 类里跨层过多的问题；一旦混多了，就默认应先升级为结构性问题，回合同层重开。

局部修改的最优粒度，默认不是改字，也不是整段一起推倒重来，而是：

1. `1` 句话
2. `1` 个信息点
3. `1` 个空缺位

### 规则 11：小修改单彼此独立时，允许并发；不能独立时，禁止并发

当前默认应理解为：`串改优先，并发例外。`

只有彼此完全独立、互不牵连的小问题，才允许极少量并发，例如：

1. 某三句话逻辑断裂
2. 某一段修辞过满
3. 某个结论和证据没接上
4. 某个局部比喻不稳

下面这些情况默认禁止并发，必须回上游重开：

1. 开头入口错了
2. 主线错了
3. 结尾回收错了
4. 多个问题共享同一主承重段
5. 一处改动会连带改动多个段落
6. 相邻句子彼此串联，一句的修改会影响下一句怎么写

如果系统能稳定支持多 Agent，并发返修的正确顺序是：

1. 门禁拆出多个小修改单
2. 每个 Agent 只改 `1` 张单
3. 主控统一合并
4. 合并后的全文重跑总审

如果系统暂时做不到稳定多 Agent，并不等于继续整篇乱改。
最小可执行替代仍然是：

1. 先拆小修改单
2. 每张单都新开一个干净 session，串行逐单处理
3. 每次只喂当前小任务，不继续背整篇长上下文
4. 最后统一合并并总审

### 规则 12：问题级别判断，最好用实稿案例先校准

为了避免把小问题误判成结构问题，或者把结构问题误判成任务理解问题，判断时最好先拿真实稿面案例校准。

以《仑卡奈单抗》改稿复盘为例：

1. `局部问题` 的典型案例
   - `LAMP1下降，说明仑卡奈单抗具有局部神经网保护效应`
   - 这里的问题不是全文任务错了，而是这一小段从指标直接跳到了抽象结论，中间证据桥没了
   - 这种就该拆成 `1` 张小修改单，局部回修，不需要整篇重开
2. `局部问题` 的另一类案例
   - 研究设计段里把“给予 mAb158 干预，并设置不同剂量组”压没，只剩“观察了哪些指标”
   - 这里的问题集中在一个研究动作链断裂，仍属于局部问题
3. `结构性问题` 的典型案例
   - 原稿开头是“从神经突触切入研究主问题”，改完却变成“先讲产品机制框架，再讲这次研究补了一环”
   - 这已经不是某一句写坏，而是全文入口换轨，读者进入文章的路径变了
   - 这种不应继续按小句修补，而应回写作合同层，重开一个改稿任务
4. `结构性问题` 的另一类案例
   - 结尾如果只剩“这项研究为理解临床价值提供了新证据支持”，把前文三条证据链都收没了
   - 这说明结尾承担的任务变了，属于全文收口结构问题，不是单句润色问题
5. `任务理解问题` 的边界案例
   - 如果同一篇稿本来要写的是“从神经突触切入解释一项新研究”，最后却被整体改写成“仑卡奈单抗机制总览”或别的稿型
   - 那就不是结构问题，而是主问题、稿型、交付目标一起漂了
   - 这类才应该回需求层重开

判断时可以先问一句最短问题：

`这次改动，伤的是一句、一段，还是整篇到底在写什么？`

对应分流就是：

1. 伤一句或一小段：局部问题
2. 伤入口、承重、收口：结构性问题
3. 伤主问题、稿型、交付目标：任务理解问题

### 规则 12：案例持续扩展，清单按案例回抽，不先封死范围

上面这些案例不是封闭列表，只是当前已经核到的第一批校准样本。

更合理的用法不是：

1. 先把清单写死
2. 以后所有问题都硬塞进现有格子

而是：

1. 持续补新案例
2. 用新案例校准旧判断
3. 再把阶段内反复出现的问题，回抽成当前版本的操作清单

所以，新增案例时默认走下面 `5` 个动作：

1. 先记录原句 / 原段 / 原稿面现象，不先下结论
2. 再记录它到底伤的是局部、结构，还是任务理解
3. 再记录这次正确分流是原地修、回合同层，还是回需求层
4. 如果它只是旧问题的新样子，就补进旧案例簇，不急着另起新条目
5. 只有连续出现、且旧案例接不住时，才把它反抽成新的清单动作

这也意味着：

1. 案例库的范围应持续扩展
2. 操作清单只代表“当前阶段最常用、最稳定的一批动作”
3. 清单可以改、可以收紧、可以换序，但不应冒充“问题世界已经封闭”

## 五、门禁层判断"模块抢主线"的具体判据

门禁层对挂载模块新增 `4` 类问题类型，每类给出可判定条件。

### 问题类型 1：`MODULE_OVERREACH`（模块抢主线）

判定条件——满足以下任意 `1` 条即成立：

1. 模块段落字数超过主线最大单段字数
2. 模块段落出现了对全文主问题的直接回答
3. 模块的结论语气强于主线结论语气
4. 读者读完模块段后，主线后续段落变成"补充信息"

### 问题类型 2：`MODULE_EXPANSION`（模块越界展开）

判定条件——满足以下任意 `1` 条即成立：

1. 模块实际篇幅超过模块卡 `max_share`
2. 模块引入了 `source_materials` 以外的新材料
3. 模块内部出现二级展开（模块里面再嵌子模块结构）
4. 模块段落数超过 `2` 段（回收模块超过 `1` 段）

### 问题类型 3：`MODULE_MISPLACED`（回收位置错误）

判定条件——满足以下任意 `1` 条即成立：

1. `回收模块` 出现在 `中段` 或更早
2. `开头` 位置的模块出现在第 3 段之后
3. 模块实际位置与模块卡 `position` 不符

### 问题类型 4：`MODULE_SHARE_OVERFLOW`（模块占比失控）

判定条件——满足以下任意 `1` 条即成立：

1. 所有模块合计篇幅超过全文的 `45%`
2. 单个模块篇幅超过其 `max_share`
3. 主线内容低于全文的 `55%`

### 门禁附加核对（首稿、压缩稿、改稿稿面必看）

以下现象不单开新问题类型，但命中时必须在问题单"具体证据"里写明：

1. `开头位模块` 把全文进入方式改写了，或把主问题落点压没了
2. `研究/证据型模块` 把最小动作链或证据桥压没了，只剩结果名词、指标名或结论标签
3. `回收型模块` 把前文证据链压成抽象总结、拔高句或广告话

### 门禁输出格式

门禁层发现模块问题时，问题单必须写成：

```text
- 问题类型：MODULE_OVERREACH / MODULE_EXPANSION / MODULE_MISPLACED / MODULE_SHARE_OVERFLOW
- 涉及模块：[模块名]
- 具体证据：[哪一段、什么现象]
- 处置建议：缩 / 移 / 删 / 降级
```

## 六、模块层与 8 公式的衔接（不新增编号）

| 公式区间 | 与模块层的关系 |
| --- | --- |
| 公式 1-2 | 锁整篇主任务、主范围。模块不参与公式 1-2 的锁定 |
| 公式 3-4 | 负责段落推进与逻辑单元。模块在这里被视为"成组逻辑单元"，挂载位置和顺序在此阶段编译 |
| 公式 5-6 | 负责表达线与节奏。模块段落必须服从主线的表达线和节奏，不得自带独立风格 |
| 公式 7-8 | 负责收口与大纲。回收模块在此阶段参与收口；非回收模块不参与收口环节 |
