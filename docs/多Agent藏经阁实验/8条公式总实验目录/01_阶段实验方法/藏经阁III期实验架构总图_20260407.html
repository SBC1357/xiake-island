III 期节点与路由总图

现在要画清的，
不是“节点名”，而是“节点怎么过”

这张图不再把 `R-01` 误看成整稿总流程，也不把模块误看成一条平行主链。 当前更准确的总逻辑是：`用户输入 -> 节点 -> 节点内路由逻辑 -> 节点输出 -> 节点间路由 -> 下一个节点`。 模块路由不是另起一条链，而是嵌进某些节点内部的一层。

节点先吃输入 不是“先有路由再有节点”，而是先进入节点，再在节点内部做判断与分支。
R-01 只是第一条节点间路由 它只管“需求对齐回执层 -> 输入冻结层”这一跳，不代表整稿从头到尾的总流程。
模块嵌在节点内部 模块不会各跑一条小主线，而是在回执、冻结、合同、门禁这些节点里被挂载、限制和判错。

一句话看图方式

先看节点自己内部怎么判断，再看两个节点之间怎么交接，最后再看模块在哪些节点已经能稳定被路由， 哪些节点还停在母图层。

节点 节点内路由逻辑 节点间路由 模块路由 节点输出 主链继续往下

III 期当前总链

这张主图把 `节点`、`节点间路由`、`节点内模块路由成熟度` 画在同一条链上。 已经正式写成规格的地方直接标明；还没有单拆的地方，不再假装已经打通。

主链只看一条：`用户输入 -> 节点 -> 节点内路由逻辑 -> 节点输出 -> 节点间路由 -> 下一个节点`。模块不是平行链，只在部分节点内部被路由和处理。

Node 01

需求对齐回执层

模块路由可跑

先锁真实场景、一级稿型、主二级稿型、主样稿母本；现在也开始在这里锁主线与挂载模块。

1 输入

原始任务包 + 当前对任务的理解。

→
2 节点内路由

先判真实场景、一级稿型、主二级稿型、主样稿母本，并分 `PASS / CORRECTION / BLOCK`。

→
3 模块路由

这里开始锁主线与挂载模块，但 `主线模块` 口径仍需收紧。

→
4 节点输出

极短回执 + 主路线锁定结果。

这一层在干什么

在动笔前先冻结“这到底是什么任务”，不让后面在错方向上写得更认真。

当前已写到哪里

已有正式路由逻辑，能分 `PASS_TO_FREEZE / CORRECTION_LOOP / HARD_BLOCK_MINIMAL`。

当前缺口

模块已经进入回执，但主线和模块的命名边界还要再收紧，不然容易双编码。

为什么重要

这层不稳，冻结、资产、合同、返修都会在错目标上继续施工。

R-01

`需求对齐回执层 -> 输入冻结层`。只管这一跳：够稳才放行，不稳就纠偏或最小 blocker。

模块路由半成型

当前已写到哪里

整稿冻结已经成型，模块冻结已被明确提出：必吃材料、不该吃的材料、样稿边界。

当前缺口

还没有一份完整冻结规格说明“缺什么可降级、缺什么必须 blocker、冻结产物怎么写”。

待单拆

首稿。

四个词，先钉死

这部分用来防止继续混词。这里不再把 `R-01` 叫成全流程，也不再把模块误看成一条平行主链。

节点

一个稳定职责点，例如需求对齐回执层、输入冻结层、资产路由层。节点负责吃输入、做判断、产出稳定结果。

是“站点”
节点内路由逻辑

节点自己内部怎么判断、怎么分支、怎么放行或打回。模块层补丁本质上就是在给多个节点补这一层。

是“站内过闸规则”
节点间路由

两个节点之间怎么交接。`R-01` 只管“需求对齐回执层 -> 输入冻结层”这一跳，不是整稿总流程。

是“节点之间怎么过”
模块路由

模块在哪些节点成立、吃什么材料、怎么挂、怎么限、怎么判抢主线。它嵌在节点内部，不是平行主链。

是“节点内部第二维”

当前建设状态

这里不再泛说“体系还可以继续细化”，而是直接把当前已经写成规格的地方、只写到原则的地方、 以及还停在母图层的地方摆出来。

节点总数

当前主链按 `8` 个关键节点理解；如果把“任务进入池”算上，就是 `8 + 1` 个入口节点。

母图主链已固定
正式节点间路由

目前只有 `1` 条已经正式写成文件：`R-01`，也就是 `需求对齐回执层 -> 输入冻结层`。

其余仍待单拆
模块路由已可跑节点

当前已经满足“用模块去跑节点逻辑”的，是 `3` 个：回执层、合同编译层、门禁层。

这是现在最硬的部分
当前最大缺口

不是再补一个模块，而是把 `资产路由层` 和 `门禁后的返修路由` 补到跟模块层同样的粒度。

最大空档仍在中游与下游交接处
这版 HTML 服务的是“节点、节点内路由逻辑、节点间路由、模块路由成熟度”的现状可视化。 后续继续改时，优先补真实状态，不要再把局部路由误写成整稿总流程。
