### 4.1 方法分工

| 方法 | 主要负责验证什么 | 不负责什么 |
| --- | --- | --- |
| 扩大样本量 | 候选最小单元、候选范围分支、候选组合方式是否在新样本中稳定重复出现 | 不能单独把对象升格为正式资产 |

| 范围筛选题 | `写作目的 * 人格 = 写作范围` 是否能在同材条件下稳定分流 | 不能替代接线验证 |
| 大纲重建题 | `内容组合 * 逻辑组合 = 文章大纲` 是否能在给定范围下稳定推出 | 不能单独证明整稿质量 |
| 组合重建题 | 规则与组合是否真能跨题重用 | 不处理最终成稿表达质量 |
| 闭卷生成题 | 现有规范是否真的可写、会写 | 不能替代路由验证 |
| 对抗题 | 系统是否抗噪声、抗误路由、抗假模式 | 不能替代基础冻结 |

3. 证明当前主链不只是静态能解释资产，还能在限定题型里稳定完成：
   - 回拆
   - 接线
   - 范围筛选
   - 大纲重建
4. 为后续更接近真实任务的验证打基础：先把当前关键阻塞位补到可回拆、可正推、可重建、可失败回指，再推进到更真实的题型阶段。

#### 5.2.2 实验方法

1. 公式位盘点法：
   - 先按 `15` 个公式位逐位判断它当前属于 `已冻结 / 待组合 / 候选 / 缺口`
   - 只把会直接影响首批对象收口和上题能力的缺口列为本轮优先补位对象
2. 候选补位法：
   - 允许针对缺内容、缺规则、缺计算位先提出候选推理
   - 候选推理只能先作为待验证判断，不得直接写成正式收口
3. 扩样验证法：
   - 用新增 `A/B/C` 类样本检验候选对象或待组合对象是否稳定重复出现
   - 扩样结果优先服务于判断“这个位要不要继续保留、拆分或写死”
4. 题型验证法：
   - 拆解题主看当前主链能否稳定回拆
   - 接线题主看上层资产能否被同一组下层关系稳定接住
   - 范围筛选题主看 `写作目的 * 人格 = 写作范围` 是否能稳定分流
   - 大纲重建题主看 `内容组合 * 逻辑组合 = 文章大纲` 与 `写作范围 * 文章大纲 = 有效大纲` 是否闭合
5. 抽样复核法：
   - 对 `I期` 已冻结上层资产做有限抽样，不做 `IIF` 式 `32` 篇逐篇复验
   - 抽样复核只回答“现有冻结口径是否还能成立”，不顺手扩大验证范围
6. 收口裁决法：
   - 任何正式结论都必须来自“扩样证据 / 题型证据”中的至少一类直接证据
   - 如果新结论会直接推翻 `I期` 冻结口径，立刻改走 `II期` 回退链，不在 `IIA` 内偷偷反写 `I期`

### 5.3 `IIB` 实验方案

本节是 `IIB` 的正式执行底稿；写的是 `IIA` 结束后下一段该怎么补，不用它本身替代执行记录。

#### 5.3.1 实验目标

1. 把当前还未显式共享落表的两个尾项，补成共享正式资产：
   - `最小有效内容单元`
   - `最小有效逻辑单元`
2. 不再只停留在“公式 `7/8` 的结果位已经稳定”，而是把当前已写死路线反压回公式 `3~6`，确认哪些中间位可以被单独写成共享规范。
3. 通过单路与双路并行重建，判清当前哪些有效单位是：
   - 真正跨样本、跨路线可复用的共享资产
   - 只对某一范围或某一支有效的局部资产
   - 仍只能留在候选或证据位的对象
4. 为 `IIC` 的闭卷生成准备直接前置条件：把中间层补到能稳定回拆、稳定重建、稳定失败回指，而不是继续只靠上层结果位运行。

#### 5.3.2 实验方法

1. 路线反压法：
   - 以已经写死的路线为起点反推，不从抽象空想开始
   - 当前优先反压的路线分三组：
     - 已闭合单路对照组：`RANGE-01 / RANGE-02 / RANGE-04`
     - 新补平单路组：`RANGE-03 + OUT-03 + E-OUT-03`
     - 新补平双路组：`RANGE-05 / E-OUT-05` 的 `结果边界澄清` 与 `患者选择沟通`
2. 内容组合重建法：
   - 给定范围位与已成立上层结果，重建它实际依赖的有效内容输入
   - 核对这些输入能否稳定回收到同一组 `内容组合`
3. 逻辑组合重建法：
   - 给定范围位与已成立上层结果，重建它实际依赖的有效逻辑输入
   - 核对这些输入能否稳定回收到同一组 `逻辑组合`
4. 对照压缩法：
   - 先看哪些单位是多路线共享底盘
   - 再看哪些单位只在特定范围内才“有效”
   - 不把“常出现”直接等同于“最小有效”
5. 失败敏感法：
   - 某候选有效单位若被拿掉、替换或错路由后，稳定导致重建失败、降级或串错路，才有资格进入“有效”层判断
   - 如果拿掉后结果不变，它就不能被写成该位的必要共享资产
6. 定向补样法：
   - `IIB` 首轮不把扩样当启动前置
   - 只有当组合重建无法判清“共享有效”与“局部有效”的边界时，才允许补定向样本，并回写 [02_扩样结果.md](../02_阶段实验结果/II期/02_扩样结果.md)
7. 正式入表阈值：
   - `最小有效内容单元` 或 `最小有效逻辑单元` 要进入正式共享口径，至少要同时满足：
     - 在已写死路线里可被重复重建
     - 边界可说清：它对哪类范围有效、对哪类范围无效
     - 能解释当前结果位，不与现有共享母表冲突
   - 如果只满足“像是有用”，但边界说不清，就继续留在候选或证据位

### 5.4 `IIC` 实验方案

本节是 `IIC` 的正式执行底稿；写的是在 `IIB` 已补平中间层尾项后，如何正式推进“闭卷生成与抗过拟合”，不用它本身替代执行记录。

#### 5.4.1 实验目标

1. 在去作者标签、去原文顺序、去模板提示的前提下，验证当前已冻结资产能否只凭处理后材料包稳定生成：
   - 范围内大纲
   - 整篇成稿
   - 可供复核的原始输出留痕
2. 在 `D类` 不齐整或含噪素材包下，验证系统是否具备：
   - 抗过拟合
   - 抗误导项
   - 抗假承重点
   - 失败回指能力
3. 明确 `IIC` 的主测项是“会不会写、会不会被噪声带偏”，不是开放全路线自由竞争；凡路线本身拿不准，不在本阶段偷做额外路由实验。
4. 为后续更接近真实任务的生成验证提供直接证据：证明当前主链不仅能解释资产，还能从材料稳定出结构、从结构稳定出文。
5. 在不预设结论的前提下，把原藏经阁 `L1 / L2.5 / L3 / L4` 的承接方式纳入 `IIC` 初步评估：判断它们是应按 `8` 公式逻辑继续重建，还是更适合作为素材索引、证据映射、补材接口或外部知识入口；本轮只做初评，不抢最终冻结。

#### 5.4.2 实验方法

1. `C类` 闭卷生成法：
   - 首轮只使用 `C类` 齐整素材包
   - 题面固定去掉作者标签、原文顺序、标题线索与模板提示
   - 只保留材料卡、必要任务约束与当前锁定范围
2. 锁定范围法：
   - `IIC` 首轮不做开放全路线竞争
   - 单路线题目直接锁定到已写死范围
   - 双路线题目最多只允许在已写死的同一家族双支里判断，不再开放到全池竞争
3. 分层生成法：
   - 每道闭卷题内部可以先看“大纲是否成立”
   - 但凡被标为“外部正式闭卷题 / 阶段冻结依据 / 会写证据”的题目，必须在同轮推进到整篇成稿
   - 同轮必须保留系统原始输出；缺少原始输出留痕的回合，只能算局部验证，不算整稿闭卷证据
4. `D类` 加噪对抗法：
   - 在材料包中混入噪声卡、冲突卡和伪承重点
   - 先看系统能否识别主承重与误导项
   - 再看它能否在不被带偏的情况下继续生成，或明确宣告失败并说明原因
5. 失败回指法：
   - `IIC` 的失败不能只写成“写不出来”
   - 必须尽量回指到：
     - 哪个有效内容输入缺失或被噪声覆盖
     - 哪个有效逻辑输入没有成立
     - 还是材料本身不足以支持安全生成
6. 最小补样法：
   - `IIC` 不把扩样当默认入口
   - 只有在“生成失败”与“材料不足”边界判不清时，才允许补最小量定向样本，并回写 [02_扩样结果.md](../02_阶段实验结果/II期/02_扩样结果.md)
7. 下层承接评估法：
   - 以当前已入表的 `L1 / L2.5` 与尚未进入现行主链冻结口径的 `L3 / L4` 为对象
   - 只做“是否需要按 `8` 公式逻辑重建”的初步评估，不抢最终结论
   - 当前优先回答三件事：
     - 哪些层更像主链重建对象
     - 哪些层更像素材索引 / 证据映射 / 补材接口
     - 当闭卷题素材不全时，应优先向哪一层、外部网络或学术期刊补材
8. 通过阈值：
   - 闭卷条件下能稳定生成范围内大纲，不依赖原文顺序或模板提示
   - 关键样段与已冻结内容位 / 逻辑位不冲突
   - 遇到噪声、冲突、伪承重点时，能识别主承重，且不被稳定带偏
   - 一旦失败，能稳定回指到内容位、逻辑位或材料缺口，而不是只给结果态抱怨

#### 5.4.3 题面样例

1. 闭卷生成题样例 `A`：`RANGE-03` 人物真相线

   ```text

   给你一组去作者标签、去原文顺序、去模板提示后的材料卡。

   1. 用一句话写出本文判断；
   2. 给出 `4` 段文章大纲，并标明每段主承重材料；
   3. 写出整篇文章；
   4. 说明为什么应采用“先设人物谜题，再沿人物线 / 问题线回钻”的写法。

   限制：
   - 不得调用原文标题、原文段落顺序、作者习惯句式；
   - 不得补造材料包中不存在的事实；
   - 若材料不足以支撑判断，必须明确指出缺口。
   ```

2. 闭卷生成题样例 `B`：`RANGE-04` 工程解释线

### 5.5 `IID` 实验方案

本节是 `IID` 的正式执行底稿；写的是在 `IIC` 已结束、白盒管线架构诊断完成后，如何围绕因果命题"在不看参考答案的情况下，白盒系统因为调用了 `8` 公式，所以能写出高质量稿"设计证明框架、审计公式缺位、清理无效证据、冻结 A/B 对照协议，不用它本身替代执行记录。

#### 5.5.0 当前触发依据与架构诊断

1. `IIC` 两道 `C类` 外部闭卷题都已判失败；两道 `D类` 加噪题只算诊断轮，不计正式通过。
2. `Round 2` 白盒局部 run 已直接显示：`genre + core_data_points + required_structure` 能让 `sindi / ctad_lb13 / zolbetuximab` 分别提升 `+43 / +51 / +36`，说明当前已有可迁移的有效提分点。
3. 用户本轮已完成人工终审并确认 `4` 篇通过稿质量确实很高。
4. **架构诊断结论**：当前白盒管线是"约束清单 + LLM"，不是"公式驱动 + LLM"。
   - 管线实际只编码了 `2/15` 个公式位：`purpose`（公式 `1` 输出）、`required_structure`（间接大纲）
   - 缺失 `13` 个公式位：`write_target`、`arrangement_rules`、`persona`、`scope`、`min_content_units`、`min_logic_units`、`effective_content_units`、`effective_logic_units`、`content_combination`、`logic_combination`、`combination_rules`（内容/逻辑各一）、`valid_outline`
   - 管线没有公式合规评分维度；scoring 只对照参考答案，不检查公式推导链是否完整
   - 因此即使接通 LLM，管线也不能"正确调用 `8` 公式"；它只能调用约束清单里的残余碎片
5. **证据失效判定**：
   - `Q2`（`ctad_lb13_c`）/ `Q3`（`compassion15_os_c`）：全貌文件由 Agent 直接写成，不经白盒管线出稿；不存在真实 runtime 产物，不能作为白盒能力证据
   - `Q1`（`sindi_os_c`）：真实白盒 runtime 尝试存在，但 LLM 调用失败后由 `total_control` 兜底出稿；只能证明"闭卷白盒能跑且部分约束进了 prompt"，不能证明"`8` 公式被正确调用"
6. **结论**：当前没有任何一份样本能同时满足"真闭卷 + 真 `8` 公式调用 + 真高质量"三个条件。`IID` 的核心任务不再是"继续在现有管线上跑题"，而是先审计公式缺位、清理无效证据、冻结 A/B 对照协议，为 `IIE` 搭建和 `IIF` 正式验证提供精确输入。

#### 5.5.1 因果命题与四步证明框架

1. **核心因果命题**：

   > 在不看参考答案的情况下，白盒系统因为调用了 `8` 公式，所以能写出高质量稿。

2. 这条命题包含三个可独立证伪的子句：
   - `闭卷`：出稿前不看参考答案（`reference_answer_visible_before_draft = false`）
   - `因为调用了 8 公式`：稿件质量的来源是 `8` 公式驱动，而不是 LLM 自身能力或其他约束
   - `高质量`：稿件通过 `IIC` 同口径评分 + `AI味儿/可交付性分` 双评
3. **四步证明结构**：

   | 步骤 | 证明什么 | 必要证据 | 缺步后果 |
   | --- | --- | --- | --- |
   | `Step 1` 真闭卷 | 系统出稿前确实没看参考答案 | 完整 runtime 产物链（`contract → materials → draft → score`），`reference_answer_visible_before_draft = false`，稿件由白盒管线真实生成而非 Agent 手写 | 无法排除答案泄露，后续步骤全部失效 |
   | `Step 2` 真 `8` 公式调用痕迹 | `8` 公式的 `15` 个公式位确实进入了执行链 | 每题一份公式调用映射表：哪个公式位 → 进入了哪层合同 → 变成了 `draft_prompt` 的哪条约束 → 变成了 `review` 的哪条检查项 | 无法证明"因为 `8` 公式"，最多只能证明"闭卷白盒能写" |
   | `Step 3` 真高质量 | 输出稿达到可交付标准 | `IIC` 同口径五维评分（`route_alignment / key_facts / audience_style / structure / hallucination_control`）+ `AI味儿/可交付性分` + `delivery_adjusted_total` + `qualified` | 无法证明"高质量"成立 |
   | `Step 4` A/B 对照 | 质量来自 `8` 公式而非 LLM 自身能力 | A 组（白盒 + `8` 公式）vs B 组（同模型、同材料、同闭卷、去掉 `8` 公式约束）；两组同题同评；A 组显著优于 B 组 | 最多只能证明"白盒闭卷能不能写高质量稿"，不能证明"`8` 公式有用" |

4. **A/B 对照设计**：
   - `A 组`（实验组）：白盒管线 + `8` 公式全量约束 → 出稿 → `IIC` 同口径评分 + `AI味儿`评分
   - `B 组`（对照组）：同一 LLM、同一题目材料包、同样闭卷条件 → 仅给基础任务描述（题型 + 材料 + 字数），不给 `8` 公式推导出的 `purpose / scope / effective_units / outline` 等约束 → 出稿 → 同口径评分
   - 两组必须同题同跑，不得 A 组跑患者向、B 组跑机制科普
   - B 组不是"什么都不给"，而是"给正常写作任务描述但不给公式推导约束"
   - 评分人/评分脚本对 A/B 盲评，不得先知道哪组是实验组
5. **裁决规则**：
   - 四步全过：因果命题成立
   - `Step 1~3` 过但 `Step 4` 不过（A/B 无显著差异或 B 组反超）：`8` 公式无增益，白盒自身能力足够或不足够另论
   - `Step 1` 或 `Step 2` 不过：当前不具备证明条件，回退 `IIE` 补基础设施
   - 任何一步出现 Agent 代写、答案泄露、缺 runtime 产物：该题作废，不计入样本

#### 5.5.2 实验目标

1. 在 `IIE` 完成 `8` 公式管线搭建后，用因果证明框架判定核心命题是否成立。
2. 每题必须同时产出 `Step 1~4` 所需的全部证据，不得选择性报告。
3. 判清当前管线与 `8` 公式之间的真实因果关系，而不是"白盒能不能跑通"。
4. 把 `IIC` 暴露的失败样式按 `8` 公式缺位归因，为 `IIE` 提供精确施工清单。
5. 产出 `8` 公式 `15` 位在当前管线中的缺失映射表，至少写清：
   - 每个公式位当前是已编码、部分编码、还是完全缺失
   - 每个缺失位应进入管线的哪一层（`contract / materials / draft_prompt / scoring`）
   - 哪些位必须在 `IIE` 补齐后才能开跑 `Step 2`
6. 明确禁止在管线未补齐公式位的情况下，用现有"约束清单"结果冒充"公式驱动"结果。

#### 5.5.3 实验方法

1. 公式缺位映射法：
   - 对 `8` 公式 `15` 个公式位逐位审计当前管线代码
   - 每位判定为：`已编码（代码路径+变量名）`、`部分编码（有残余但不完整）`、`完全缺失`
   - 产出一张 `15` 位 × `4` 层（`contract / materials / draft_prompt / scoring`）的缺位矩阵
   - 该矩阵直接作为 `IIE` 施工输入
2. 失败归因重分类法：
   - 把 `IIC` 所有失败样式重新按 `8` 公式缺位归因
   - 不再只写"成稿不好""材料不够"，改写"公式 `X` 的 `Y` 位缺失导致 `Z` 失败样式"
   - 归因结果直接挂到缺位矩阵
3. 证据失效清理法：
   - 把 `Q2 / Q3` 从正式证据链移除；在状态卡标注"`Agent 代写，非白盒 runtime，不计入因果证据`"
   - 把 `Q1` 降级为"部分证据"；标注"`白盒 runtime 存在但 LLM 调用失败，total_control 兜底，只证明管线可启动`"
4. A/B 对照预设计法：
   - 在 `IIE` 补齐公式位前，先写死 A/B 两组的执行协议：
     - A 组 prompt 模板（含公式推导约束）
     - B 组 prompt 模板（去掉公式推导约束，只留基础任务描述）
     - 评分盲评流程（打分者不知道哪组是 A/哪组是 B）
     - 最小有效样本量（至少 `3` 题同时跑 A/B，不得只做 `1` 题）
   - A/B 协议必须在 `IIE` 搭建完成后、正式开跑前冻结，不得边跑边改
5. 公式调用痕迹留存法：
   - 每题出稿后，必须同步产出一份 `formula_trace.md`，记录：
     - 每个公式位的输入值
     - 每个公式的推导过程
     - 每个公式输出如何编码进 `contract / draft_prompt / scoring`
     - 哪些公式位在本题被跳过及原因
   - 缺少 `formula_trace.md` 的题一律按 `Step 2 不过` 处理
6. 双分 + 一闸评分法（沿用原 `IID` 已有规则）：
   - `IIC同口径内容分`：`weighted_total / qualified / route_alignment / key_facts / audience_style / structure / hallucination_control`
   - `AI味儿 / 可交付性分`：`ai_delivery_score`
   - `delivery_adjusted_total = round(iic_weighted_total * 0.6 + ai_delivery_score * 0.4)`
   - A/B 两组用完全相同的评分口径
7. 原始资料直读先行法（沿用原 `IID` 已有规则）：
   - `raw_source_analysis.md` 属于开跑前工程补充，不属于主实验变量
   - 每题开跑前都必须先把原始资料读成合格的 `raw_source_analysis.md`
   - 任一题若从 `gold / 成稿 docx / 参考样稿` 取材，该题直接按无效处理
8. 通过阈值：
   - `8` 公式 `15` 位缺位矩阵已产出，且已与 `IIE` 施工清单对齐
   - `IIC` 失败样式已全部按公式缺位重新归因
   - `Q2 / Q3` 已从证据链移除，`Q1` 已降级标注

   - `IIE` 首批施工包已锁定

5. `Q2 / Q3` 不得以任何形式被引用为白盒因果证据。
6. `Q1` 只能作为"管线可启动"的工程证据，不能作为"`8` 公式有效"的因果证据。

10. 原 `IID` 的 `raw_source_analysis.md` 前置要求继续生效：每题开跑前必须先有合格的原始资料解读稿。

### 5.6 `IIE` 实验方案

本节是 `IIE` 的正式执行底稿；写的是如何把 `8` 公式 `15` 个公式位真实编码进白盒管线，使 `IID` 的 A/B 因果对照具备开跑条件，不用它本身替代执行记录。

#### 5.6.1 实验目标

1. 把 `IID` 产出的 `15` 位缺位矩阵逐位补齐，让白盒管线从"约束清单 + LLM"升级为"公式驱动 + LLM"。
2. 补齐后的管线必须能：
   - 从 `write_target × arrangement_rules` 自动推出 `purpose`
   - 从 `purpose × persona` 自动推出 `scope`
   - 从 `scope × min_content_units / min_logic_units` 自动推出 `effective_content_units / effective_logic_units`
   - 从 `effective_units × combination_rules` 自动推出 `content_combination / logic_combination`
   - 从 `content_combination × logic_combination` 自动推出 `outline`
   - 从 `scope × outline` 自动推出 `valid_outline`
3. 每个公式位必须能在 `contract / materials / draft_prompt / scoring` 至少一层被消费，并留下可审计的 trace。
4. 补齐公式合规评分维度，使 scoring 不仅对照参考答案，还检查公式推导链是否完整。
5. 把 `8` 公式的成稿表达线一并补入：让系统从"会推导"升级为"会写"。
6. 产出 B 组对照 prompt 模板：精确定义"去掉 `8` 公式约束后，保留什么、去掉什么"。

#### 5.6.2 实验方法

1. 接口驱动搭建法：
   - 所有搭建都必须对着 `IID` 的缺位矩阵补，不允许新发明平行概念
   - 每补一个公式位，必须同时写清它进入了管线的哪一层、变量名是什么、被谁消费
2. 公式编码法：
   - 把 `8` 公式已冻结结果编码成机器可读资产（`JSON / YAML / Python config`）
   - 编码结果只服务执行链，不另起新的理论层
   - 每条公式的输入位、输出位、推导规则必须写成可程序化执行的格式
3. A/B prompt 差分法：
   - A 组 prompt 必须显式包含每个公式位的推导输出
   - B 组 prompt 必须精确去掉这些推导输出，只保留基础任务描述
   - 差分必须可审计：给定 A 组 prompt，能精确说出删掉哪些行/段就变成 B 组
4. 回归验证法：
   - 每补完一批公式位，用 `IIC` 已知失败题做回归
   - 回归只回答"新补的公式位是否修复了对应的失败样式"
   - 回归结果只算搭建证据，不算正式因果证据（正式证据在 `IIF` 的 A/B 对照中产出）
5. 成稿表达线补位法：
   - 先补 `L1 / L2.5` 的成稿表达线，让系统不再只会解释、不会成稿
   - 再补证据抽取 / 证据映射线，让数字、图表值与证据落卡稳定进入正文
   - 只有在前两条仍无法解决问题时，才把 `L3` 中直接服务主链的部分选择性接回
6. 通过阈值：
   - `15` 个公式位在管线 `4` 层中的编码覆盖率 >= `12/15`（允许 `3` 个低优先级位暂缓）
   - 公式合规评分维度已上线
   - A/B 两组 prompt 模板已冻结且差分可审计
   - 至少 `2` 道 `IIC` 失败题的回归结果显示对应失败样式已被修复
   - `formula_trace.md` 模板已定型且可自动生成

### 5.7 `IIF` 实验方案

本节是 `IIF` 的正式执行底稿；写的是在 `IIE` 完成 `8` 公式管线搭建后，如何用正式 A/B 对照裁决因果命题并做 `II期 -> III期` 升期判断，不用它本身替代执行记录。

#### 5.7.1 实验目标

1. 用正式 A/B 对照回答核心因果命题："在不看参考答案的情况下，白盒系统因为调用了 `8` 公式，所以能写出高质量稿。"
2. 每题必须同时完成四步证明（`Step 1~4`），不允许跳步或选择性报告。
3. 以最小正式题组完成因果裁决与 `II期 -> III期` 升期判断。
4. 如因果命题被反证（A/B 无差异或 B 组反超），必须正视而不是继续加题硬刷。

#### 5.7.2 实验方法

1. 正式 A/B 对照法：
   - 必须使用 `IIE` 已冻结的 A/B prompt 模板
   - 最小正式题组：`3` 题同时跑 A/B（`1` 道患者向 + `1` 道机制科普 + `1` 道加噪题）
   - 每题产出：A 组稿件 + B 组稿件 + 两组评分 + `formula_trace.md`（A 组专属）
   - 评分盲评：打分前不告知哪组是 A/哪组是 B
2. 四步同步验证法：
   - 每题同步验证 `Step 1~4`
   - `Step 1`（真闭卷）：检查 runtime 产物链完整性，确认 `reference_answer_visible_before_draft = false`
   - `Step 2`（公式痕迹）：检查 `formula_trace.md`，确认 `15` 位中已编码位全部有 trace
   - `Step 3`（质量）：`IIC` 同口径评分 + `AI味儿/可交付性分`
   - `Step 4`（A/B 对比）：比较 A/B 两组的 `delivery_adjusted_total` 与各维度分
3. 裁决法：
   - A 组在 `3` 题中至少 `2` 题同时满足：`qualified = true` 且 `delivery_adjusted_total` 显著高于 B 组（差值 >= `10` 分）→ 因果命题成立
   - A 组 `qualified = true` 但与 B 组无显著差异 → `8` 公式无增益，白盒自身能力足够
   - A/B 两组都 `qualified = false` → 当前系统整体未达标，回退 `IIE`
   - B 组反超 A 组 → `8` 公式可能引入负约束，需诊断
4. 回退法：
   - 任一核心题 `Step 1` 或 `Step 2` 不过，不继续硬跑，直接回退 `IIE` 修管线
   - A/B 两组都失败时，不在 `IIF` 里加题硬刷，回退 `IIE` 重新诊断
5. 通过阈值：
   - `3` 题 A/B 对照全部完成且 `Step 1~4` 留痕齐全
   - A 组至少 `2/3` 题 `qualified = true`
   - A 组相对 B 组有可审计的质量增量
   - `formula_trace.md` 证实公式位确实被消费
   - 因果裁决结论已写入正式文件

   - [00_总留痕.md](../03_过程强留痕/00_总留痕.md)

   - [04_阶段结论.md](../02_阶段实验结果/II期/04_阶段结论.md)
   - [00_总留痕.md](../03_过程强留痕/00_总留痕.md)
5. 任何中断、跑偏、新意外，先回写 [00_状态卡.md](../02_阶段实验结果/II期/00_状态卡.md)；如果影响边界，再补写本方案；最后回写 [00_总留痕.md](../03_过程强留痕/00_总留痕.md)。
6. 任何关键判断、边界变化、异常发现、修正决定，不得等到收尾再补；一旦形成，就立刻回写对应正式文件与 [00_总留痕.md](../03_过程强留痕/00_总留痕.md)。
7. 如果 `II期` 发现 `I期` 冻结结果存在硬错，不得直接改写 `I期` 共享冻结表；先回写 [04_阶段结论.md](../02_阶段实验结果/II期/04_阶段结论.md)、[00_状态卡.md](../02_阶段实验结果/II期/00_状态卡.md) 与 [00_总留痕.md](../03_过程强留痕/00_总留痕.md)，再由总协调裁决是否反写 `I期`。
8. 任何只存在于聊天、未写入正式文件和 [00_总留痕.md](../03_过程强留痕/00_总留痕.md) 的判断，一律按未发生处理。
9. 任何一次补救动作，只要未来可能再次发生，就必须在本轮结束前沉淀成自动规则，不能只修这一次。
10. 每条新补救规则至少写清：触发条件、自动动作、正式落点、升级条件。
11. 同类事故再次出现时，不再重复写零散说明，而是升级既有规则。
12. 任何活跃过程文档超过 `1500` 行后，必须先原地压缩再继续追加；每压缩一次都要更新文件头部的 `压缩轮次：x/6`。
13. 任一过程文档累计压缩达到 `6` 次后，当前文件封存，在同目录按序号递增新建下一卷继续。
14. 如果被封存的是当前活跃链引用的过程文档，必须同步改掉活跃入口链接，确保新 session 永远先落到最新在写卷。
15. 主执行 session 必须持续判断自己的上下文质量；出现混口径、误判当前阶段、反复漂移、重复劳动或明显依赖聊天记忆时，必须主动提醒用户切换 session。
16. 凡 `IIC` 外部正式闭卷题 / 阶段冻结依据 / 会写证据题，结果卡必须同时保留 `系统原始输出` 与 `系统输出摘要`；整稿题缺任一项，一律按证据不完整处理，不得报“已完成”“会写成立”或进入冻结判断。
17. 如果执行完成后才发现缺少整稿验收或原始输出留痕，按“验收口径漏检事故”处理：先在 [00_状态卡.md](../02_阶段实验结果/II期/00_状态卡.md) 止血降级，再在 [03_题型验证结果.md](../02_阶段实验结果/II期/03_题型验证结果.md) 改判证据级别；如影响边界，再补写本方案，最后回写 [00_总留痕.md](../03_过程强留痕/00_总留痕.md)；同类再发，升级为总协调问题。
16. 每次建议切换前，先回写 [00_状态卡.md](../02_阶段实验结果/II期/00_状态卡.md) 与 [00_总留痕.md](../03_过程强留痕/00_总留痕.md)。
17. 每次 session 切换后，默认要开一轮冷启动子 session 继承测试；如果主执行 session 不能可靠判断自己何时降智，则在当前活跃过程文档压缩轮次达到 `5/6` 时自动提醒用户切换。
