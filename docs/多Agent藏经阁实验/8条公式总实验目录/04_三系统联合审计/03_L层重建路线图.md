| 层 | 是否需要 8 公式重建 | 工作量 | 核心动作 |

## 一、L2.5 / L-NORM：8 公式机器编码（新建）

### 1.1 为什么需要这个

8 公式当前只存在于人类可读的 markdown 实验文档中。管线无法消费 markdown——需要一组结构化 JSON 资产。

### 1.2 需要编码的内容

```
writing_norms/
├── objectives.json         ← OBJ-01~08
├── arrangement_rules.json  ← ARR-01~07 + MAP-01~03
├── purposes.json           ← PUR-01~07
├── writing_ranges.json     ← RANGE-01~05（含 RANGE-05 双路）
├── content_units.json      ← 8 个最小内容单元
├── logic_units.json        ← 3 个最小逻辑单元 + 2 个范围专属
├── effective_content_map.json  ← 范围→有效内容单元映射表
├── effective_logic_map.json    ← 范围→有效逻辑单元映射表  
├── content_combination_rules.json  ← 5 条内容组合规则
├── logic_combination_rules.json    ← 5 条逻辑组合规则
├── content_combinations.json   ← 5 种内容组合
├── logic_combinations.json     ← 5 种逻辑组合
├── outlines.json              ← OUT-01~05
├── effective_outlines.json    ← E-OUT-01~05
└── formula_routing.json       ← 公式 1~8 的计算路由规则
```

### 1.3 编码原则

1. 每个 JSON 文件只编码一个公式位，保持与 markdown 文档的一一对应。  
2. 每个对象带 `id`、`name`、`description`、`status`（frozen / candidate / deprecated）。  
3. 映射表带 `range_id → [unit_ids]` 的显式关系，不留隐含推断。  
4. `formula_routing.json` 编码 8 条公式的输入 → 输出计算关系，供 Planning 阶段消费。

### 1.4 落点

这组资产应该落在 `藏经阁/l2/writing_norms/` 或 `publish/current/consumers/xiakedao/norms/`，取决于最终消费调度。但不落在实验文档目录。

## 二、L2 医学策略层重建

### 2.1 当前问题

L2 当前的 `m3_strategic_plays.json`（6 种策略）和 `narrative_arcs.json`（6 步弧）是旧体系的产物：

- 6 种策略 = 监管承认 / 监管重建 / 终点失利 / 机制新定义 / 证据缺口攻防 / 安全性差异化定位  
- 6 步弧 = HOOK → SETUP → TEARDOWN → SOLUTION → PROOF → CTA

这些与 8 公式的对应关系是松散的、不一致的：

| 旧资产 | 8 公式对应 | 差距 |
| --- | --- | --- |
| 6 plays | 5 内容组合 + 5 逻辑组合 | 粒度不匹配，plays 混合了内容和逻辑 |
| 6 步 arc | 5 文章大纲（OUT-01~05） | 旧 arc 是通用模板，8 公式大纲是范围专属的 |
| play_id 路由 | 8 公式公式 1~2 路由 | 旧路由靠 domain 关键词，8 公式靠目标×排列×目的×人格 |

### 2.2 重建动作

1. **替换路由入口：** 用 `formula_routing.json` 替换 `play_id + arc_id` 作为 Planning 的主路由。
2. **重编码 plays：** 把 6 个 plays 映射到 8 公式的 5 种内容组合 + 5 种逻辑组合，保留不能被 8 公式覆盖的独特价值（如合规攻防策略），降级为辅助参考。
3. **重编码 arcs：** 把 6 步 arc 映射到 5 种文章大纲（OUT-01~05），保留 arc 的渲染层价值（段落衔接、节奏控制），降级为 RenderPlan 的可选输入。
4. **保留无须重建的资产：**
   - `m5_compliance_schema.json` → 继续作为 Quality 阶段输入
   - `medical_syntax_rules.json` → 继续作为 Writing 阶段输入
   - `sentence_patterns.json` → 继续作为 Writing 阶段输入
   - `persona_medical_overlay.json` → 继续作为 L1 人格的医学扩展
   - `m4_medical_blocks.json` → 继续作为修辞执行工具

### 2.3 与写作 DNA SPEC 的对接

写作 DNA SPEC 要求新增：
- `task_intents.json` → 可用 8 公式 objectives.json 替代
- `evidence_archetypes.json` → 8 公式当前未覆盖，建议保留 SPEC 的设计放在 L3
- `reasoning_contract_operators.json` → 可用 8 公式 content_combination_rules.json + logic_combination_rules.json 替代

融合方案：以 8 公式冻结资产为主编码，SPEC 里 8 公式未覆盖的部分（evidence_archetypes）从 L2 移到 L3 作为领域层辅助。

## 三、L1 写作心法层桥接

### 3.1 需要做的

L1 的 16 份资产本身质量没问题，但缺少被 8 公式路由选中的接口。具体需要加的是：

1. **人格路由接口：** 让 8 公式的公式 2（`写作目的 * 人格 = 写作范围`）能够从 `persona_kernels.json` 里选择人格，并与 `m0_persona_templates.json`（L2）做叠加。
2. **表达选择接口：** 让 8 公式的"写作范围"能决定从 `expression_base.json` + `register_levels.json` 选择什么语域层级。
3. **风格叠加接口：** 让 `lighthouse_*.json` 在有效大纲确定后叠加，而不是在路由阶段就介入。

### 3.2 不需要做的

- 不需要重写 `m1_argument_logic_rules.json`（论证逻辑规则仍然独立有效）
- 不需要重写 `m2_narrative_primitives.json`（叙事基本单元仍然独立有效）
- 不需要重写 `m4_rhetoric_blocks.json`（修辞块继续作为执行工具）
- 不需要重写任何 `lighthouse_*.json`

### 3.3 估计工作量

- 增加 2~3 个路由适配 JSON + 修改 AssetBridge 的选择逻辑
- 不涉及 L1 资产内容本身的重写

## 四、L3 疾病通识层清理

### 4.1 需要做的

1. **清模板残留：** `m1_thesis_derivation_rules.json` 中的 `template`、`template_variants`、`representative_title`、`recommended_m3_plays` 等字段必须清退（写作 DNA SPEC 已明确要求）。
2. **保留领域知识：** `disease_knowledge.json` 和 `source_registry.json` 继续作为背景知识供给。
3. **可选新增：** 
   - `evidence_interpretation_overlays.json`（不同证据类型在该疾病下的误读修正）
   - `question_families.json`（疾病领域稳定问题族）
   - `domain_boundaries.json`（判断禁区与前提条件）

这些新增项来自写作 DNA SPEC，与 8 公式正交——它们提供的是"关于这个疾病应该注意什么"，而不是"文章应该怎么组织"。

### 4.2 不需要做的

- 不需要按 8 公式重建 L3 目录结构
- 不需要把 disease_knowledge.json 改成 8 公式格式
- L3 和 8 公式的关系是：L3 提供领域输入 → 8 公式决定怎么用它

## 六、侠客岛管线改造

### 6.1 新增：范围检测阶段

在 Evidence 和 Planning 之间加一级 Range Detection：

```
输入：task_description + evidence_summary
输出：recommended_range（RANGE-01~05，含 RANGE-05 双路判断）
```

这一步实现 8 公式的公式 1~2：
- 从任务描述推断写作目标（OBJ）
- 从目标 × 排列规则推断写作目的（PUR）
- 从目的 × 人格推断写作范围（RANGE）

### 6.2 改写：Planning 阶段

**输入变更：**
- 旧输入：`product_id + domain + evidence_facts`
- 新输入：`product_id + domain + evidence_facts + recommended_range + effective_outline`

**路由变更：**
- 旧路由：从 L2 选 `play_id + arc_id`
- 新路由：从 L-NORM 选 `effective_outline + content_combination + logic_combination`

**输出变更：**
- 旧输出：`thesis + outline + play_id + arc_id`
- 新输出：`thesis + structural_outline + content_constraints + logic_constraints + coverage_contracts`

### 6.3 升级：Writing 阶段

**prompt 编译变更：**
- 旧方式：把 thesis + outline 编成 prompt，规则作为弱提示
- 新方式：把 8 公式的内容组合规则和逻辑组合规则编成 **硬约束** 注入 system_prompt，把有效大纲编成 **结构骨架** 注入 user_prompt

### 6.4 升级：Quality 阶段

**新增门禁：**

| 门禁 | 对应 8 公式验收 |
| --- | --- |
| `outline_alignment` | 初稿是否匹配选定的有效大纲 |
| `content_rule_compliance` | 是否遵守 5 条内容组合规则 |
| `logic_rule_compliance` | 是否遵守 5 条逻辑组合规则 |
| `effective_content_coverage` | 范围→有效内容单元映射是否被兑现 |
| `effective_logic_coverage` | 范围→有效逻辑单元映射是否被兑现 |
