## 一、最初主目标

**打造医学文章自动化写作系统**

| 维度 | 内容 |
|------|------|
| 输入 | 需求 + 材料 |
| 过程 | 系统有序调用各模块、各资产 |
| 输出 | 高质量文章 |

---

## 二、阶段0：六段链（原系统）

### 系统组成
- 侠客岛主编排器
- 藏经阁资产库

### 问题
**资产调取逻辑不对**

发现契机：新增大量写作系统最缺的资产后，调取时发现没有任何提高。

### 根因
调用逻辑黑盒，不可见、不可追溯。资产存在，不等于资产被调用。

---

## 三、阶段1：藏经阁通用储存模式实验（第一次重建）

### 目标
证明是否存在一种通用储存模式，能把好文章拆成可复用、可验证、可跨域重组的对象。

### 竞争假设

| 假设 | 内容 |
|------|------|
| H0 | 不存在通用模式，只能整包样板 |
| H1 | 存在，可存的是动作、构件、规则 |
| H2 | 当前对象不够，需补拓扑、触发条件等 |
| H3 | 通用模式只在某些稿型内成立 |

### I期结论
- 整包样板不适合做底层存储
- 最可信方向：动作、构件、规则
- 拓扑层很可能必须单列

### 后续失败暴露
**IIE 提前终止 -> IIF 最小单元审计**

失败原因：前置说法不是最小单元，而是有限组合。当用这些组合态去解释新样本时，会出现“看起来有规则，但实际无法稳定应用”的问题。

---

## 四、阶段2：8公式实验（第二次重建）

### 目标
把资产按最小单元拆分归类。

### 结果
**已完成**

### II期关键推进

| 子阶段 | 状态 |
|--------|------|
| IIA | 扩样验证完成 |
| IIB | 重建完成 |
| IIC | 闭卷生成失败，但失败点已暴露 |
| IID | 缺位审计完成 |
| IIE | 管线搭建完成 |
| IIF | A/B 对照验证完成 |

### IIF 裁决结果

| 维度 | 结果 |
|------|------|
| 因果性 | 不成立（Δ=+3.33 < 5） |
| 样稿增益 | 成立（Δ=+13） |
| 系统上限 | 有价值底稿（mean=76，0/3≥85） |

### 阶段结论
8 公式解决了“资产按最小单元拆分归类”的问题，但没有证明“只靠公式调用就能显著提分”。

这意味着：

1. 资产拆分问题已经基本解决。
2. 系统调用问题还没有解决。
3. 下一阶段必须从“资产怎么拆”转向“资产如何被系统稳定调用、验证、回指”。

## 五、阶段3：III期临床试验启动

### 启动原因
III 期不是从零开始另起一套系统，而是在 II 期结论基础上，正式转向解决最初那条一直没有真正闭环的问题：

**8 公式和藏经阁资产，怎样被系统稳定调用。**

换句话说：

1. 第一次重建主要在问“资产能不能被拆成通用对象”。
2. 8 公式主要在解“资产怎样按最小单元落表”。
3. III 期要解的是“这些对象怎样进入写作主链，变成可执行、可验证、可反推的系统动作”。

### 启动目标
把 8 公式和藏经阁资产，变成**可执行、可验证、可反推**的节点模块化写作系统。

### 启动后的主链

1. 任务进入池
2. 需求对齐回执层
3. 输入标准化与冻结层
4. 稿型识别与资产路由层
5. 大纲过闸与写作合同层
6. 首稿生成层
7. 审稿与门禁层
8. 定向返修层
9. 经验回流层

### 启动后的四层结构

| 层 | 职责 |
|---|------|
| 00_入口索引 | 冷启动入口、任务卡、真相源地图 |
| 01_散落经验裁决层 | 经验来源、裁决总表、待验证经验池 |
| 02_节点模块规则明细层 | 各节点工作版标准、通过/不通过、回流目标层 |
| 03_完整架构层 | 系统总图、关键节点路由表 |

## 七、关键理解

### 三次重建分别解决什么

| 重建 | 解决的问题 | 结果 |
|------|-----------|------|
| 第一次 | 如何储存资产 | 失败：说法不是最小单元 |
| 8公式 | 资产按最小单元拆分 | 成功：15 个公式位已显式落表 |
| III期 | 资产如何进入系统主链并被稳定调用 | 已启动：进入节点模块化建设 |

### 冷启动是执行纪律，不是 III 期主目标

冷启动接手能力的意思是：新 Agent 不靠聊天记录，也能正确接手 III 期临床。

它是 III 期建设必须满足的执行纪律和完备状态要求，但不是 III 期主目标本身。

III 期主目标仍然是：

**把 8 公式和藏经阁资产变成可执行、可验证、可反推的节点模块化写作系统。**

### 当前进度应放在哪

从这版开始：

1. 背景文档只负责历史与启动逻辑。
2. 当前进度统一回总入口和任务卡。
3. runtime 状态卡只承接过程验证状态，不替代正式背景文。
