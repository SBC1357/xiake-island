## 一、待验证条目

### V-01 情绪 / 逻辑 / 语气的量化评分锚点

当前想法：
- 可用 `0~100` 的方式给情绪、逻辑、语气做区间锚定

为什么暂不升正式规则：
- 目前只有方向，没有稳定量表
- 还缺足够多的正反案例

需要补什么：
- 至少 `1` 组情绪锚点案例
- 至少 `1` 组语气锚点案例
- 至少 `1` 组逻辑偏差案例

### V-02 开头 / 结尾的专门生成策略

当前想法：
- 开头和结尾因为天然缺一侧上下文，应该有独立生成与回修策略

为什么暂不升正式规则：
- 方向稳定，但具体策略还没定

需要补什么：
- 开头补写失败案例
- 结尾回收失败案例
- 一组稳定的补写法

### V-03 八公式递归调用的精确边界

当前想法：
- 八公式不是整篇一次性套用，而要在大纲层、模块层、局部编排层递归使用

为什么暂不升正式规则：
- 目前有共识，但“递归到哪一级”还没完全定死

需要补什么：
- 公式位与节点的逐项映射
- 递归触发条件
- 不该递归使用的边界

### V-04 多 Agent 并发微改的真实上限

当前想法：
- 并发只适用于完全独立的空缺位，默认串改优先

为什么暂不升更细规则：
- 当前“可并发到什么程度”还没有做足够多实验

需要补什么：
- 相邻句 / 同段 / 跨段修改的并改对照实验

### V-05 Agent 疲劳与长线程质量下降

当前想法：
- 人和 AI 在超长工作后都可能出现判断变钝、长阅读能力下降

为什么暂不升正式规则：
- 更像观察和直觉，缺可执行定义

需要补什么：
- 会话长度、返修轮次与错误率之间的对照数据

### V-06 工具故障与材料优先级切换纪律

当前想法：
- 只要已经拿到等价可读材料，就应立刻切回材料本身，不继续把时间耗在工具修理上

为什么暂不升正式规则：
- 方向是对的，但现在更像执行纪律，还缺足够多的写作场景案例来说明边界

需要补什么：
- 至少 `2` 组“继续折腾工具反而拖慢写作”的对照案例
- 至少 `1` 组“工具仍然必须优先修复”的反例
### V-07 DashScope coding plan 端点不兼容 compatible-mode（2026-04-10 首次验证）

当前想法：
- DashScope coding plan key（`sk-sp-*`）必须打在 `coding.dashscope.aliyuncs.com/v1`，不能打在 `dashscope.aliyuncs.com/compatible-mode/v1`（前者 200，后者 401/404）
- 同一个 key 在 coding 子域走 `/v1` 而非 `/compatible-mode/v1`

为什么暂不升正式规则：
- 仅 1 个 key 验证，不确定是否所有 coding plan key 都如此
- DashScope 端点策略可能随时间变化

已有证据：
- 4 轮端点测试直接验证（同轮 curl 记录在 runtime）
- 成功生稿 1 枪（lecanemab_patient 72.45/100）

### V-09 Preflight 真相源冲突检测需排除语境否定（2026-04-10 首次验证）

当前想法：
- 预检脚本扫描"真相源冲突"时，如果目标词出现在否定语境（如"不作为"、"不承担"），应排除而非报警
- 否则产生假阳性，阻塞 smoke 启动

为什么暂不升正式规则：
- 只有 1 个假阳性案例（`benchmark_runner.py` 在计划文档中的排除性说明）
- 需要更多 preflight 运行积累假阳性/真阳性比例

已有证据：
- preflight 首跑假阳性 → 修复 → 二跑 `ready_for_smoke=True`

- 至少需要 3 题以上才有最基本的分布参考

为什么暂不升正式规则：
- 方向明确但"几枪才有统计性"没有定量标准

已有证据：
