# III期总则与需求对齐回执规则

状态：整合稿 / 待继续补证  
日期：2026-04-11  
用途：收束 III 期正式主链的总则和最上游需求对齐回执规则，回答“什么条件下才允许系统往下走”  
边界：本文件只写适用边界、回执层职责和放行口径，不展开冻结、首稿、返修微操

## 证据摘要

1. 已直接核对 III 期母图与接口路由文档，确认需求对齐回执层已经是整个系统的上游硬闸门。
2. 已直接核对冷启动配置与稿型总表，确认回执层当前必须锁定的核心信息已经相对稳定。
3. 已直接核对仓库卫生最高禁令，确认“有多解先反向确认，没确认不准开工”已是仓库级硬约束。
4. 已直接核对本轮 IIIA 样稿 / prompt / 成稿偏航事实，确认上游必须允许“候选主题”合法存在，不能再把所有任务都强行提前锁成正式主题。

## 一、总则

III 期当前只吸收下面这两类内容：

1. 已经在现行方法文档里反复出现、可执行、可回指的规则
2. 已经过 `01_散落经验裁决层` 裁过，且状态为 `保留 / 收紧 / 合并` 的经验

下面这些内容不在本层直接冻结：

1. 单次逐字稿里的摇摆想法
2. 还没有稳定边界的实验猜想
3. 需要大量案例才能站稳的评分体系

这里的 `必须 / 默认 / 禁止` 都只表示：

1. 当前工作版先按这套规则执行
2. 不等于 III 期实验已经最终证明
3. 一旦后续挑战和案例不支持，仍然允许回退、收紧或改写

## 二、回执层是最上游硬闸门

没过需求对齐回执层，不准做下面这些动作：

1. 进入输入冻结
2. 进入稿型路由
3. 直接下发写作任务
4. 宣称“已经理解任务”

## 三、回执层最少必须锁住什么

当前最小必锁项：

1. 真实写作场景
2. 一级稿型
3. 主二级稿型
4. 主样稿母本
5. 当前交付物是新写、改稿，还是整理
6. 受众 / 平台 / 篇幅
7. 主角度 / 主问题（允许候选态）
8. 主证据链（允许候选态）
9. 必写点（允许候选态）
10. 禁区 / 不可漂移点

## 四、回执层只允许四种输出

1. 放行到输入冻结层（正式主题已基本成立）
2. 放行到输入冻结层，但标记为候选态（后续必须进入主题探索，不得直接写）
3. 回一次极短纠偏
4. 给最小阻塞说明

这里的“最小阻塞”只能要求：

1. 原始材料
2. 缺失的样稿全文
3. 冲突信息里的关键澄清

不允许要求用户先做：

1. 中间摘要件
2. 人工路由表
3. 写作合同
4. 预整理卡片

## 四点五、候选态放行纪律

当用户需求已足够锁住稿型，但正式主题尚未成链时，回执层允许放出“候选态”。

候选态的硬要求：

1. 必须明确写出：当前已锁住什么、尚未锁住什么。
2. 主角度 / 主问题、主证据链、必写点允许处于候选态。
3. 候选态只能放行到输入冻结与主题探索，不得直接下传写作合同。
4. 候选态必须显式带上：当前不确定点 + 下一轮最小扩证方向。

## 五、回执层的两条硬纪律

1. 有多解先反向确认，没确认不准开工。
2. 样稿或目标写法没全文直读前，不得宣称“已对齐”。

## 六、最容易犯的错

1. 还没锁真实写作场景，就急着进稿型。
2. 主二级稿型还在摇摆，就提前绑定样稿。
3. 把改稿任务误当成新写任务。
4. 用户已经纠偏过，系统又滑回旧理解。

### 7.2 需求对齐时最小补问规则

20样稿泛化验证发现以下输入缺口在回执阶段必须捕获：

| 缺口 | 最小补问 | 不补的后果 |
| --- | --- | --- |
| 多研究串联 | "这篇稿需要串联哪几个研究？有主次之分吗？" | g0卡无法区分主源和辅源，key_facts覆盖失控 |
| 企业管线信息 | "需要我查阅公司公开信息吗？核心看哪些产品/管线？" | 写作时拿不到商业数据，凭空编造企业细节 |
| KOL人格信号 | "这篇署名KOL是谁？写作风格/立场有参考吗？" | 文风无锚点，R级别判定偏差 |
| 框架预设 | "有没有系列文档、栏目模板或已有期数的风格参考？" | 系列文章连续性断裂 |
| 对比型结构 | "这篇需要对比几个药物/方案？对比维度是什么？" | 大纲过闸不知道按什么结构编排 |

### 7.3 A↔C↔D三向摇摆的补问策略

当系统在A/C/D三条路线间无法80%以上确信锁定时，反向确认用下面这一道补问：

> "确认一下：这篇稿的核心论证靠的是①某个研究的硬数据（OS/PFS/ORR等），②一个分子/通路的机制解释，还是③多个研究/事件串起来的行业趋势判断？"

- 用户答①→ Route-A
- 用户答②→ Route-C
- 用户答③→ Route-D
- 用户答"①+②都有"→ 看篇幅占比，追问"哪个占主体？"
- 用户说不清 → 按输入材料判断：有单一研究PDF→A，有机制文献→C，有多源/企业信息→D

## 八、标准件

这一节包含已升级为工作版的正式标准件，以及仍在预留位的标准件。

### 标准件标准

1. 标准件必须能挂到一个节点。
2. 标准件必须说清楚：解决什么问题、未来从哪来、当前是什么状态、现在能不能用、边界在哪里。
3. 标准来源优先级先按下面顺序挂：
   1. 仓库级红线
   2. 用户硬纠偏
   3. 门禁问题单 / 返修问题单
   4. 实战经验归档
   5. 样稿母本
   6. 8公式 / 15公式位
   7. 藏经阁 `L1 / L2` 资产
4. 只有来源可回指、节点落点明确，才允许从预留位升级成正式标准。

### 来源状态词

1. `待补来源`：还没有直接来源。
2. `有候选来源`：已有候选，但还没定。
3. `可回指`：能直接回到原始证据。
4. `已可用`：当前工作版可以直接引用。

### 标准件 S-00-01：需求对齐最小回执标准

| 字段 | 内容 |
| --- | --- |
| **标准件名称** | 需求对齐最小回执标准 |
| **所属节点** | 需求对齐回执层 |
| **要解决什么问题** | 防止 Agent 带着多解直接开工，防止没确认就进入冻结和写作 |
| **固定起点** | 用户给出原始任务描述（对话框文字 / 文件路径 / 附件，任意组合） |
| **固定终点** | 回执卡进入下面两种合格态之一：**A. 正式锁定态**（`10` 个字段全部明确）或 **B. 候选放行态**（`1~6` 与 `10` 已明确，`7~9` 允许标注为候选态，但必须写明当前不确定点与下一轮最小扩证方向） |
| **输入** | 用户原始任务描述 + 附件 + 样稿 + 任何补充上下文 |
| **输出** | 需求对齐回执卡，包含下面 `10` 个字段：① 真实写作场景 ② 一级稿型 ③ 主二级稿型 ④ 主样稿母本 ⑤ 交付物类型（新写 / 改稿 / 整理） ⑥ 受众 / 平台 / 篇幅 ⑦ 主角度 / 主问题 ⑧ 主证据链 ⑨ 必写点 ⑩ 禁区 / 不可漂移点。**正式锁定态**要求 `10` 字段均明确；**候选放行态**允许 `⑦⑧⑨` 标注为“候选中”，但每项必须写明当前候选内容、当前缺口和下一轮最小扩证方向。 |
| **通过标准** | 同时满足以下全部：(1) `1~6` 与 `10` 无一为空；(2) 用户收到回执后，未提出纠偏，或纠偏后重填已被用户确认；(3) 如主样稿母本已有全文，标注"已全文直读"；如无全文，标注"仅借用表层特征，未全文直读"；(4) 如为候选放行态，`⑦⑧⑨` 虽可未最终锁定，但必须写出候选内容、当前缺口和下一轮最小扩证方向 |
| **不通过标准** | 命中以下任一即不通过：(1) `1~6` 或 `10` 任一为空；(2) 用户提出纠偏后未重填；(3) 存在多解未做反向确认；(4) 样稿未全文直读却标注"已对齐"；(5) 把候选态当正式锁定态直接下传写作链；(6) `⑦⑧⑨` 标注为候选态但未写清当前缺口与下一轮最小扩证方向 |
| **失败后回流到哪一层** | 留在本层。如缺原始材料，向用户发最小阻塞说明（只能要原始材料 / 缺失样稿全文 / 冲突信息澄清，不能要中间整理件） |
| **自由度压缩规则** | (1) `10` 个字段逐个填，不允许合并或跳过；(2) 不允许用"综合考虑"替代具体字段值；(3) 有多解时必须先列出所有解，等用户选定后才能填入；(4) 反向确认必须用用户听得懂的话写清"理解成了什么、准备做什么、会交付什么" |
| **是否可自动化** | 半自动 |
| **自动化信号或判定来源** | 字段完整度检查可自动化（检查 `10` 个字段是否非空）；用户确认环节必须人工 |
| **正式来源** | (1) `仓库卫生最高禁令.md` 1.1 节反向确认条款；(2) 本文件第三节"回执层最少必须锁住什么"；(3) 经验裁决总表 E-20（反向确认是硬闸门）；(4) 经验裁决总表 E-22（样稿未直读不宣称对齐） |
| **当前状态** | **工作版** |
| **边界备注** | 本标准只管回执卡字段完整性和用户确认。不管字段内容的业务正确性（业务正确性由后续主题探索、门禁和返修兜底）。候选放行态只说明“当前允许继续探索”，不等于正式主题已经成立。 |
