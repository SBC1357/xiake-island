## 一、当前正式主链

当前白盒正式主链不是旧的 `真实写作场景 -> 一级稿型 -> 主二级稿型 -> 主样稿母本` 三级整合口径，而是下面这条冻结链：

`materials -> layer1 -> writing_contract -> viewpoint -> outline -> draft -> post_review -> three_flow_loop`

这条链的职责拆分如下：

1. `materials` 只负责把原始材料完整导出为当前 run 可见文本，不负责判断稿型。
2. `layer1` 只负责第一层冻结与分流，不直接进入正文写作。
3. `writing_contract` 只负责冻结任务边界、受众、必写点、禁区、交付物和验收口径。
4. `viewpoint` 只负责冻结文章立场、主角度和证据组织方向。
5. `outline` 只负责把 `viewpoint` 编译成可执行大纲。
6. `draft` 只负责在大纲、合同和资产约束下生成正文。
7. `post_review` 只负责对成稿做正式裁决。
8. `three_flow_loop` 只负责承接后审回流，不是新写作入口。

## 二、`layer1` 状态机与边界

`layer1` 是当前正式规则层的第一道冻结闸，不是样稿分类器，也不是正文生成器。

### 2.1 `layer1` 的真实状态

`layer1` 只承认三类入口判断：

1. `direct`
2. `sp`
3. `sample_boundary`

其中：

1. `direct` 表示用户意图、材料定位和验收标准已经足够清楚，可以继续向 `writing_contract` 冻结。
2. `sp` 表示当前信息不足或关键约束缺失，必须先补答，不能直接放行。
3. `sample_boundary` 表示样稿边界已识别，但必须检查能借什么、不能借什么，不能把参考样稿当事实源。

### 2.2 `layer1` 的边界纪律

1. `direct` 只有在任务目标、材料定位、受众和验收标准都能冻结时才可成立。
2. `sp` 一旦命中，当前 run 必须停在冻结前，不得越过 `writing_contract`。
3. `sample_boundary` 只允许提供结构、语气和节奏参照，不允许借事实、数字、结论或标题。
4. 如果参考样稿与 benchmark 参考答案是同一文件，必须阻断，不能静默吞过。
5. `layer1` 的放行标准不是“看起来能写”，而是“可冻结、可追责、可验收”。

### 2.3 `layer1` 的输出

`layer1` 需要至少冻结以下结果：

1. 路由判断
2. 任务意图包
3. 证据包
4. 样稿边界
5. 冻结状态

这些输出必须足以支撑后续 `writing_contract`，但不能直接替代它。

## 三、`writing_contract` 到 `draft` 的正式规则

### 3.1 `writing_contract`

`writing_contract` 是正式规则层的合同冻结点，职责是把“能写”变成“按什么条件写”。

它必须冻结：

1. 受众
2. 平台
3. 篇幅
4. 交付物类型
5. 主角度 / 主问题
6. 主证据链
7. 必写点
8. 禁区
9. 验收口径

### 3.2 `viewpoint`

`viewpoint` 是合同后的观点冻结点，不是自由发挥层。

它只做三件事：

1. 固定正文主张
2. 固定证据排序
3. 固定文章立场和解释方向

### 3.3 `outline`

`outline` 只消费 `viewpoint`，把观点编译成结构。

它必须显式保留：

1. 主线论证顺序
2. 关键证据落点
3. 必写段
4. 风险段
5. 回流点

### 3.4 `draft`

`draft` 只负责在已冻结的大纲和合同约束下生成正文。

它不能：

1. 反向改写合同
2. 偷换观点
3. 自行扩展 must-cover
4. 把样稿边界当事实源

### 3.5 `pre_review`

`pre_review` 是并行侧审，不是首稿前硬闸门。

它的真实边界是：

1. 只能在 `writing_contract` 后启动
2. 只能并行观察，不抢主链放行权
3. 主要负责提前发现结构性风险、证据缺口和表达偏航
4. 可以发预警，但不能替代 `post_review` 的正式裁决

## 四、`post_review` 三流回流

`post_review` 是正式裁决口，必须把结果收成可执行分流，不能只给评价。

### 4.1 `pass`

`pass` 表示当前稿件已经满足本轮验收口径，可以结束当前 run。

### 4.2 `requirement_rework`

`requirement_rework` 表示问题根在任务定义、意图冻结或证据方向，必须回到 `layer1` 重新冻结。

### 4.3 `whole_revise`

`whole_revise` 表示当前主干方向基本成立，但整篇需要重做，不能只做局部修补。

### 4.4 `local_patch`

`local_patch` 表示正文主体可保留，但局部段落、事实点或措辞需要修补。

`local_patch` 进入小闭环后，修补结果必须再回到 `post_review` 复验，不能直接视为完成。

### 4.5 `post_review` 的硬纪律

1. `post_review` 不能把结构性问题伪装成局部问题。
2. `whole_revise` 不能被压扁成 `local_patch`。
3. `requirement_rework` 不能留在后审层自转，必须回到第一层冻结。
4. `pass` 只有在主链、合同和边界都成立时才可给出。

## 五、`supplementary_evidence` 的真实边界

`supplementary_evidence` 是独立确定性侧链，不进默认主链。

它的职责是：

1. 在主证据包之外补足可验证材料
2. 只在确有缺口或明确需要时介入
3. 保持与主证据包的边界隔离

它不负责：

1. 替代主证据包
2. 默认接管写作主链
3. 把未来可能的实现方式误写成当前主链事实

当前正式规则必须明确：`supplementary_evidence` 可以存在，但它不是默认主链的一部分。

## 六、最低门槛、失败分支与冻结边界

### 6.1 最低门槛

1. `layer1` 必须能明确给出 `direct / sp / sample_boundary` 的冻结结果。
2. `writing_contract` 必须能把受众、交付物、必写点和禁区冻结下来。
3. `pre_review` 可以并行启动，但不能替代主链放行。
4. `post_review` 必须能产出 `pass / requirement_rework / whole_revise / local_patch` 之一。
5. `supplementary_evidence` 必须保持独立侧链边界，不得写成默认执行事实。

### 6.2 失败分支

1. 用户需求不清，走 `sp`。
2. 样稿边界冲突，阻断放行。
3. 证据方向错配，走 `requirement_rework`。
4. 整篇主干错误，走 `whole_revise`。
5. 只是局部问题，走 `local_patch`。

### 6.3 冻结边界

1. 没有冻结 `layer1`，不得进入 `writing_contract`。
2. 没有冻结合同，不得进入 `viewpoint`。
3. 没有固定观点，不得进入 `outline`。
4. 没有大纲，不得进入 `draft`。
5. 没有正式裁决，不得把后审结果写成通过。

## 七、旧三级稿型口径的处理

旧的 `真实写作场景 -> 一级稿型 -> 主二级稿型 -> 主样稿母本` 可以保留为历史分类记忆，但不再是当前正式规则链。

它现在最多只能承担两件事：

1. 作为历史路由经验的归档入口
2. 作为样稿风格和题型经验的粗粒度参考

它不能再承担：

1. 当前活状态冻结
2. 当前正式主链定义
3. 当前后审裁决标准
