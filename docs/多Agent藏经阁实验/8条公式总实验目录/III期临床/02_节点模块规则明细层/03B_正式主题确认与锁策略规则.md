边界：本文件只写正式主题确认与锁策略，不展开大纲、写作合同和评分实现细节

## 一、本层解决什么问题

本层只解决下面四件事：

1. 正式主题何时成立
2. 推荐主线何时升级成唯一正式主题
3. 当前任务的 `theme_lock_policy` 取哪一档
4. 当前是否已达到成文闭环，可以进入写作合同

## 二、本层输入

进入本层前，必须已有：

1. `候选主题包`
2. 推荐主线 / 备选主线
3. 关键支撑点
4. 大致结论 / 落点

没这四件，不得进入本层。

## 三、正式主题成立的最小门槛

正式主题成立，至少要同时满足：

1. 推荐主线已明确
2. 主线下的关键支撑点已明确
3. 大致结论 / 落点已明确
4. 当前不可漂移点已明确

没到这一步，只能继续停留在候选态。

## 四、主题锁策略

当前任务的 `theme_lock_policy` 先只允许三档：

1. `strict`
   - 后续扩证不得改题
2. `bounded`
   - 后续若出现更优主线，可带理由回到确认层重新确认
3. `open`
   - 允许多轮扩证-改题-再确认

这里收紧一条：

1. 锁策略不是系统常量
2. 锁策略必须作为当前任务参数显式写出来
3. 锁策略未定时，不得假装“默认可改”或“默认不可改”

## 五、成文闭环判断

正式主题成立，不等于立刻可写。  
本层还必须判断一次当前是否达到成文闭环。

当前工作版先按下面原则执行：

1. 成文闭环看的是 `稿型 × 篇幅级别`
2. 若当前证据足以支撑目标稿型和目标篇幅，可放行到写作合同层
3. 若主题已成立但证据还不够支撑目标篇幅，回主题探索层继续扩证
4. 若目标篇幅已明显超出当前稿型的自然跨度，应回任务定界层重判稿型，不得默认继续扩证

## 六、本层的合法输出

本层只允许三类输出：

1. `正式主题卡`
2. `theme_lock_policy_card`
3. `composition_closure_check`

并且只允许两种放行结果：

1. 主题已成立且成文闭环已满足 → 放行到大纲过闸与写作合同层
2. 主题已成立但成文闭环未满足 → 回主题探索层继续扩证，或回任务定界层重判稿型

## 七、最容易犯的错

1. 推荐主线刚成链，就跳过正式主题确认。
2. 不写 `theme_lock_policy`，让系统后面自由漂移。
3. 正式主题刚成立，就误判成已经够写。
4. 目标篇幅已经超出稿型自然跨度，却还在原稿型里硬扩。

## 八、标准件

### 标准件标准

1. 标准件必须能挂到一个节点。
2. 标准件必须说清楚：解决什么问题、未来从哪来、当前是什么状态、现在能不能用、边界在哪里。
3. 标准来源优先级按 `00_总则与需求对齐回执规则` 的统一口径走。
4. 只有来源可回指、节点落点明确，才允许从预留位升级成正式标准。

| **标准件名称** | 正式主题确认标准 |
| **所属节点** | 正式主题确认与锁策略层 |
