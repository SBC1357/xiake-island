# III期首稿生成规则

状态：整合稿 / 待继续补证  
日期：2026-04-08  
用途：收束 III 期首稿生成的最小规则，回答“首稿怎么在约束下写，不是怎么凭空空写”  
边界：本文件不展开返修 loop

## 证据摘要

1. 已直接核对改稿复盘与近期逐字稿，确认写作质量最稳的场景是“上下文定死后的局部补写”，不是“只有逻辑和提纲的空写”。
2. 已直接核对现有模块补丁文档，确认“好稿共用约束”已经不只服务改稿，也应进入首稿生成。
3. 已直接核对白盒首稿脚本，确认当前工程已具备两条硬边界：`reference_answer.txt` 在首稿前不可见；`reference_sample` 若被放开，必须与 benchmark 参考答案物理隔离且只作结构/语气锚点，不得充当事实源。

## 一、首稿生成的基本认识

首稿生成不是“给定任务后直接从空白开始写”。  
它默认建立在：

1. 已过闸的大纲
2. 已冻结的输入
3. 已编译的写作合同
4. 已绑定的样稿母本

之上。

## 二、上下文粒度优先级

当前工作版先按下面这个优先级使用：

1. `给上文 + 给下文 + 补中间一句`，质量最高
2. `给上一句 + 接下一句`，质量次一级
3. `只有提纲和逻辑，让它从头写开头`，质量最低

## 三、开头和结尾的特殊性

1. 开头和结尾天然缺一侧上下文。
2. 它们默认比中段更难写。
3. 所以开头和结尾不应按“和中段一样容易稳定”的预期来设计。

## 四、生成全稿 Agent 与局部修改 Agent 的分离

当前最小规则：

1. 生成全稿 Agent 负责把过闸后的大纲展开成首稿。
2. 局部修改 Agent 负责在定死上下文下修 `一句话 / 一个信息点 / 一个空缺位`。
3. 这两个 Agent 不应共用一套内置 prompt。

## 五、答案泄露与样稿边界

1. 首稿层不得看见 `reference_answer`、`gold` 对照结果或任何答案形摘要。
2. benchmark 参考答案只允许在审稿/评分层物化，不得在首稿前提前落入出稿目录并被下游读取。
3. 参考样稿默认不可见；只有当前任务显式放开、样稿合同已冻结、且样稿与 benchmark 参考答案物理隔离时，才允许进入首稿提示词。
4. 参考样稿进入首稿提示词后，也只能承担：
   - 结构锚点
   - 语气锚点
   - 返修锚点
5. 参考样稿不得承担：
   - 事实源
   - 数字源
   - 判断源
   - `gold` 替身

## 六、首稿最容易犯的错

1. 只给提纲，就要求它直接写出高质量开头。
2. 还没定死上下文，就要求局部句子级高质量。
3. 开头、结尾、中段不分难度，统一按一个生成策略对待。
4. benchmark 参考答案在首稿前可见。
5. 参考样稿与 benchmark 参考答案混成同一路径，形成伪样稿泄露。

## 七、与下游的关系

1. 首稿写坏了，先看是大纲没立住，还是上下文粒度给错了。
2. 门禁与返修不负责弥补上游完全空写造成的全部损失。

## 八、标准件

这一节包含已升级为工作版的正式标准件，以及仍在预留位的标准件。

### 标准件标准

1. 标准件必须能挂到一个节点。
2. 标准件必须说清楚：解决什么问题、未来从哪来、当前是什么状态、现在能不能用、边界在哪里。
3. 标准来源优先级按 `00_总则与需求对齐回执规则` 的统一口径走。
4. 只有来源可回指、节点落点明确，才允许从预留位升级成正式标准。

### 标准件 S-05-01：首稿防泄露与样稿边界标准

| 字段 | 内容 |
| --- | --- |
| **标准件名称** | 首稿防泄露与样稿边界标准 |
| **所属节点** | 首稿生成层 |
| **要解决什么问题** | 防止 benchmark 参考答案、`gold` 对照结果或答案形摘要提前进入首稿链，同时防止参考样稿被偷换成事实源或 `gold` 替身 |

| **固定终点** | 首稿提示词已编译完成，且满足：`reference_answer` 在首稿前不可见；参考样稿若可见，则用途和边界已冻结 |
| **输入** | 过闸大纲 + 写作合同 + 取材包 + （可选）参考样稿合同 + benchmark 评价合同 |
| **输出** | 首稿提示词编译结果 + 出稿清单。清单至少写明：① 首稿前 `reference_answer` 是否可见 ② 参考样稿是否进入提示词 ③ 样稿路径与角色边界 |
| **通过标准** | 同时满足以下全部：(1) 首稿前 `reference_answer` / `gold` / 答案形摘要不可见；(2) 若参考样稿进入首稿提示词，则样稿合同已冻结，且样稿路径与 benchmark 参考答案路径不同；(3) 参考样稿用途被显式限定为结构/语气/返修锚点，不承担事实源、数字源、判断源或 `gold` 替身角色；(4) 首稿提示词与出稿清单都能回指上述边界 |

| **失败后回流到哪一层** | benchmark 边界不清或样稿合同未冻稳 → 回大纲过闸与写作合同层；只是提示词编译未落实边界 → 留在本层修正 |

| **是否可自动化** | 是 |
| **自动化信号或判定来源** | 可自动化：首稿前 `reference_answer` 可见性检查、参考样稿路径与 benchmark 参考答案路径冲突检查、样稿合同字段存在性检查、出稿清单字段检查 |
| **正式来源** | (1) 本文件第五节“答案泄露与样稿边界”；(2) 白盒首稿脚本对 `reference_answer_visible_before_draft` 和 `reference_sample_matches_benchmark_reference` 的直接守卫；(3) 当前白盒出稿清单中关于 `reference_answer_visible_before_draft` 与 `reference_sample_enabled_for_draft` 的直接留痕 |
