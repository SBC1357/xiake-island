## 一、架构定位

III 期完整架构回答的是：

`这个系统到底是什么，节点怎么接，哪些规则已经冻结，哪些仍待验证。`

它不回答：

1. 这一轮试验怎么安排
2. 每条经验的原始全文是什么
3. 所有 prompt 级微操
4. 模块库和映射表的完整细目
5. 小修改单和句位级返修手法

它只收：

1. 会改变节点职责的规则
2. 会改变主链路由的规则
3. 会改变上下游输入输出边界的规则

## 二、当前主链

当前主链先收成下面 `11` 段：

1. 任务进入池
2. 需求对齐回执层
3. 输入标准化与冻结层
4. 稿型识别与资产路由层
5. 主题探索与扩证层
6. 正式主题确认与锁策略层
7. 大纲过闸与写作合同层
8. 首稿生成层
9. 审稿与门禁层
10. 定向返修层
11. 经验回流层

## 三、最关键的系统性变化

### 1. 上游先收窄

III 期不再默认“用户给一堆材料，直接往下写”。  
而是先做：

1. 真实写作场景识别
2. 一级稿型识别
3. 主二级稿型识别
4. 主样稿母本绑定

### 2. 输入先净化

当前工作版的输入标准化目标是：

`不丢失 + 降噪 + 不多不少`

### 3. 大纲单独过闸

大纲不再当附属物，而是独立过闸点。  
没过闸的大纲，不该直接进首稿。

### 4. 首稿不再按“空写”理解

首稿生成默认建立在：

1. 冻结输入
2. 写作合同
3. 过闸大纲
4. 样稿母本

之上。

### 5. 门禁后的返修是主链

返修 loop 不是补丁，而是 III 期的正式链路。

### 5A. 返修层正式拆成双分支

当前稳定到足以写进架构的只有这两条：

1. `whole revise`：处理路线错、结构错、主承重错
2. `local revise`：处理 `1` 句话 / `1` 个信息点 / `1` 个空缺位

当前还没有冻结的是：

1. 自动 whole/local 切换阈值
2. 每轮 patch 上限
3. 轮数上限

### 6. 经验回流被拆成独立节点

经验不再从案例直接跳架构。  
而是必须先经过来源、抽象、落点和裁决。

### 7. 稿型先行，但主题不再提前锁死

III 期白盒当前收紧为：

1. 先由用户需求判稿型
2. 再围绕稿型进入主题探索
3. 候选主题不得直接下传写作链
4. 正式主题要在成链后再确认

### 8. 主题确认前后被拆成两层

当前工作版先把大纲层前再拆出两层：

1. `主题探索与扩证层`
2. `正式主题确认与锁策略层`

这两层解决的是：

1. 何时允许只有候选主题继续往下走
2. 何时必须先扩证
3. 何时才算正式主题成立
4. 后续扩证是否允许改题

### 9. Agent 负责理解判定，代码负责验证执行

当前白盒架构先把职责边界收成：

1. Agent 负责：
   - 稿型理解
   - 样稿理解
   - 候选主题形成
   - 扩证成链
   - 推荐主线 / 备选主线
2. 代码负责：
   - 状态机流转
   - 对象冻结
   - 工件落盘
   - 验证与留痕
   - 下游执行

### 10. 模块级 LLM profile 已进入工作版

当前已经稳定到足以写进架构的不是“哪一个模型最好”，而是：

1. 节点可以挂独立 `profile`
2. `profile` 可以独立绑定：
   - 模块角色上下文
   - `base_url`
   - `model`
   - `protocol`
   - `api_key`
3. 这些配置按模块隔离，不再靠一个全局 `.env` 相互覆盖

当前已落地的工作版 profile：

1. `draft`
2. `review_score`
3. `whole_revise`
4. `local_task_router`
5. `local_patch`
6. `local_patch_audit`

当前稳定边界：

1. 强 persona / 认知框架优先用于：
   - `draft`
   - `review_score`
   - `whole_revise`
2. 轻 execution-role 优先用于：
   - `local_task_router`
   - `local_patch`
   - `local_patch_audit`
3. 模块级 API 隔离通过 profile 级 env 名完成，不再需要多个模块共抢一组全局值。

## 四、当前稳定的节点间逻辑

### 1. 需求对齐回执层

作用：
- 锁定任务理解，禁止带着多解直接开工

### 2. 输入标准化与冻结层

作用：
- 把散乱输入压成下游正式输入包

### 3. 稿型识别与资产路由层

作用：
- 决定当前任务该走哪种写法路线、绑定哪种样稿母本、调哪些资产

### 4. 主题探索与扩证层

作用：
- 在已锁稿型下形成候选主题
- 组织逻辑链 / 故事链
- 给出推荐主线和备选主线

### 5. 正式主题确认与锁策略层

作用：
- 把候选主题升级成正式主题
- 锁本轮 `theme_lock_policy`
- 判断是否已达到成文闭环

### 6. 大纲过闸与写作合同层

作用：
- 先把故事线和主承重段立住，再允许首稿生成

### 7. 首稿生成层

作用：
- 把已过闸的大纲展开成首稿

### 8. 审稿与门禁层

作用：
- 不只看文风和事实，还要判断问题级别
- 正式产出可路由的问题单
- benchmark 模式下允许额外挂 `reference compare`，但 comparator 不得替代问题单
- 当前工作版同时输出：
  - 硬层 `qualified / weighted_total`
  - 软层 `editorial_total / editorial_dimensions`

### 9. 定向返修层

作用：
- 把问题单按级别送回 `local revise`、`whole revise`、主题确认层或需求层

### 10. 经验回流层

作用：
- 把案例、纠偏和问题单重新压回经验裁决层
- 只把已经稳定到足以改主链的东西再升层

## 五、当前最重要的系统纪律

1. 有多解先反向确认，没确认不准开工。
2. 输入先做标准化，不把预处理甩给用户。
3. 候选主题不得直接冒充正式主题下传写作链。
4. 大纲必须独立过闸。
5. 返修必须先分级，再决定回哪一层。
6. 经验回流必须先裁决，再决定是否升层。
7. 03 层不吸收稿型专用技巧、模块库细目和句位微操。

## 六、当前还没有完全冻结的部分

1. 情绪 / 逻辑 / 语气的量化评分锚点
2. 八公式递归调用的精确边界
3. 并发微改的真实上限
4. 开头 / 结尾专门生成策略
5. 各模块的最优具体模型组合、few-shot 素材和跨模型路由策略

这些内容暂留在经验池，不直接写死成完整架构。
