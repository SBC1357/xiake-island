# 历史单题全貌模板

状态：历史模板 / 非当前活态真相源  
日期：2026-04-11  
用途：历史归档参考模板。保留人工审核的填写骨架，便于回看这题当时如何被整理、模型当时看到了什么、当时如何被约束、当时稿件写成了什么、为什么当时过或不过、当时下一刀该改哪里。工程核验看文末附录。

## 历史模板口径

1. 这份单题全貌模板以“同轮 runtime 直证 + 人工审稿可读性”为历史参考，不再接受只贴路径的空壳版。
2. `主阅读层` 作为归档骨架，仍保留回答 5 个问题的字段：写给谁、喂了什么、怎么下指令、写成了什么、当时怎么改。
3. `本轮真正喂给写手的材料（人工整理版）` 在归档回看时应写成人能读的整理稿；实际 runtime 若混入 OCR / 页级碎片 / Accepted manuscript 水印，也要明确标为噪声，不要只留 `materials_full.json` 路径。
4. `给模型的 prompt 全文` 作为历史记录，保留 `system prompt` 与 `user prompt` 的贴出位置；只写路径仍不算完整。
5. `审核意见与返修抓手` 作为历史记录，应写出总分、达标结论、各维度关键失分与当时可执行改法；只写“分低/要优化”不算完整。
6. 历史上未产出的独立载体，需写“当时不存在 + 原因”，不要伪造 `planning prompt`、`quality prompt` 等不存在文件。
7. 评分已经落盘时，文末工程附录仍应保留 `Runtime 指针 / 评分结果 / 证据摘要`，便于回查。
8. 归档回看时可运行：`python scripts/validate_case_fullview.py <本文件或目录>`

## 汇报前最小核对

1. 单题全貌历史模板仍以“人工审核可读性”作为填写硬门；人工无需点路径也能开始审稿。
2. `主阅读层` 至少具备：人工审核先看、材料整理版、prompt 全文、稿件全文、审核意见与返修抓手。
3. `materials` 若存在 OCR / 页级碎片 / Accepted manuscript 水印，必须在正文里明说，不得假装是清洁资料。
4. 只要 runtime 已存在 `score.json / summary.json`，就应同步写出 `评分结果` 与 `审核意见与返修抓手`。
5. 没有生成稿全文，不得把这份档案算作“已完成的单题全貌”；对历史模板来说，这一条仅作为当时的填写检查项。
6. 汇报前运行 `python scripts/validate_case_fullview.py <本文件或目录>`；校验不过，不得汇报“单题全貌已齐”。
