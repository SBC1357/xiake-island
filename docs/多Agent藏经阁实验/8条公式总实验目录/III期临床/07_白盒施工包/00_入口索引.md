## ⚠ 目录结构一致性检查（每次打开执行任务前必做）

任何 Agent 进入本施工包执行任务之前，必须先核对一遍当前磁盘目录结构与下方"当前目录结构"一节是否一致。

检查内容：

1. 子目录和文件是否多了或少了（含新增轮次目录）
2. 各轮入口文件是否仍然存在且可打开
3. 是否出现了本索引未登记的散落文件或临时产物

不一致时的处理：

- 磁盘多了未登记的文件/目录：先判断是脏产物还是正式新增。脏产物立刻清走；正式新增必须先补进本索引再继续。
- 磁盘少了已登记的文件/目录：停止执行，报告缺失。
- **检查完成之前，不得开始执行任何施工、审计、回写任务。**

## 一、目录职责

本目录只做三件事：

1. 承接白盒编排器多轮施工方案
2. 承接每一轮白盒实验的施工入口与审计入口
3. 把每轮暴露出来的稳定问题回指到 `02/03` 两层

本目录不做：

1. 代替 `02_节点模块规则明细层` 冻结长期规则
2. 代替 `03_完整架构层` 冻结系统总图
3. 承接仓外 runtime 原始运行产物

## 二、真相源纪律

1. `07_白盒施工包/00_入口索引.md` 只负责：
   - 当前白盒施工包总入口
   - 当前活动轮次指针
   - 各轮施工包目录索引
2. 每一轮具体执行口径、范围、顺序、验收，只认该轮子目录下的总方案。
3. 任何已经稳定到足以升层的节点规则或系统边界，不在施工包里另写一版长期口径，而是回写 `02/03`。

## 三、当前冻结主链（本文件为文档侧唯一真相源）

代码侧真相源：`侠客岛白盒编排器/config/node_io_protocol.json` + `run_whitebox_main_chain.ps1`

当前主链：

`物料提取 -> 第一层 -> 写作合同 -> 观点层 -> 大纲 -> 成稿 -> 后审 -> 四流`

后审四流（且只允许这四流）：

1. `pass` — 通过，主链停止
2. `requirement_rework` — 根错在前面，回第一层重跑
3. `whole_revise` — 整篇写歪，回成稿节点整改模式（不是新顶层）
4. `local_patch` — 局部修改，进 `local_patch -> local_patch_audit -> 后审` loop

当前调度口径：

- 写作合同与观点层**当前串行**执行（writing_contract → viewpoint → outline）
- 设计目标仍保留"写作合同层与观点层可并行"的关系，待真实样例验证稳定后决定是否恢复并行调度
- 当前调度实现以 `run_whitebox_main_chain.ps1` 为准

**其他文件如需引用主链，只引用本节或代码侧真相源，不得自行另写一版。**
