## 这一关到底交付什么

1. `draft_prompt.md`
   - 真正喂给模型的完整 prompt。

   - 这是 runtime 原始落盘的正文文件。

3. `score.json`
   - 审稿打分关回写给这篇成稿的评分结果。

## 这几条提示词到底在干什么

### 基线 / 默认 draft profile

- 想做的事：
  - 忠实执行合同和证据边界
  - 在不乱编事实的前提下，把稿子写顺、写稳

### route1

- 想做的事：
  - 先把稿子写稳
  - 先守合同和题目主线
  - 不抢文气，不冒险

### route2

- 想做的事：
  - 优先守证据边界
  - 材料不够就保守写
  - 不脑补、不推演、不擅自补强判断

### route3

- 想做的事：
  - 在不越界前提下，把稿子写得更像成熟编辑交稿
  - 强化交付感、节奏和收束

## 我对这套提示词的判断

1. 旧文档里只摘一句“关键提示词”给你看，是错的。
   - 因为真正影响结果的不是那一句，而是后面整套合同、公式约束、证据锚点卡和材料摘录。

2. 这三条 route 的差异，不在“多了一句空话”，而在取向不同：
   - `route1` 偏稳
   - `route2` 偏守边界
   - `route3` 偏交付感

3. 从结果看，这三条提示词这轮都没打赢基线。
   - 所以当前不能说“提示词改得已经有效”。
