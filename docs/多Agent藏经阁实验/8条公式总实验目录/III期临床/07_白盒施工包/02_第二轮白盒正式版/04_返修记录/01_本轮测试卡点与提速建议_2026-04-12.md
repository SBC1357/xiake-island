## 基于直接证据的工程判断

下面这部分是工程判断，不是假装成日志原文。

### 1. 当前测试粒度太大，不适合交互式一把跑完

- 单样本已经是 18 个 route case。
- 若直接扩到 3 样本，就是一个很重的长链任务。
- 这种任务不适合“坐等一个总结果再看”，因为一旦中途某处失败，前面的有效结果就容易散在 runtime 目录里，用户体感就是“等了很久没结论”。

### 2. 总控缺少“增量收尾”和“续跑能力”

- 当前最痛的不是前面没跑，而是跑了一大半以后没有自动收成总表。
- 如果总控具备：
  - 每完成一个 `module + route + candidate` 就增量写 summary
  - 支持从现有 `runRoot` 续跑
  - 自动跳过已经完成的 case
- 那这轮即使中断，也不会让结果看起来像“全没了”。

### 3. 现在不该默认整套 full run

- 当前最合适的交互式测试单位，应该是：
  - 单样本
  - 单模块
  - 三路线对照
- 而不是一上来就是：
  - 三样本
  - 六模块
  - 全路线

## 下次怎么避免再跑两小时

### 先改默认测试策略

- 默认先跑：`单样本 + 单模块 + 3 路线`
- 推荐顺序：
  - `draft`
  - `review_score`
  - `whole_revise`
  - `local_task_router`
  - `local_patch`
  - `local_patch_audit`
- 只有单模块跑出稳定信号后，再扩到下一个模块。

### 总控必须补 3 个能力

1. `resume`
   - 给定现有 `runRoot` 继续跑，不重新开整套。
2. `skip completed`
   - 发现某个 `module/route/candidate` 已经有完成产物，就直接跳过。
3. `incremental summary`
   - 每完成一个 case 就刷新一次总表，而不是只在最末尾统一写。

### route 级别的立即动作

- `review_score route3` 先不要进 full run。
- 原因不是“它效果差”，而是这轮直接证据已经表明它有稳定性问题：`无法解析评分 JSON`。
- 下一轮应该先单独修它，再重新入列。

### baseline 不要每次重跑到底

- 当前运行里，baseline 成本本身也不低。
- 如果 baseline 已冻结并且输入没变，下次应该优先复用 freeze，而不是默认从头再跑 baseline phase。
