## 这几个 JSON 里到底有什么

如果你点开 `score.json`，当前能直接看到的关键交付是这些：

1. `weighted_total`
   - 这一稿的硬层总分。

2. `qualified`
   - 这一稿是否过线。

3. `dimensions`
   - 五个通用维度的细分打分。

4. `editorial_dimensions`
   - 可发表性、推进力、读者贴合度、自然度这些软层判断。

5. `formula_trace`
   - 公式位逐条命中、漏掉还是部分命中。

6. `blocking_issues`
   - 真正阻塞过线的问题。

7. `missing_or_misaligned`
   - 缺了什么、歪了什么。

8. `backtrace`
   - 问题回指到哪一层合同或材料。

9. `next_actions`
   - 下游接着改什么。

## 这一关和后面模块的责任边界

### 审稿打分关负责

- 判断稿子是否过线
- 判断稿子哪里有问题
- 判断问题属于哪一层
- 输出结构化问题和下一步动作

### 审稿打分关不负责

- 不负责把问题分流成 patch 路径
- 不负责决定走局部改句还是整稿返修
- 不负责真正执行修稿

路径分流归：
- `local_task_router`

真正执行修稿归：
- `whole_revise / local_patch`
