## 任务背景
当前白盒系统的设计链路不是“单点写作调参”，而是上游输入治理与下游模块评估之间的固定架构链：先完成输入标准化与冻结，再按稿型路由到下游主线，最后进入模块化 PK 与返修闭环。  
`01_输入标准化与冻结规则.md` 明确要求输入层只做 `客户 brief 冻结`、`原始事实锚点`、`参考样稿冻结`、`禁区与必须点`、`主题探索前置卡` 五件事，不提前产出正式写作合同；`02_稿型识别与路由规则.md` 又要求路由按“真实写作场景 -> 一级稿型 -> 主二级稿型 -> 主样稿母本”走，并把回执层字段（场景、稿型、交付物、证据链等）作为下游执行前的硬入口。  
`whitebox_llm_profiles.json` 则把系统定义为 6 个节点职责已分工的编排链：`draft`、`review_score`、`whole_revise`、`local_task_router`、`local_patch`、`local_patch_audit`，每一段都绑定自己的提示与模型环境口径。  

现实运行路径现在出现落差：`module_pk_summary.json` 当前标记为 `partial`，且写着 `summary rebuilt from existing runtime artifacts`；说明当前回执不是一次完整主控收口输出。也就是说，已有可读运行数据，但当前主控链路在可验证口径上仍不闭环，和“先冻结-再路由-再模块执行”的单一架构态不一致。  

## 任务目的
这轮第一目标不是修某一个 prompt，而是把白盒总控恢复到“架构先行、结果可核验、模块可复用”的状态：  
1. 将“输入层冻结口径、路由口径、模块执行口径、返回结果口径”恢复为一条可追溯链。  
2. 让每个环节的输入、输出和边界可读可核验，避免下游补丁被当成上游真相。  
3. 建立下一步能稳定复用的状态模型：模块 PK 可以继续跑，但必须先有统一入口与统一验收口径。  

## 任务必要性
先改 route/prompt 或局部参数不会根治架构偏位，因为它只改变单点行为，不会修复 “谁定义输入、谁做冻结、谁触发模块、谁产出主控结论” 这条总控链。  
如果不先做这次架构修复，后续每次试验都会重走“有局部数据却没有全链路可决策证据”的循环，用户仍然要翻 runner/runtime 才能判断可否替换配置。  
这也是目前最关键的堵点：当前的任务目标是先把系统架构透明化并钉住真相口径，再谈各模块参数和稿型修正。  

## 第一层修复方案：加入LLM任务理解与反问门
第一层不再是“字段登记卡”，而是任务理解与冻结层。第一层要先确认“这次任务究竟要什么”，把目标、范围、约束、验收边界明确后再冻结，不允许先写作或先分配模块。  

第一层必须加入 LLM 任务理解器。它直接读取用户输入和上下文，输出统一理解状态：任务类型、目标对象、禁区与必须点、可用材料是否足够、是否可直接进入路由。  

第一层不是“等用户讲完才决定是否上 SP”。它要预挂一个主提示词，用户一开口就已生效。主提示词只做两件事：先读用户输入，再做分流判断。分流结果只到两个分流提示词——`直通提示词` 和 `SP 提示词`。`SP` 不属于后补插件，而是主提示词内置的分流出口。  

第一层必须有必要反问逻辑，且反问为强制门槛。触发条件是以下任一：  
1. 任务目标语义不清（例如“改改”“优化”未说明对象、规模或交付标准）；  
2. 关键约束缺失（受众、风格、严谨度、时效性要求缺失）；  
3. 材料不足（仅有标题或摘要，缺原始锚点、用途边界）；  
4. 无法定义验收标准（无可验证完成口径）。  

反问输出形式固定为最多 3 个短问题，默认只问 1-3 条，问题风格保持可直接回答：  
- “你要的是改写、重写还是扩写？优先级哪个？”  
- “哪些内容只能用、哪些必须保留原样？”  
- “你会用什么标准判断本次完成（例如：字数、风格、证据覆盖）？”  

第一层还要做一次分流：  
1. 如果用户目标已经清楚，直接进入“理解冻结”路径；  
2. 如果用户自己也不清楚到底要写什么，直接进入 `SP 模式` 路径，不得假装已经理解完任务。  

`SP 模式` 在第一层里的职责不是写稿，而是先帮用户把任务想清楚。它必须做到：  
1. 一次只追一个关键问题，不并发发问；  
2. 先把当前理解说成人话，再指出哪里还不清；  
3. 必要时给出 2-3 条真实路线，让用户选，不得替用户偷偷收敛；  
4. 用户没有选定路线前，不得进入写作合同。  

因此，第一层的 LLM 提示词结构应固定为 `1 个主提示词 + 2 个分流提示词`：  
1. 主提示词：预挂在入口，先读用户输入，再判断走 `直通` 还是 `SP`；  
2. 直通提示词：任务已清楚时使用，负责冻结意图与证据；  
3. SP 提示词：任务未清楚时使用，负责澄清、反问、路线比较和用户确认。  

第一层内部并发只保留 3 个模块：  
1. 证据提炼；  
2. 样稿边界判断；  
3. 用户意图提炼。  

第一层最终只落 3 个机器产物：  
1. `意图包 JSON`：装用户原话、真正要写什么、给谁看、主问题、隐性要求、禁区、放行状态；  
2. `证据包 JSON`：给合同层消费，装硬事实、出处锚点、证据身份、使用权限、冲突点和缺口；  
3. `证据包 md`：给成稿层消费，由大模型对给定材料做逐页、逐图、逐段预处理后生成，适用对象包括图片、文献、幻灯、docx 等各种格式材料。它的目标不是写摘要，也不是只提炼几点信息，而是尽可能用文字把内容完整性展现出来：图表类材料要明确告诉下游“这张图/这个表是什么、在表达什么”；正文类材料要尽量把文字信息完整转成可读文本；如果材料本身噪音很大，例如 200 页指南或 50 页文献，则只提取和当前任务目标直接相关的部分，其余边缘信息要写清为什么不进入下游。最终产物是一份下游可快速食用的 md 文档，供没有视力的 LLM 顺读。  

下游消费关系固定为：  
1. 合同层吃 `意图包 JSON + 证据包 JSON`；  
2. 大纲层吃 `意图包 JSON + 写作合同 + 证据包 JSON`，先把 `8公式` 的命中模块、段落承重、解释桥和证据锚点处理成可执行骨架；  
3. 大纲层输出两份：`大纲 JSON` 给机器流转，`大纲 md` 只作人工复核镜像；  
4. 成稿层主吃 `大纲 JSON + 证据包 md`，不直接裸吃 `意图包 JSON`；  
5. 审核层主吃 `意图包 JSON + 写作合同 + 当前稿 + 原始数据`，证据包只作索引和辅助，不得替代原始核验。  

未通过第一层（含反问未清、意图包未冻结、证据包未冻结）不得进入写作合同。  

审核链路固定分成两拍：  
1. `前审`：与成稿并行执行，不默认阻塞成稿。它只吃 `意图包 JSON + 写作合同 + 证据包`，不吃原始数据，只判断“按当前证据包是否够开写”。命中硬阻断时，直接打回前端返修。  
2. `后审`：成稿完成后执行。它吃 `意图包 JSON + 写作合同 + 当前稿 + 原始数据`，回原始材料核事实、边界和路线。  

后审之后固定只分四流：  
1. `pass`：通过，主链停止；  
2. `需求返修`：根错在前面，直接回 `第一层 -> 写作合同 -> 成稿 -> 后审`；  
3. `整改`：前面没错，但整篇写歪，直接回主链里的成稿节点重跑，不单开新层；  
4. `局改`：只改局部，走 `局改 -> 局部验收 -> 后审` 的 loop。  

整改不是回第一层。整改回成稿节点时，主指令改为 `整改卡`，它取代 `意图包` 成为本轮唯一主指令。整改模式输入固定为：`当前稿 + 整改卡 + 大纲 JSON + 证据包 md`。  

因此，成稿节点必须固定配置两个预提示词：  
1. `首稿预提示词`：首稿模式使用，主指令是 `意图包`；  
2. `整改预提示词`：整改模式使用，主指令是 `整改卡`。  

## 本轮形成过程（给冷启动 Agent）
这轮不是先画理想图，而是从一个真实问题顺着往下追：第一层到底输出什么，谁真正消费，哪里开始信息衰减。一路追到后面，逐步钉死了几件事。  
第一，第一层不能只是脚本登记卡，因为那样会丢掉用户隐性要求，也没有 LLM 反问和放行门；所以第一层被改成“主提示词分流 + 直通/SP + 三并发模块 + 双证据视图”。  
第二，证据不能只做薄事实卡。真实下游里，合同层、成稿层、审核层吃的不是同一种证据视图，所以证据被拆成 `证据包 JSON` 和 `证据包 md`；其中 `md` 不是摘要，而是给无视力 LLM 顺读的预处理研究包。  
第三，审核不能只审成稿。前审先看“按当前证据包够不够开写”，后审再回原始材料核“稿子有没有对得起原始数据”。  
第四，整改不是另起一个顶层世界，而是回主链里的成稿节点重跑；只有根错在需求、意图、证据方向时，才回第一层。  

## 明确排除的方案与原因
### 1. 第一层只出一张薄任务卡
已排除。原因：这种卡只会登记“写什么、给谁看、写多长”，会把用户脑子里的隐性要求、禁区、判断力度、材料用途全部压扁；下游拿到的是缩水任务，不是更清楚的任务。  

### 2. 第一层不加 LLM，只靠脚本登记
已排除。原因：没有 LLM，就没有反问、没有澄清、没有放行门；系统会在任务还没想清时直接往下跑。  

### 3. 第一层只出一个总 JSON
已排除。原因：意图和证据是两类完全不同的信息。把它们强塞进一个总 JSON，会让“用户要什么”和“材料能支撑什么”混在一起，下游很难按职责消费。  

### 4. 证据包只做事实提炼卡
已排除。原因：下游成稿层需要的是一份可顺读的研究/材料预处理包，不只是几个数字。图、表、壁报、长文献都要先被转成能读的文字材料。  

### 5. 证据预处理模块负责判断证据是否有缺口
已排除。原因：证据预处理只该做“把已有材料处理干净”，不该兼做“够不够写”的判断。是否存在缺口，应由审核侧根据任务目标、合同要求和当前证据来判。  

### 6. 审核层只吃证据包，不回原始数据
已排除。原因：这样只能复核上游提炼结果，抓不住“证据包本身就提错了”这种问题。后审必须回原图、原表、原文核。  

### 7. `gold/标准答案` 进入生产主链
已排除。原因：真实生产环境里没有这个东西。它只能挂在实验侧链，做 benchmark 或对照评分时才启用。  

### 8. 补充证据回填进主 `证据包 JSON`
已排除。原因：这样会污染主证据真相源，分不清“原始主证据”和“后补证据”。补充证据必须单独保存为 `补充证据 JSON + 补充证据 md`。  

### 9. 整改单独开一个顶层新层
已排除。原因：整改本质上不是新节点，而是成稿节点的第二种工作模式。把它独立成大层，只会让主链更长、更乱。  

### 10. 前审串行卡住成稿
已排除。原因：这样会让默认任务启动过慢。前审应该并行侧审，只在命中硬阻断时把任务退回前端 loop。  

## 冷启动 Agent 执行口径
冷启动 Agent 接手这份活时，不要再从 route、赛马、展示层、六模块总览入手，而是直接按主链落地。  
先实现第一层：预挂主提示词，接上 `直通/SP` 两个分流提示词，再接 `用户意图提炼 / 证据预处理 / 样稿边界` 三并发模块，最后稳定产出 `意图包 JSON + 证据包 JSON + 证据包 md`。  
再实现写作合同层：只吃 `意图包 JSON + 证据包 JSON`，不再自己猜任务，也不再直接碰原始材料。  
再实现大纲层：吃 `意图包 JSON + 写作合同 + 证据包 JSON`，先把 `8公式` 的模块化与递归处理成 `大纲 JSON + 大纲 md`。  
再实现审核链：前审只看结构化包够不够开写，后审回原始数据核事实。后审之后只允许三种回流：回第一层、回成稿整改模式、回局改 loop。  
如果用户明确说要补充证据，可以不等前审，直接启动补充证据模块；补充来源默认先走藏经阁，不够再去互联网搜，除非用户明确指定先搜。  
这份文档的目的不是讨论理想架构，而是把当前已经裁定的修复方案固定下来；冷启动 Agent 必须按这里的裁定继续施工，不能再把已经排除的方案重新捡回来。  

## 当前冻结口径（本轮讨论收束版）
主链固定为：`第一层 -> 写作合同层 -> 大纲层 -> 成稿节点 -> 后审 -> 三流 loop`。  
第一层输出固定为：`意图包 JSON + 证据包 JSON + 证据包 md`。  
写作合同层固定只吃：`意图包 JSON + 证据包 JSON`。  
大纲层固定只吃：`意图包 JSON + 写作合同 + 证据包 JSON`；固定输出：`大纲 JSON + 大纲 md`。  
成稿节点固定主吃：`大纲 JSON + 证据包 md`；首稿模式主指令是 `意图包`，整改模式主指令是 `整改卡`。  
前审固定只吃结构化包，不吃原始数据；后审固定回原始数据核。  
后审后三流固定为：回第一层、回成稿整改模式、回局改 loop。
