# 模块化提示词库评估与调用建议

- 状态：评估态 / 待落地
- 对象：`医学写作系统_模块化提示词库_v1.md`
- 结论先行：这份库有价值，但它更像“可路由的提示词资产库”，不是“让统筹 LLM 随手自由拼装的一大段长 prompt”。

## 价值是什么

这份提示词库的最大价值，不在于句子多，而在于它把医学写作里最容易混在一起的东西拆开了。

- 它把“工作姿态”单独拿出来，先决定模型是认真写、谨慎写，还是先找问题再写。
- 它把“站在谁的视角看”单独拿出来，区分临床读者、患者读者、审稿人、反方委员这些不同席位。
- 它把“不能做什么”单独拿出来，把事实、推断、判断、边界、风险分开管。
- 它把“怎么核验”单独拿出来，不让正文一边写一边偷跑。
- 它把“最后怎么交付”单独拿出来，便于统一输出样式和阅读负担。

换句话说，它的价值是把提示词从“文案”变成“制度件”，让统筹 LLM 有东西可选、有规则可守。

## 当前可用性如何

当前可用，但只适合“受约束地用”，不适合“整库自由拼”。

- 这份库已经足够作为路由库和底座库来用。
- 它也已经能支持按环节挑模块：观点层、大纲、初稿、审稿各有不同需要。
- 但它还不是一套可以让统筹 LLM 完全自由组合、随意重排、随意叠加的成熟件。

目前更像是“好用的半成品底座”，而不是“可随手拼装的成品积木”。

## 为什么现在不适合让统筹 LLM 完全自由拼装

原因不是库不好，而是库里不同类型的东西混在一起，粒度并不一样。

- 有些模块是姿态，有些模块是约束，有些模块是核验，有些模块是输出。
- 有些模块本来就是强约束，不能和别的模块随便并排。
- 有些模块之间存在明显重叠，比如表达、转场、通俗化、去机味这些内容，本来就容易互相重复。
- 有些模块更适合上游结构阶段，不适合再灌进成稿阶段。
- 如果让统筹 LLM 完全自由拼，最后很容易变成“看上去很聪明，实际上重复注入、互相打架、审计困难”。

对白盒系统来说，这种风险比少一个模块更麻烦，因为它会把口径弄散。

## 推荐路线

推荐路线不是“继续堆模块”，而是“先做几种固定底座，再按写作目标做局部替换或补丁”。

统筹 LLM 的任务应该是先判断这篇稿子的目标和风险，再决定从哪个底座出发、再补哪几块，不是直接从全库里无边界地抓一大串模块。

## 建议的固定底座

下面这四组可以先作为起步底座，后面再按任务目标局部替换：

- 观点底座：`TRUST_POSTURE` + `CONSTITUTION` + `OPPOSITION_PANEL` + `CLAIM_TYPER`
- 大纲底座：`SECTION_GOAL` + `QUESTION_BACKBONE` + `ARGUMENT_SKELETON` + `REVERSE_OUTLINE`
- 初稿底座：`EVIDENCE_LEDGER` + `LIMITATION_BINDER` + `OUTPUT_CONTRACT` + `PLAIN_LANGUAGE`
- 审稿底座：`MEDICAL_SAFETY_AUDIT` + `CONTRADICTION_HUNT` + `SOURCE_SUPPORT_CHECK` + `OMISSION_AUDIT`

这四组不是最终定死的架子，但它们比“让统筹 LLM 自由拼一整库”更稳，也更适合先落地。

## 短建议

- 先把“底座”固定住。
- 再让统筹 LLM 只做小范围替换，不做整库重写。
- 先把观点层、大纲、初稿、审稿四个环节分清，再谈更细的自由组合。
- 先保证可审计，再谈更灵活。

## 短证据摘要

- `./医学写作系统_模块化提示词库_v1.md` 里已经把模块分成姿态、席位、场景、约束、推理、核验、输出几大类，并且给出了按环节的默认组合，说明它本身就是一套可路由的模块库。
- `../../../../侠客岛白盒编排器/run_whitebox_main_chain.ps1` 目前仍然保持固定主链，说明这份库更适合先做“受约束的调用层”，不适合直接把主链变成自由拼装器。
