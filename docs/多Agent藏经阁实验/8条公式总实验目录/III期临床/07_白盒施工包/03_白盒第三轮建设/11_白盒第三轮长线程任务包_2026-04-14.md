## 本轮必须落地的建设对象

### 1. 统筹 LLM 的预注入提示词策略

先把统筹 LLM 变成“选底座、选补丁、选阶段”的路由器，不让它直接把整库提示词自由拼成一大段正文。它要先看稿子的目标、受众、风险、证据成熟度，再决定这一轮该注入哪一类提示词。

### 2. 新解耦模块建设

把观点层单独拎出来，让它只负责原始主张推理链、策略方向和医学立场，不再和写作合同、大纲、成稿混成一团。后续还要把它抽象成可单独调用的模块接口，保留以后演化成医学策略模块的可能。

### 3. 统一各文件口径

本轮要把阶段名、模块名、状态名、真相源名称统一起来，避免同一件事在多个文件里写出不同说法。凡是会变化的内容，只认一份真相源，其他文件只做回指或摘要，不再各写一版。

### 4. 推动真实测试与验证

不能只写构想，必须用真实案例去验。至少要覆盖主链样例、观点层样例、以及能暴露策略边界或补证据需求的样例。没有直接验证，就只能写待验证，不能冒充通过。

### 5. 目录清污

要把 `03_白盒第三轮建设/` 里已经不符合当前架构口径、又没有继续保留价值的旧文档清走，不留历史垃圾继续污染判断。清污不是乱删，而是先盘点、再判定、再删除。

- 范围：默认只处理 `03_白盒第三轮建设/` 及其被本轮接管的旧入口文档。
- 对象：只删“已被当前架构口径替代、且不再被真相源引用”的历史文档。
- 流程：先出“删除候选清单”和“保留清单”，不确定的先挂起，不准直接删。
- 产物：删完要留下删除记录，写清文件名、删除理由、替代真相源。

### 6. 提示词基座组合避雷测试

这一轮不用急着选最优组合，但至少要先把明显互相打架、明显容易误导、明显会把层级搅乱的组合排掉。也就是说，先做“避雷清单”，再谈“最优清单”。

- 测试方式：只测“预先定义的固定底座 + 局部补丁”组合，不允许统筹 LLM 临场自由拼整库。
- 测试目标：找出会互相打架、重复注入、越界、失真的组合，输出风险清单，不是选冠军。
- 测试样本：至少覆盖主链样例、观点层样例、大纲样例、成稿样例、一个边界例或负例。
- 测试结果：产出组合矩阵、失败模式、禁用组合、可保留组合，不写“已最优”。

## 明确执行阶段

### 阶段一：读库与定底座

先读提示词库，判断哪些模块适合做固定底座，哪些模块只适合做局部补丁，哪些模块不该进成稿层。这里的目标是先把“怎么选”说清楚，不急着改正文。

### 阶段二：定模块接口

把观点层、写作合同层、大纲层、成稿层的输入输出边界写清楚，特别是观点层要输出什么、谁来吃、谁不能吃。接口先定住，再决定后面怎么接。

### 阶段三：统一口径与落点

把目录、索引、状态卡、模块命名、阶段命名统一到一套说法里，避免用户看完还要猜“到底谁是真相源”。这一步优先解决可审计性。

### 阶段四：真实测试与回写

挑真实任务跑验证，记录直接锚点，只写看得到的文件、看得到的结果、看得到的分工。若结果不稳，就继续回到阶段二或阶段三，不要硬往前冲。

### 阶段五：目录清污与基座避雷测试

先做文档清污，删掉已经不代表当前架构、且与真相源平行冲突的旧口径文档；再做提示词基座组合测试，先排除明显冲突的搭配，留下可继续测试的底座名单。
