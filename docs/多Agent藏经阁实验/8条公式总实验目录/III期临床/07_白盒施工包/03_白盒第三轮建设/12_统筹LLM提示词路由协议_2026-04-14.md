# 统筹 LLM 提示词路由协议

- 状态：待验证 / 纸面定义完成
- 裁定路线：**固定底座 + 局部补丁**
- 真相源：本文件是统筹 LLM 提示词路由方式的唯一定义；模块原始内容回 `./医学写作系统_模块化提示词库_v1.md`
- 上游冻结依据：`./10_模块化提示词库评估与调用建议_2026-04-14.md`

## 裁定结论

统筹 LLM 不做"整库自由拼装"，采用"固定底座 + 局部补丁"。

理由：

1. 提示词库里不同类型模块粒度不一致，姿态类、约束类、核验类、输出类不能随意叠加。
2. 自由拼装容易导致重复注入、互相打架、口径发散、审计困难。
3. 白盒主链本身已经是固定结构，提示词路由也应该是受约束的。
4. "限定自由组合"在当前库成熟度下风险高于收益——先稳后灵活。

## 底座定义

每个底座是一组固定的、已验证过不互相打架的模块组合。统筹 LLM 按写作环节选底座，不从全库中零散挑。

### 观点底座（viewpoint_base）

适用环节：观点层 / 观点推理阶段

| 模块 ID | 类别 | 作用 |
|---------|------|------|
| `TRUST_POSTURE` | 姿态 | 不装懂、不逃避 |
| `CONSTITUTION` | 约束 | 宪法级约束 |
| `OPPOSITION_PANEL` | 席位 | 内部反方压力 |
| `CLAIM_TYPER` | 约束 | 声明分类器 |

### 大纲底座（outline_base）

适用环节：大纲层

| 模块 ID | 类别 | 作用 |
|---------|------|------|
| `SECTION_GOAL` | 推理 | 板块目标 |
| `QUESTION_BACKBONE` | 推理 | 问题驱动骨架 |
| `ARGUMENT_SKELETON` | 推理 | 论证骨架 |
| `REVERSE_OUTLINE` | 推理 | 逆向大纲验证 |

### 初稿底座（draft_base）

适用环节：成稿节点

| 模块 ID | 类别 | 作用 |
|---------|------|------|
| `EVIDENCE_LEDGER` | 约束 | 证据台账 |
| `LIMITATION_BINDER` | 约束 | 局限性绑定 |
| `OUTPUT_CONTRACT` | 输出 | 输出契约 |
| `PLAIN_LANGUAGE` | 输出 | 通俗化 |

### 审稿底座（review_base）

适用环节：后审节点

| 模块 ID | 类别 | 作用 |
|---------|------|------|
| `MEDICAL_SAFETY_AUDIT` | 核验 | 医学安全审计 |
| `CONTRADICTION_HUNT` | 核验 | 矛盾猎手 |
| `SOURCE_SUPPORT_CHECK` | 核验 | 来源支持验证 |
| `OMISSION_AUDIT` | 核验 | 遗漏审计 |

## 局部补丁规则

统筹 LLM 在选定底座后，可以按以下规则追加最多 2 个补丁模块：

### 补丁触发条件（硬触发）

| 触发特征 | 强制补丁 |
|----------|---------|
| 涉及疗效/安全性/机制/比较结论 | `CLAIM_TYPER`（如底座未含） |
| 涉及研究解读/证据包/原始数据 | `EVIDENCE_LEDGER`（如底座未含） |
| 面向患者/家属/媒体/大众 | `MISUNDERSTANDING_FIREWALL` |
| 进入终审 | `MEDICAL_SAFETY_AUDIT`（如底座未含） |
| 做核心主张/标题/差异化叙事 | `OPPOSITION_PANEL`（如底座未含） |
| 已有提纲但逻辑链断裂 | `AMNESIA_EDITOR` |
| 需要高密度压缩 | `COMPRESSION_EDITOR` |

### 补丁禁止规则

1. 单次总注入不超过 6 个模块（底座 4 + 补丁最多 2）。
2. 不允许同时注入 `PLAIN_LANGUAGE` 和 `COMPRESSION_EDITOR`——目标冲突。
3. 不允许同时注入 `OPPOSITION_PANEL` 和 `CHAIR_EDITOR`——前者做攻，后者做裁，须分步。
4. 不允许在成稿环节注入 `REVERSE_OUTLINE`——逆向验证是大纲阶段的事。
5. 不允许在观点层注入 `OUTPUT_CONTRACT`——输出格式是下游的事。

## 统筹 LLM 路由流程

```
1. 接收当前环节标识 (stage: viewpoint | outline | draft | review)
2. 加载对应环节底座
3. 读取任务特征 (audience, risk_level, evidence_maturity, task_goal)
4. 检查硬触发条件，追加必要补丁（不超过 2 个）
5. 检查补丁禁止规则，冲突则放弃低优先级补丁
6. 组装最终模块列表，注入下游节点
```

## 统筹 LLM 不做什么

1. 不从全库中无边界地自由挑模块。
2. 不在运行时改底座定义。
3. 不同时给一个节点注入两个目标相反的模块。
4. 不把模块正文整段塞给下游——只传模块 ID 和指令正文，不传元数据、说明和使用建议。
5. 不在没有环节标识的情况下做路由。

## 注入格式

统筹 LLM 向下游节点传递提示词时，格式固定为：

```
--- PROMPT MODULES ---
[MODULE: TRUST_POSTURE]
{模块指令正文}

[MODULE: CONSTITUTION]
{模块指令正文}
--- END PROMPT MODULES ---
```

下游节点只读 `--- PROMPT MODULES ---` 到 `--- END PROMPT MODULES ---` 之间的内容，不读外部库元数据。

## 与现有提示词的关系

当前各节点已有固定提示词文件（`prompts/` 目录下）。本协议定义的模块注入是**追加层**，不替代节点原有提示词。

拼装顺序固定为：

```
1. 节点原有提示词（如 draft_first.md）
2. 统筹 LLM 注入的模块指令
```

节点原有提示词定义职责和输出格式；模块指令定义工作姿态、约束和核验习惯。两者不互相覆盖。
