# 提示词基座组合避雷清单

- 状态：纸面分析完成 / 待真实样例验证
- 真相源：本文件是基座组合冲突排查的唯一定义
- 上游依据：`./12_统筹LLM提示词路由协议_2026-04-14.md`、`./医学写作系统_模块化提示词库_v1.md`

## 测试目标

本轮不选最优组合，只做避雷：找出会互相打架、重复注入、越界、失真的组合，输出禁用清单和风险清单。

## 测试方法

纸面分析 + 逻辑推演。每组底座的 4 个模块按"指令方向"和"行为效果"做交叉检查，识别三类问题：

1. **方向冲突**：两个模块对同一行为给出相反指令
2. **重复注入**：两个模块说的其实是同一件事，叠加后过度强化
3. **越界风险**：某模块在特定环节使用时会侵入其他环节的职责

## 一、观点底座（viewpoint_base）避雷

组成：`TRUST_POSTURE` + `CONSTITUTION` + `OPPOSITION_PANEL` + `CLAIM_TYPER`

### 内部冲突检查

| 组合 | 冲突类型 | 风险评级 | 说明 |
|------|---------|---------|------|
| `TRUST_POSTURE` + `OPPOSITION_PANEL` | 轻微张力 | 低 | TRUST 要求诚实探索，OPPOSITION 要求攻击式审查。方向兼容但力度可能拉扯——TRUST 偏"慢慢想清楚"，OPPOSITION 偏"快速挑毛病"。实际可共存，OPPOSITION 的攻击在 TRUST 的诚实框架内运作。 |
| `CONSTITUTION` + `CLAIM_TYPER` | 轻微重叠 | 低 | 两者都涉及"什么能说什么不能说"，但 CONSTITUTION 管宪法级底线，CLAIM_TYPER 管声明分类。不冲突，但注入时要注意不要在约束层重复说"不能把推断写成事实"。 |

**观点底座结论：可用。内部无硬冲突。**

### 常见补丁兼容性

| 补丁 | 与底座兼容 | 风险 |
|------|-----------|------|
| `CHAIR_EDITOR` | ⚠ 不可同时用 | OPPOSITION 做攻、CHAIR 做裁，两者是分步动作，不能同时注入同一轮 |
| `NO_BLUFF_POSTURE` | ✅ 兼容 | 与 TRUST_POSTURE 方向一致，略有重叠但不冲突 |
| `SLOW_IS_SMOOTH` | ✅ 兼容 | 加强观点层的审慎性 |
| `MISUNDERSTANDING_FIREWALL` | ✅ 兼容 | 患者向任务时追加 |
| `OUTPUT_CONTRACT` | ❌ 禁用 | 观点层不该管输出格式 |

## 二、大纲底座（outline_base）避雷

组成：`SECTION_GOAL` + `QUESTION_BACKBONE` + `ARGUMENT_SKELETON` + `REVERSE_OUTLINE`

### 内部冲突检查

| 组合 | 冲突类型 | 风险评级 | 说明 |
|------|---------|---------|------|
| `QUESTION_BACKBONE` + `ARGUMENT_SKELETON` | 方向张力 | 中 | QUESTION 以"问题驱动"做结构，ARGUMENT 以"论证驱动"做结构。两者不矛盾但可能给模型两个平行的结构化指令。需要在注入时说清：先 QUESTION 定问题框，再 ARGUMENT 填论证链。 |
| `REVERSE_OUTLINE` + 其他三个 | 时序问题 | 低 | REVERSE_OUTLINE 是"写完后反向验证"，但在大纲阶段没有"已写完的东西"可以反向。适用性存疑。 |

**大纲底座结论：基本可用，但 `REVERSE_OUTLINE` 在大纲阶段的适用性需要验证。** 如果大纲层输出的是"骨架"而不是"成文"，REVERSE_OUTLINE 可能没有足够内容可逆向。建议先保留，待真实样例验证后决定是否替换。

### 常见补丁兼容性

| 补丁 | 与底座兼容 | 风险 |
|------|-----------|------|
| `MECHANISM_TO_MEANING` | ✅ 兼容 | 帮助把机制转译成临床意义 |
| `BRIDGE_INFERENCE_MARKER` | ✅ 兼容 | 标记推理桥接点 |
| `COMPARE_BY_AXIS` | ✅ 兼容 | 比较性任务时追加 |
| `EVIDENCE_LEDGER` | ⚠ 注意 | 大纲阶段用证据台账可能过早——大纲是框架不是正文 |
| `COMPRESSION_EDITOR` | ❌ 禁用 | 大纲阶段不做压缩 |

## 三、初稿底座（draft_base）避雷

组成：`EVIDENCE_LEDGER` + `LIMITATION_BINDER` + `OUTPUT_CONTRACT` + `PLAIN_LANGUAGE`

### 内部冲突检查

| 组合 | 冲突类型 | 风险评级 | 说明 |
|------|---------|---------|------|
| `EVIDENCE_LEDGER` + `LIMITATION_BINDER` | 无冲突 | 低 | 一个管证据落地，一个管局限性标注，方向一致。 |
| `OUTPUT_CONTRACT` + `PLAIN_LANGUAGE` | 轻微张力 | 中 | OUTPUT_CONTRACT 对格式有硬约束，PLAIN_LANGUAGE 追求通俗化。如果 OUTPUT_CONTRACT 要求严格学术格式，PLAIN_LANGUAGE 的通俗化方向可能冲突。需要在注入时明确：格式按 OUTPUT_CONTRACT，语言风格按 PLAIN_LANGUAGE。 |
| `PLAIN_LANGUAGE` + `EVIDENCE_LEDGER` | 轻微张力 | 中 | PLAIN_LANGUAGE 追求"说人话"，EVIDENCE_LEDGER 追求"每句都有证据"。两者兼容但可能让模型在"写得通俗"和"写得严谨"之间犹豫。需要明确优先级：先保证证据落地，再追求通俗表达。 |

**初稿底座结论：可用，但 `OUTPUT_CONTRACT` 和 `PLAIN_LANGUAGE` 之间的格式/风格张力需要在真实样例中验证。**

### 常见补丁兼容性

| 补丁 | 与底座兼容 | 风险 |
|------|-----------|------|
| `COMPRESSION_EDITOR` | ❌ 禁用 | 与 PLAIN_LANGUAGE 目标冲突（压缩 vs 通俗展开） |
| `MISUNDERSTANDING_FIREWALL` | ✅ 兼容 | 患者向任务时追加 |
| `HOOK_DISCIPLINE` | ✅ 兼容 | 传播型稿件追加 |
| `DEEP_DIVE_MODE` | ⚠ 注意 | 与 PLAIN_LANGUAGE 方向相反，深度稿时需替换掉 PLAIN_LANGUAGE |
| `SLIDE_MODE` | ⚠ 注意 | PPT 输出时需替换掉 PLAIN_LANGUAGE 和 OUTPUT_CONTRACT |

## 四、审稿底座（review_base）避雷

组成：`MEDICAL_SAFETY_AUDIT` + `CONTRADICTION_HUNT` + `SOURCE_SUPPORT_CHECK` + `OMISSION_AUDIT`

### 内部冲突检查

| 组合 | 冲突类型 | 风险评级 | 说明 |
|------|---------|---------|------|
| 全部四个 | 无冲突 | 低 | 四个核验模块方向完全一致：都是"挑问题"。不同维度（安全、矛盾、来源、遗漏）互不重叠。 |

**审稿底座结论：最稳的一组。可直接使用。**

### 常见补丁兼容性

| 补丁 | 与底座兼容 | 风险 |
|------|-----------|------|
| `ASSUMPTION_EXPOSURE` | ✅ 兼容 | 追加假设暴露维度 |
| `FIGURE_TABLE_ALIGNMENT` | ✅ 兼容 | 有图表时追加 |
| `READER_RETELL_TEST` | ✅ 兼容 | 验证可理解性 |
| `COURTROOM_CROSS_EXAM` | ⚠ 注意 | 力度极大，可能导致过度否定，只在高风险任务时用 |
| `HOSTILE_PEER_REVIEW` | ⚠ 注意 | 同上 |

## 五、跨底座禁用组合汇总

| 禁用组合 | 原因 | 适用场景 |
|---------|------|---------|
| `PLAIN_LANGUAGE` + `COMPRESSION_EDITOR` | 目标冲突：展开 vs 压缩 | 全局 |
| `OPPOSITION_PANEL` + `CHAIR_EDITOR`（同轮） | 攻裁分步，不可同轮 | 观点层 |
| `REVERSE_OUTLINE`（成稿环节） | 越界：逆向验证是大纲/审稿的事 | 成稿阶段 |
| `OUTPUT_CONTRACT`（观点层） | 越界：输出格式不是观点层管的 | 观点层 |
| `EVIDENCE_LEDGER`（大纲环节） | 过早：大纲不需要逐句证据台账 | 大纲阶段 |
| `COMPRESSION_EDITOR`（大纲环节） | 越界：大纲不做压缩 | 大纲阶段 |
| `DEEP_DIVE_MODE` + `PLAIN_LANGUAGE`（同轮） | 方向冲突：深度 vs 通俗 | 初稿阶段 |

## 六、待真实验证清单

以下问题无法仅靠纸面分析解决，需要用真实样例跑验证：

1. `REVERSE_OUTLINE` 在大纲层是否真的有用——大纲是骨架不是成文，逆向验证是否有足够内容可操作？
2. `OUTPUT_CONTRACT` + `PLAIN_LANGUAGE` 在初稿层的格式/风格张力——具体到什么程度会互相打架？
3. `QUESTION_BACKBONE` + `ARGUMENT_SKELETON` 在大纲层——模型是否会因为两套并行结构化指令而犹豫？
4. 补丁上限 2 个是否够用——如果同时命中 3 个硬触发条件怎么办？
5. 观点底座用于观点层的实际效果——观点层还没有真实运行过，底座是否匹配待验证。
