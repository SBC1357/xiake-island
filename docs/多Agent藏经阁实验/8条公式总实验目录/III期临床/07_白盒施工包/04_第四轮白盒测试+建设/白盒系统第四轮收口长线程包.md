## 拉尔夫循环：阶段级硬闭环机制

本任务默认启用拉尔夫循环，而且是硬机制，不是可选建议。  
只要进入某一阶段，就必须先在该阶段内完成一轮闭环，再决定能不能进下一阶段。

### 核心定义

拉尔夫循环固定只有 `6` 步：

1. 执行本阶段当前目标
2. 阶段内自查挑错
3. 发现问题直接修补
4. 重新做静态验证；命中 live 条件时补最小 live 验证
5. 把真实结果回写到当前真相源
6. 由主 Agent 明确裁定：过闸 / 留在本阶段继续一轮 / 阻塞封板

### 强制规则

1. 不允许跨阶段裸推进。  
   任何阶段如果还没完成本轮自查、复验、回写，就不得宣布“进入下一阶段”。
2. 不允许把“已执行”冒充成“已闭环”。  
   做了实现但没复验，按未闭环处理。
3. 不允许把“已发现问题”冒充成“已吸收问题”。  
   找到问题但没修、没复验、没回写，按未闭环处理。
4. 不允许只在聊天里完成拉尔夫循环。  
   阶段结论、最近验证锚点、下一棒第一步，必须进当前正式真相源。
5. 不允许把主 Agent 的过闸裁定下放给子 Agent。  
   子 Agent 可以执行、检查、挑错、验证，但“是否过闸”“是否写回真相源”“是否进入下一阶段”只能由主 Agent 裁定。
6. 不允许用 `teams` 替代拉尔夫循环。  
   `teams` 只解决分工问题，不解决阶段闭环问题；开多 Agent 后，拉尔夫循环仍然逐阶段强制生效。
7. 不允许把“阶段内未解决问题”带病推进到下一阶段。  
   除非主 Agent 明确裁定该问题不影响当前阶段过闸，并把理由写回真相源。
8. 任何阶段一旦命中直接验证失败、口径冲突、真相源未回写，状态必须留在本阶段，或改成 `blocked / gated`，不得口头放行。

### 阶段过闸最低条件

每个阶段至少同时满足下面 `4` 条，才允许主 Agent 宣布过闸：

1. 本阶段目标已真实执行，不是纸面推演。
2. 本阶段新增或改动内容已做对应级别验证，至少达到“静态已核”；命中 live 条件时已补最小 live。
3. 本阶段结论、最近验证锚点、当前阻塞或下一步，已回写进当前唯一真相源。
4. 本阶段没有未裁定的关键口径冲突。

四条缺一，按不过闸处理。

### 与 `teams` 的关系

如果本轮开 `teams`：

1. 可以把执行、静态检查、log 收集、单点修补、冷审下派给子 Agent。
2. 不可以把阶段定义、真相源选择、最终裁定、是否过闸下派给子 Agent。
3. 每个子 Agent 的结果回到主 Agent 后，主 Agent 仍必须亲自补完拉尔夫循环的第 `5`、第 `6` 步。

### 失败处理

任何阶段如果连续一轮拉尔夫循环后仍未过闸，必须二选一：

1. 留在当前阶段继续下一轮，并明确写清当前唯一主假设。
2. 正式标记为 `blocked / gated`，写清最近失败锚点和下一棒第一步。

不允许第三种状态：

- 看起来做了很多，但没有阶段裁定
- 说“基本完成”但没有回写真相源
- 说“先往后做，回头再补”

### 阶段 7 当前产出

#### 迁移前风险清单

1. `layer1` 仍是包着 `route -> sp/direct -> intent/evidence/sample -> direct_gate` 的大盒：迁移时只要路径、编码、换行或退出码语义有差异，最前置的分流、证据、边界、放行就会一起受影响。证据锚点：`架构双图_真实_vs_理想.md` 中对 `layer1` 大盒与 `5` 个半 Agent 串行接法的说明。
2. `pre_review` 当前只是合同后的并行侧审，不是默认硬闸门：迁移后如果并发调度或落盘时点变化，这个节点容易“看起来跑了，实际上没拦主链”。证据锚点：`架构双图_真实_vs_理想.md` 中对 `pre_review`“默认不阻塞大纲和首稿”的说明。
3. `supplementary_evidence` 是独立确定性侧链，不是默认主链节点：迁移时最容易被误接成主链，或把补证结果误当主证据回填。证据锚点：`架构双图_真实_vs_理想.md` 中的 `deterministic_sidechain` 与“不回填污染主证据包”。
4. `post_review` 只消费当前可见的原始材料片段，且最多截到 `20000` 字：不同平台的文本抽取、编码、换行或文件读取差异，会直接改变后审看到的材料面。证据锚点：`架构双图_真实_vs_理想.md` 中对 `materials_manifest.full_text` 与截断上限的说明；`全貌.md` 中 `materials_compiled.txt` 已真实变成 `full text`。
5. 当前入口与调度仍高度依赖 `.ps1`、`whitebox_common.ps1`、环境变量和 Windows 路径语义：这仍是明显的平台耦合面。证据锚点：`架构双图_真实_vs_理想.md` 中大量节点直连 `run_whitebox_*.ps1`；`全貌.md` 中的 `XIAGEDAO_WHITEBOX_PROFILE_CONFIG_PATH` 覆盖路径。
6. 第一层分流和样稿边界对输入格式非常敏感：已经出现过 `intent_extraction` JSON 解析失败和 `sample_boundary` 正式阻断锚点。证据锚点：本任务包中保留的 `sample_boundary_probe.log`、`reference_sample_same_as_benchmark` 锚点与 `32% / 55%` 边界说明。
7. Persona 注入并不是全节点通用能力：理解/构思节点适合 Persona，协议/裁定/局改节点需要保持中性；迁移时若不守住这条边界，容易把风格层污染到规则层。证据锚点：`03_Persona注入适配分析.md` 中对 Persona 适配边界的裁定。

#### 公共层候选项

1. `prompt stack` 组装层：候选边界是 `profile.prompt_prefix + prompt_file + runtime 注入 + 输出协议` 的统一拼装机制；值得抽公共层，因为它已经是多节点真实共用的承载方式。
2. 节点输入输出契约层：候选边界是 `主吃 / 禁止吃 / 主产 / 回流` 四元组；值得抽公共层，因为当前节点分工已稳定，重复写在脚本、prompt、文档里只会放大口径漂移。
3. 冻结态状态机层：候选边界是 `route -> sp/direct -> intent/evidence/sample -> direct_gate`；值得抽公共层，因为它是整条链路的结构骨架。
4. 返修分诊层：候选边界是 `pass / requirement_rework / whole_revise / local_patch` 与 `local_patch_audit` 配对；值得抽公共层，因为这是稳定的返修决策接口。
5. 受众 / Persona 绑定层：候选边界是 `specialist / primary_care / patient_education / public_education` 与认知框架的映射；值得抽公共层，因为当前已证明它可通过 `prompt_prefix` 与运行时注入统一承载。
6. 证据与素材边界层：候选边界是原始材料、`materials_preview/full_text`、`evidence_pack json/md`、样稿边界、L3 资产选择的分层；值得抽公共层，因为它直接决定污染面且跨节点复用价值高。
