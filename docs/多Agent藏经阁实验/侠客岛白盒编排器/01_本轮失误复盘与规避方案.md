# 本轮失误复盘与规避方案

## 目标重述

本轮用户要的不是“若干建议”或“半套脚本”，而是：

1. 先不要碰侠客岛黑箱编排
2. 在本目录建设独立白盒编排器
3. 尽可能把 `II期` 自动化基座补全
4. 能由 Agent 自己解决的，必须先自己解决
5. 尽量把人工介入压到最后一次终审

## 本轮已发生失误

### P0：把半成品入口说成可直接交给新 session 跑

错误表现：

- 在白盒编排器只有“合同层”时，就给出了直接执行阶段入口的口径
- 没有先把取材、出稿、评分回指补齐
- 让用户误以为当前已经接近“只差终审”

为什么严重：

- 直接误导任务完成度
- 让下个 session 接到的是半成品，不是基座
- 本质上把后续关键施工外包给未来 session

规避方案：

1. 以后只有在“合同层 + 取材层 + 出稿层 + 评分回指层”都存在并真实串过后，才允许给出执行入口
2. 所有“直接跑这个就行”的说法，必须先有真实冒烟证据
3. 半成品入口只能叫“当前阶段入口”，不能叫“可托管基座”

### P0：让用户充当 QA 和缺陷发现器

错误表现：

- 目录落点错了，要用户指出
- 没先意识到白盒目录必须独立，要用户再强调
- 没先意识到新 session 不能只拿脚本，还要先读规则文档，要用户再提醒
- 没先意识到“只给建议、不给施工”正在违背任务目标，要用户反复追着纠偏

为什么严重：

- 把本应由 Agent 自己完成的自检责任转移给用户
- 让用户连续做需求翻译、缺口提醒和 bug 揪错
- 直接消耗用户时间和耐心

规避方案：

1. 用户明确说过“你能干的先干”后，必须把“继续建议”视为违规倾向
2. 任何要交给用户或新 session 的入口，先自己做“缺什么还没补”的逆向检查
3. 用户一旦连续指出两个以上同类问题，必须立即进入复盘与止血，而不是继续推进旧节奏

### P1：白盒目录落点过晚

错误表现：

- 白盒编排器一开始混在 `scripts` 相关目录推进
- 与用户要的“实验区独立目录”不一致

为什么严重：

- 容易和仓内其他自动化混口径
- 让后续优化对象不聚焦

规避方案：

1. 新建设工作开始前先确定最终长期落点
2. 涉及实验性基座时，先建独立目录，再施工
3. 不允许先混放，后迁移，除非有明确临时理由和迁移计划

### P1：把“继续补”当成阶段性输出

错误表现：

- 多次使用“如果你要，我下一步可以继续补……”
- 在目标已明确为施工时，仍不断给建议型半交付

为什么严重：

- 拖慢执行
- 让用户不断重复“继续做”的指令

规避方案：

1. 用户目标明确后，不再输出建议型半交付
2. 只允许两种输出：已完成施工结果，或真实阻塞证据

### P1：外部模型依赖没有预先降级设计

错误表现：

- 白盒出稿与评分最初直接把缺少外部模型 key 当成终点阻塞

为什么严重：

- 对“托管 session”场景不友好
- 容易把环境问题提前甩给用户

规避方案：

1. 缺 key 时至少自动产出 `draft_prompt.md` 与 `score_prompt.md`
2. 把阻塞从“用户继续处理”改成“托管 session 继续执行”
3. 先降级，再谈环境问题

### P1：没有第一时间把“Codex 宿主异常”和“整机网络栈异常”区分开

错误表现：

- 最初把白盒出稿/评分卡死理解成“缺 key”或“模型没接好”
- 没先验证“Codex 嵌入式 PowerShell”与“系统外部 PowerShell”是否表现一致
- 一度把整机网络栈也一并怀疑进去，阻塞归因过宽

为什么严重：

- 会把宿主级传输故障误写成整机级问题
- 会拖慢止血节奏
- 让“补 key”“换模型”这种无效动作排在真正问题前面

规避方案：

1. 以后只要白盒链路第一次触网，先做一次“当前宿主”和“系统外部 PowerShell”的对照诊断
2. 模型选择、认证、传输层要分开报错，不允许混成一个“调用失败”
3. 一旦确认只是 Codex 宿主异常，优先切到“外部 PowerShell 直跑白盒脚本”口径；只有外部 PowerShell 也失败，才切到“提示词落地 + session 回灌 + resume”

### P1：续跑入口首次实现时存在脚本参数透传 bug

错误表现：

- `run_ii_whitebox_resume.ps1` 最初用字符串数组把参数转发给 `run_ii_whitebox_draft.ps1 / run_ii_whitebox_score.ps1`
- 实际执行时发生参数错位，把题目 ID 误传成候选池路径

为什么严重：

- 续跑入口会在第一次回灌时直接崩
- 会让“已补 resume”变成假完成

规避方案：

1. 以后脚本间转发命名参数，一律优先用哈希表 splat
2. 新入口第一次接通后必须做一次真实续跑，不只做语法解析
3. 任何新加的 `resume / continue / replay` 类入口，都要有一次“从既有目录继续”的真实证据

### P0：把 `II期` 阶段目标偷换成局部执行项

错误表现：

- 一度把 `IIC` 讲窄成“前两篇稿子”或“当前两道题”
- 没及时把阶段建设工作、冻结条件、后续 `IID / IIE / IIF` 的设计链一起摆出来
- 需要用户反复追问“`IIC` 还有没有别的内容”“阶段本来还包括什么”

为什么严重：

- 直接把阶段设计问题降格成局部执行问题
- 会误导后续施工顺序
- 会让用户以为 Agent 没有真正理解 `II期` 结构

规避方案：

1. 以后凡是讨论阶段，必须先拆成“题目执行 / 阶段建设 / 冻结条件”三层
2. 不得先用局部显眼产物代表整个阶段
3. 用户一旦追问“这个阶段不止这些吧”，立即视为阶段目标表达失效，先收口再继续施工

### P0：证据链不完整时先给正向裁决

错误表现：

- 在原始整稿未单独留档时，先给出“通过 / 有条件通过”口径
- 后续又因证据不足和成稿质量问题改判为失败

为什么严重：

- 会让阶段判断失真
- 会把摘要级判断误包装成正式证据
- 会直接损害后续实验口径可信度

规避方案：

1. 原始整稿、评分依据、对照依据三者不齐，不得先给通过裁决
2. 只能先写“待裁决 / 证据不完整 / 仅诊断轮”
3. 同类情况以后视为高优先级流程事故，不再按普通漏项处理

### P0：把局部自动化能力说成整段 `II期` 自动化能力

错误表现：

- 在 runner、白盒入口、候选池只打通局部能力时，就给出了“其余自动化底盘已经尽量收口”这类口径
- 需要用户反向指出：`II期` 整段自动化还远没到“只差人工终审”

为什么严重：

- 会误导任务真实完成度
- 会让下一个 session 接到错误预期
- 本质上属于能力越级宣称

规避方案：

1. 以后自动化进度必须明确区分：单题、单批、单阶段、整段 `II期`
2. 没有打通上一层，不得越级描述下一层已完成
3. 所有“已收口 / 已搭好 / 可直接跑”的说法，必须带当前层级说明

### P0：关键设计纠偏与阶段口径纠偏没有同轮同步进主账

错误表现：

- 白盒 / 黑箱边界
- 藏经阁调取问题必须回流
- 后来又把白盒局部 `Round 3 / Round 4 / 5轮回路` 误升成 `II期` 主阶段口径

这些关键边界一开始只落在局部目录或聊天里，没有同轮同步到主总留痕；其中“`5轮回路`”后来还进一步污染了 `II期` 阶段判断。

为什么严重：

- 会让主账和局部目录口径分叉
- 会让后续接棒 Agent 只能看到局部真相
- 需要用户再次指出“为什么没留痕”

规避方案：

1. 凡是关键设计纠偏，必须同轮更新局部文档和主总留痕
2. 白盒局部 `round` 标签不得升格成 `II期` 主阶段；阶段裁决只认 `IIC / IID / IIE / IIF`
3. 只改局部目录不改主账，或只改白盒口径不改 `II期` 正式文件，均视为未收口
4. 用户已经明确纠偏过的设计项，不得再停留在聊天记忆里

### P0：在 `4/7` 后提前宣布进入人工终审，没有先把提分反推成系统优化

错误表现：

- 在白盒局部 run 出现 `4/7` 通过后，执行链直接往“人工审核阶段”跳
- 没先把提分点反向写回 `writing_contract / input_contract / orchestration_contract / materials_coverage`
- 也没先补齐 PDF 阅读链，再开下一轮验证

为什么严重：

- 会把“中间态高分”误包装成“系统已经收口”
- 会跳过真正关键的系统优化步骤
- 会让后续 loop 失去比较意义，因为提分原因还停留在局部现象，没沉到系统里

规避方案：

1. 以后只要本轮出现提分，下一步默认动作必须是“先反向优化系统”，而不是“升到人工终审”
2. `4/7`、单次高分、任一局部好看结果，都只能算中间证据，不得单独触发升环
3. PDF 阅读链未收口前，不得把任何一轮写成“系统已验证成功”
4. 只有 `5` 轮 loop 跑满后，才允许新增人工终审裁决

## 用户明确愤怒点

本轮用户的愤怒点不是一句“做慢了”，而是下面这些：

1. 需求已经说得足够清楚，却还要用户不断重复
2. 半成品被提前包装成“可以直接跑”
3. 关键缺口和 bug 还要靠用户自己来发现
4. 花了大量时间，却没有把 `II期` 自动化基座搭到用户要的层级
5. `II期` 设计链里很多关键漏项还要靠用户一条条追回，而不是 Agent 自己成串发现并同步固化

这些点都成立，不能粉饰。
