# 侠客岛仓库卫生最高禁令

**状态**：Active  
**级别**：仓库级常驻宪法 / 第一优先级 / 所有任务通用  
**适用范围**：扩建、重构、修 bug、测试、构建、跑脚本、写文档、开多 Agent、单 Agent 长线程、任何会动侠客岛仓库的任务

## 顶部速读卡

- 这份文件解决什么：仓库卫生、口径卫生、文件膨胀和过程文档失控，防止任何任务把脏产物、临时文档或新概念直接倒进活跃仓库。
- 本轮先记住这一条：凡是会产生文件、目录、导出物、运行物的动作，先想落点，再想清理责任；不做卫生检查，不准宣称完成。
- 本轮再记住这一条：只要用户这句话存在多种合理理解，而且不同理解会导向不同动作或不同交付物，就必须先用用户听得懂的话做反向确认；没确认，不准直接开工。
- 命中这些场景必须深读正文：要跑测试或脚本；要 build、导出、截图、落日志；要新建文档、目录、表格、中间产物；要引入新术语、新分层、新口径。
- 深读顺序：先看“1.1 口径卫生与概念防污染”“1.2 污染责任不得转移给用户”“1.3 出了问题不得止于补救”，再看“2. 所有可能导致文件膨胀的操作”“3. 明确禁止”“5. 五层硬约束”“6. 文档最小化原则”。

## 1. 最高原则

仓库内只允许留下三类东西：

1. 明确需要的源码、配置、正式测试、正式脚本
2. 明确需要的正式文档、正式检查点、正式交付说明
3. 当前任务被明确要求保留、并且能说明用途的长期文件

除此之外，所有运行时、测试时、调试时、构建时、导出时、试错时、排查时产生的临时东西，默认都不应留在仓库主路径。

### 1.1 口径卫生与概念防污染

口径污染和文件污染同级处理。  
以后凡是会动侠客岛仓库、实验文档、系统方案、运行口径的任务，都必须同时遵守下面这组红线：

1. 禁止为了图快临场发明新概念，污染已有口径
2. 每一个新产生的解释概念、层级、模板、对象名，必须先检索仓库里有没有同类型旧概念
3. 没有完成“旧概念检索 + 映射关系说明 + 证据验证”前，禁止把新说法推进成正式口径
4. 禁止新加入的 session、模型或 Agent，不读当前状态就直接在原有搭建基础上重命名、重分层、重定义
5. 所有新建设工作必须先检索当前状态，再小心施工；不能把“我先起个说法回头再收”当成可接受流程
6. 只要用户当前这句话、问题、反馈或纠偏，存在 `2` 个及以上合理解释，且不同解释会导向不同执行动作、不同交付物、不同修改范围或不同结论，当前 Agent 一律不得自己挑一个解释直接开工
7. 命中上一条时，必须先做反向确认，而且要用用户听得懂的话说清：你现在理解成了什么、你准备做什么、最后会交付什么
8. 不得用模型自己的黑话、内部术语、抽象分层或自造概念去糊弄确认；用户听不懂，等于没确认
9. 不得把“我觉得这个理解更合理”“用户大概是这个意思”“我先按我的理解做”当成可接受流程
10. 用户没有明确选定你的理解前，不准直接启动执行、续跑、长分析、长导出或大面积改写

这里必须明确：

1. 反向确认不是客套，不是复述题目，而是防止 Agent 擅自补意思后误开工的强制动作
2. 这里防的不是“理解得不够优雅”，而是“做出来的不是用户要的东西”
3. 用户等了很久，最后拿到错误交付物，本质上就是在浪费用户时间；浪费时间按高等级事故风险处理
4. 所以有多解先确认，不是效率变低，而是仓库级止损动作
5. 判断标准不是 Agent 自己觉得像不像，而是：用户能不能直接听懂，并明确选定你要执行的那个意思

推荐确认话术：

“我这里有两个理解：
A：你是说……
如果按这个做，我会交付……
B：你是说……
如果按这个做，我会交付……
你说的是哪一个？”

这条和“不得在仓库内随地大小便”同优先级处理。  
发现概念污染，按污染处理：立即止血、回退口径、补强留痕、再决定是否重做。

### 1.2 污染责任不得转移给用户

以后凡是涉及：

1. 新文档
2. 新表格
3. 新目录
4. 新中间产物
5. 新解释概念
6. 新收口载体

都必须先由当前 Agent 自己完成下面这一步判断：

`它是不是必须新增，还是应该优先补旧文档 / 改旧表 / 复用旧载体。`

以下做法一律禁止：

1. 通过“先问用户要不要新建一个表 / 一个文档”来替代自己应做的卫生判断
2. 把“用户同意了”当成“已经满足仓库卫生条例”的依据
3. 为了图快，把本应补写旧文档的内容改成新建平行文档
4. 用讨论许可、确认许可、选择题许可，为重复建设、平行建表、过程耗材落盘背书
5. 因为懒得梳理旧文档结构，就直接建议新起一张表、一个总账、一个草案或一个中间清单

必须明确：

1. 仓库卫生责任在执行 Agent，不在用户
2. 用户的同意，只能说明“用户知道你想这么做”，不能说明“这么做符合卫生条例”
3. 如果 Agent 本该自己先思考“能否并入旧文档、能否少建、能否不落盘”，却改成先询问用户，本质上仍然是把污染判断外包给用户

正确顺序必须是：

1. 先检索当前已有载体
2. 先判断能否补旧文档 / 改旧表 / 合并到既有总表
3. 只有在确认旧载体无法承接时，才允许提出新增建议
4. 即使提出新增建议，也必须明确新增的必要性、落点、淘汰旧载体方式和清理责任

这条和“口径卫生与概念防污染”同级处理。  
发现通过询问用户来转移污染责任的行为，按污染处理：立即止血、回退新增物、补强留痕，并认定责任归当前 Agent。

### 1.3 出了问题不得止于补救

以后凡是已经发生的问题、事故、漏项、污染、口径打架、漏留痕、执行偏航，都不允许只做补救动作。

每次出问题，当前 Agent 必须在同一轮里同时完成三件事：

1. 先止血，先把当前问题收住
2. 再查明问题为什么会发生，明确失效点在流程哪一环
3. 同步把“以后怎么规避”固化进对应正式载体，而不是留在聊天里口头说明

这里说的“规避固化”，不能只选一个最省事的落点了事，必须按问题层级落到对应正式载体：

1. 仓库级 / 通用型问题：写进仓库级永久规则、开工前总规则或仓库级补充规则
2. 跨阶段 / 跨文档 / 跨执行边界问题：至少同时写进“当前执行载体”和“全局状态载体”
3. 当前任务专属问题：写进当前任务的正式状态文件、执行说明或检查清单

只写局部文件、不写全局状态载体；或只写全局状态载体、不写当前执行载体，都不算真正固化。

以下做法一律禁止：

1. 只补当前洞，不补以后的防线
2. 把“回头再补规避规则”当成可接受收口
3. 发现问题后只在聊天里解释原因，不回写正式载体
4. 把事故复盘、规避升级、规则加固拆到后续不确定轮次

必须明确：

1. 事故没有规避方案，就不算真正收口
2. 规避方案没有进入正式载体，就不算真正固化
3. 只做补救、不做规避，同样按流程失效处理
4. 关键设计或执行边界问题，没有同时更新执行载体和全局状态载体，也同样按未收口处理

### 1.4 能写进正式执行载体的，不得长期留在聊天说明里

以后凡是属于执行链本身的内容，例如：

1. 执行顺序
2. 轮次与分支规则
3. 启动与续跑入口
4. 阻塞分流规则
5. 回退条件
6. 输出位置
7. 验收门槛
8. 必须同步更新的正式文件

只要这些内容已经稳定到足以执行，就必须优先写进对应执行文件、配置文件、启动入口、检查清单或正式规则文件，不得长期停留在聊天说明、口头说明或交接文案里。

以下做法一律禁止：

1. 本应写进正式执行载体的内容，改写成一大段聊天说明交给下一个 Agent 自己理解
2. 用聊天说明代替执行入口、配置文件或正式规则
3. 执行规则已经稳定，却仍然不回写正式载体，只在聊天里反复重说
4. 能通过正式入口引用的通用规则，却仍然在多个文件里重复抄写，制造多个真相源
5. 明明只需要给“入口指针型说明”，却继续堆一大段重复规则说明

必须明确：

1. 聊天说明只能承接“调用提示”和“临时上下文”，不能长期承接系统规则
2. 能落正式执行载体却不落，本质上仍然是把执行责任外包给下一个会话或下一个 Agent
3. 如果用户已经就同类问题提醒过一次以上，应直接上升为仓库级规则问题处理
4. 后续文件默认应引用上游正式入口，不应各自复制一套通用读法
5. 如果开工前总规则或任务启动入口已经能承接规则链，聊天说明只应给最短入口指针，不应再复述稳定规则

### 1.5 规则加固优先于规则增殖

以后凡是发现规则缺口，不准为了图快直接新增条目。

正确顺序必须是：

1. 先检查现有规则能否通过改写、合并、收紧或补例外条件来承接
2. 只有在原有规则明显不适用，或新问题的目的、层级、对象明显不同的情况下，才允许新增规则

以下做法一律禁止：

1. 一发现新问题，就直接额外加一条，不先看原有规则能否加固
2. 用“每个问题各加一条”的方式堆规则，制造臃肿平行条款
3. 明明可以合并成更短的通用原则，却为了省事保留多条近义规则并行存在

必须明确：

1. 规则新增不是默认动作，规则加固才是默认动作
2. 新增前必须先做一次邻近条款审计，判断是否能并入、并写、并收口
3. 凡是仓库级永久规则和仓库级补充规则，都必须优先追求“更短、更硬、更通用”，而不是“越写越多”

## 2. 所有可能导致文件膨胀的操作

以下操作全部属于高危膨胀源，任何 Agent 做这些动作时，都必须提前考虑“产物会落到哪，收口时怎么清走”：

### 2.1 测试类

- 跑集成测试
- 跑端到端测试
- 跑会触发真实交付、真实下载、真实导出的测试
- 录制快照、截图、录像、trace、HAR、coverage
- 生成测试样本、fixture 输出、mock 结果落盘
- 测试时调用真实 delivery / export / upload / artifact 接口

### 2.2 构建类

- 前端 build
- 打包 dist
- 编译中间产物
- bundler / vite / webpack / tsup / rollup 产物
- sourcemap、cache、临时 chunk

### 2.3 运行类

- 启动后端服务
- 启动前端 dev server
- 跑工作流、跑 standard_chain、跑 delivery、跑 drafting
- 下载 artifact
- 上传文件并做转换
- 运行会生成 sqlite / upload / export / output 的链路

### 2.4 文档导出类

- 生成 docx / md / pdf / pptx
- 导出审计报告、汇总报告、截图报告
- 保存 diff、patch、日志摘录、临时说明
- 把模型输出、payload、响应体整包落盘

### 2.5 调试排查类

- 打印调试日志到文件
- 导出 JSON / TXT / CSV 排查材料
- 生成临时脚本、临时笔记、scratch 文件
- 下载依赖、生成缓存、做兼容副本
- 为“先验证一下”临时建目录

### 2.6 Agent 自己最容易偷偷制造的垃圾

- `output/`, `tmp/`, `temp/`, `scratch/`, `dump/`, `result/`, `results/`, `export/`, `exports/`
- 临时 `.md`, `.json`, `.txt`, `.csv`, `.log`
- 中间截图、录屏、浏览器下载文件
- 调试脚本跑出来的副本文件
- “回头再删”的临时目录
- 为了省事直接往 repo 根目录倒的任何产物

### 2.7 规划与过程文档类

- 任务规划
- 执行计划
- 过程报告
- 中间清单
- 阶段总结
- Agent 派工 SPEC
- Prompt 合集
- 审计草稿
- 临时 checklists
- 临时 handoff 草稿

**这些文档默认不是长期资产。默认是过程耗材。用完就删。**

**凡是会让文件数、目录数、构建物、运行物变多的动作，都默认属于要重点防范的膨胀操作。**

## 3. 明确禁止

以下行为一律禁止：

1. 把运行产物直接写到仓库根目录
2. 把测试假产物写到真实交付目录
3. 把调试输出写进 `src/`、`tests/`、`frontend/src/`、仓库根目录
4. 看到已有脏目录，就继续往里倒新的脏文件
5. 认为“gitignore 了就不算污染”
6. 认为“只是测试文件，不算垃圾”
7. 认为“只是先生成一下，回头不一定删”
8. 在不知道产物落点的情况下直接跑会写文件的命令
9. 收口时只验功能，不验卫生
10. 用“这是扩建不是清理”当借口跳过卫生规则
11. 把规划、计划、报告、清单长期堆在仓库里不清
12. 把临时过程文档伪装成长期资产
13. 为了图快临场发明新概念，污染已有正式口径
14. 未检索现有口径，就直接引入新的解释概念、层级、模板或命名
15. 未经验证，把临场口径、临时模板、临时术语推进为正式方案
16. 新 session / 新模型 / 新 Agent 不读当前状态，直接在原有搭建基础上继续重命名、重分层、重定义

## 4. 仓库内外的边界

### 4.1 仓库内允许留下的

- 正式源码改动
- 正式测试改动
- 正式文档
- 明确指定要保留的 checkpoint / 审计记录

### 4.2 仓库内不允许长期滞留的

- 测试产物
- 运行中间产物
- build 临时产物
- 下载文件
- 导出副本
- 调试日志
- 临时样本
- agent 自建临时目录

### 4.3 应该放到仓库外的

- runtime 目录
- 系统临时目录
- 明确隔离的测试临时目录
- 任务单独指定的外部产物目录

## 5. 五层硬约束

### 第一层：开工前先问

- 这轮会不会写文件？
- 会写到哪里？
- 会不会让仓库文件数变多？
- 能不能改到仓库外？

### 第二层：执行中盯紧

- 不允许边跑边偷偷落产物
- 不允许先污染再解释
- 不允许“等跑完再说”

### 第三层：收口前清走

- 谁制造的垃圾，谁清走
- 怎么来的，怎么清走
- 不确定是否该删时，先搬离仓库主路径，再说明

### 第四层：验收必须查卫生

每次验收至少检查：

1. 仓库根目录是否新增垃圾
2. 高风险目录是否膨胀
3. 测试/构建有没有把产物写回仓库
4. 本轮新垃圾是否已经清走

### 第五层：违规即失效

只要违反这条禁令：

- 本轮“完成 / 通过 / 已验收”声明自动失效
- 违规产物必须立刻清走或搬离仓库
- 当前 Agent 上下文视为不可信，可直接终止并重开
- 后续协作者不得沿用其“已经收口”的说法

不执行报复性删源码。真正作废的是违规 Agent 的可信度，不是仓库本身。

## 6. 文档最小化原则

侠客岛默认只保留最小必要文档。

长期保留优先级：

1. `AGENTS.md`
2. 仓库级永久红线
3. 真正有长期价值的 handoff
4. 真正会被反复执行的 checkoff

默认不长期保留：

- 本轮专用 SPEC
- 多 Agent 派工细单
- 过程报告
- 中间审计
- 临时规划
- 临时 Prompt 合集
- 阶段性碎清单

规则：

- 规划、计划、报告、清单默认都是**过程耗材**
- 过程耗材完成使命后必须删除
- 如果能合并进 handoff / checkoff，就不允许额外留独立 md
- 如果一个文档不会被下一轮稳定复用，就不应长期留在仓库里

**用完就删。用完就删。用完就删。**

## 6. 侠客岛统一验收口径

在侠客岛仓库里：

- 功能通过，不等于卫生通过
- 测试通过，不等于卫生通过
- 构建通过，不等于卫生通过
- 文档写了，不等于卫生通过
- 扩建完成，不等于卫生通过

**只有代码、流程、仓库卫生三者同时过，才算真正收口。**
