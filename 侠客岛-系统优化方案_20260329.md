## 1. 总体判断

当前侠客岛的主要问题不是“核心代码里到处写死绝对路径”，而是以下 4 类结构性耦合仍然比较重：

1. 运行态文件与仓库工作区此前强耦合；`P0` 已经把默认落点迁出仓库，但运行态边界仍需继续固化。
2. 上传链的基础持久化已经补齐，但还缺少更强的全局索引治理、保留策略和迁移策略。
3. app-level 主链测试已经和 provider 环境解耦，但真实模型 smoke test 仍应和日常回归分层维护。
4. 前后端入口的基础解耦已经补上，但默认仍是 bundled 模式，前端产物仍是当前默认入口的一部分。

这意味着后续优化应该沿着“运行态边界继续收口”和“测试稳定化”推进，而不是先做大规模重构。

## 3. 优化优先级

### P0: 运行态出仓

目标：

- 让任务库、上传目录、日志目录默认不再写进仓库工作区

建议动作：

1. 把 `XIAGEDAO_DATA_DIR` 默认值从仓库内 `./data` 改成仓库外的可配置运行目录
2. 把 `XIAGEDAO_UPLOAD_ROOT` 默认值改成同一运行根目录下的 `uploads/`
3. 明确区分：
   - `repo state`
   - `runtime state`
4. 开发环境脚本统一注入运行目录，而不是依赖“代码默认值写回仓库”

预期收益：

- 清理仓库时不再和运行验证互相污染
- 运行历史可以保留，但不再成为工作区噪音

当前状态：

- 已实现：
  - 新增统一运行态路径层 [src/runtime_paths.py](/mnt/d/汇度编辑部1/侠客岛/src/runtime_paths.py)
  - 日志存储默认从仓库内 `./data/tasks.db` 迁到外部 runtime root
  - 上传链默认从仓库内 `src/data/uploads/` 迁到外部 runtime root
  - 共享 `TaskLogger` 从导入时初始化改为按需初始化
  - 后端开发脚本会默认创建并注入仓库同级 runtime root
- 已验证：
  - `pytest` 定向子集通过
  - `/health` 不再创建仓库内 `data/tasks.db`
  - `/v1/tasks` 首次访问时会在外部 runtime root 下创建 `tasks.db`

### P1: 补齐上传链持久化

目标：

- 让上传记录和落盘文件使用同一套持久化边界

建议动作：

1. 为 `UploadService` 增加持久化索引层
2. 至少持久化：
   - `upload_id`
   - 原始文件路径
   - 转换状态
   - 页面图像清单
   - 证据片段摘要
3. 明确“重启后如何恢复上传状态”
4. 再决定是否继续保留当前“文件落盘 + 内存索引”的一期模式

预期收益：

- 上传链不再“文件还在、系统却失忆”
- 后续才能安全做真正的上传回放、审计和问题复现

当前状态：

- 已实现基础版：
  - 每个 upload 目录现在都会持久化 `manifest.json`
  - `UploadService` 初始化时会自动回读 manifest 恢复内存索引
  - 失败态和完成态都会被持久化
- 已验证：
  - `tests/modules/test_upload_persistence.py` 通过 `2/2`
  - 完整上传链和失败上传链都能在服务重建后恢复状态
  - `tests/integration/test_upload_api.py` 已验证单 upload 删除接口可清除运行态文件与索引
- 仍待后续补齐：
  - 全局索引治理

  - 批量生命周期/清理策略

### P1: 主链测试 hermetic 化

## 4. 建议执行顺序

建议按这个顺序推进：

1. `P0 运行态出仓` 已完成
2. `P1 上传链持久化` 与 `P1 主链测试 hermetic 化` 的基础版已完成
3. `P2 入口解耦` 的基础版已完成
4. 下一步进入“扩展前基线提交/归档”，而不是继续处理低价值硬编码噪音

原因：

- 前 3 项直接影响“能不能稳定优化”和“工作区会不会持续被运行态污染”
- 当前 P3 的高价值部分已经落地，后续重点应转向基线固化和新功能扩展

## 5. 与当前清理工作的关系

这轮文件夹清理与 `P0/P1` 实现一起证明了三件事：

- `data/tasks.db` 和 `data/uploads/` 可以删除后按需重建
- 它们已经可以默认迁出仓库，而不影响主运行链
- 上传链元数据已经可以跟随 upload 目录一起跨重启恢复

同时，`P2` 也证明：

- 后端现在可以在 `api-only` 模式下独立验证，而不必总是依赖 `frontend/dist`

这恰好说明“运行态出仓”是可行的，因为它们本来就不是源码静态资产，而是运行时生成物。

同时，这轮清理也证明：

- `frontend/dist` 目前仍承担后端根入口职责

所以入口解耦要放在后面做，而不是一上来就把它当普通缓存处理。

## 6. 不建议现在就做的事

- 不建议一上来改大范围模块接口
- 不建议先重命名大量文档与环境变量
- 不建议在主链测试仍不稳定时直接做大重构
- 不建议把上传链直接接到现有任务库里而不先定义恢复边界
