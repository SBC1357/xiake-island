> **适用对象**：像你这样用 vibe coding 搭系统、但没有代码背景的人；后续接手系统的 Agent；需要继续开发的人  
> **文档目的**：用人话把当前系统讲清楚，同时不把“理想状态”写成“已经实现”  
> **系统组成**：侠客岛（执行层）+ 藏经阁（知识层）  
> **本次整理时间**：2026年3月26日  
> **本次直接核对范围**：模块目录、前端路由、主工作流路由、藏经阁消费者目录、测试文件数量  
> **重要说明**：本次是文档结构优化，不是全量重新验收；文中凡涉及“当前可见结构”“当前目录状态”的内容，优先以本次直读结果为准

## 1. 先记住这 7 句话

如果你没有代码背景，只要先记住下面 7 句话，就已经抓住了这个系统的主干。

1. **这不是一个“一键自动出终稿”的魔法机器。**
   它更像一条有多个工位的写作流水线。
2. **侠客岛负责“干活”，藏经阁负责“提供知识和规则”。**
   前者是执行层，后者是知识层。
3. **标准主链现在是 6 步。**
   `Evidence -> Planning -> Writing -> Drafting -> Quality -> Delivery`
4. **Writing 不是正文。**
   Writing 现在主要产出的是“写作指令 / Prompt”。
5. **Drafting 才是正文。**
   真正的文章内容是在 Drafting 这一步生成的。
6. **Delivery 才是正式交付。**
   它负责把正文做成 `docx` 等可交付文件。
7. **这个系统已经能跑主链，但还不是完全产品化。**
   它现在更像“能工作的系统 + 可继续打开的灰盒”，不是完全收口的商业成品。

---

## 2. 这套系统到底是什么

一句话说：

**它是一套医学内容写作系统。侠客岛负责把任务跑起来，藏经阁负责给它提供写作所需的知识、规则和表达资源。**

如果再说得更直白一点：

- **侠客岛**像“写作工厂”
- **藏经阁**像“知识仓库 + 规则仓库”
- 你给系统一个写作任务后，侠客岛会去藏经阁取材料、定结构、编写作指令、生成正文、做质检、再输出正式文件

这套系统最核心的价值，不是“会不会写一篇文章”这么简单，而是：

1. 让写作过程拆成多个可观察的步骤
2. 让知识和执行分层
3. 让后续 Agent 接手时，不用只靠读代码猜系统意图

## 3. 侠客岛和藏经阁，分别负责什么

### 3.1 侠客岛：执行层

侠客岛负责把任务真正跑起来。

你可以把它理解成一个“写作流水线控制台”。  
它现在直接核对到的业务模块有 8 个：

- `opinion`
- `semantic_review`
- `evidence`
- `planning`
- `writing`
- `drafting`
- `quality`
- `delivery`

但如果只看你现在最关心的“医学文章主链”，最核心的是下面 6 个：

- Evidence：取证
- Planning：规划
- Writing：编写作指令
- Drafting：生成正文
- Quality：审正文
- Delivery：交付文件

### 3.2 藏经阁：知识层

藏经阁不是外部黑盒。  
它是**你可以直接维护的知识层**。

你可以把它理解成：

- 产品资料库
- 疾病知识库
- 写作规则库
- 医学表达与风格库

侠客岛默认**只读**藏经阁，不回写。  
也就是说：

- 侠客岛负责“用”
- 藏经阁负责“存”
- 你和后续 Agent 可以直接改藏经阁里的规则和知识文件

## 4. 你最容易混淆的 3 个东西

这是整套系统里最值得先讲清的一件事。

### 4.1 Prompt

Prompt 可以理解成“给大模型的写作指令”。

在这套系统里，Writing 这一步目前最主要产出的就是：

- `system_prompt`
- `user_prompt`
- `writing_trace`

所以 **Prompt 不是正文**。

### 4.2 正文

正文是系统真正生成出来的文章内容。

在当前主链里，正文主要出现在 Drafting 这一步，常见字段是：

- `content`
- `word_count`
- `trace`

所以 **正文不是 Prompt**，也不是交付文件。

### 4.3 交付物

交付物是最后可以给人拿走、下载、归档的东西。

在当前主链里，Delivery 会把正文变成：

- `docx` 正式文件
- `md` 预览文件

所以 **交付物也不是正文本身**。

一句话总结这三个概念：

- **Writing = 告诉模型怎么写**
- **Drafting = 真正写出正文**
- **Delivery = 把正文做成正式文件**

## 5. 一次写作任务，系统内部到底怎么走

如果你给系统一个医学写作任务，当前最值得理解的是“标准六段链”。

它现在的真实主链是：

`Evidence -> Planning -> Writing -> Drafting -> Quality -> Delivery`

下面这张表，是给非代码背景读者看的主链说明。

| 阶段 | 这一步在干什么 | 当前真实产出 | 你最容易误会什么 |
|---|---|---|---|
| Evidence | 去藏经阁找跟这次任务有关的证据和事实 | 事实记录、筛选结果、取证追踪 | 它不会自动帮你联网找最新论文 |
| Planning | 根据证据决定“这篇文章想说什么、怎么排结构” | 论点、结构、大纲、策略编号、叙事弧编号 | 这不等于已经产出成熟策划案 |
| Writing | 把规划结果编译成“给模型的写作指令” | `system_prompt`、`user_prompt`、`writing_trace` | 这一步不是正文 |
| Drafting | 根据 Prompt 生成真正的正文 | `content`、`word_count`、生成追踪 | 这一步才是正文生成 |
| Quality | 审查正文能不能进入交付 | 通过/警告/失败，以及对应问题 | 它已经有规则能力，但还不是完全统一的终审体系 |
| Delivery | 把通过质检的正文变成正式交付文件 | `docx`、`md`、输出路径、交付字数口径 | 它负责交付，不再应该被理解成“主线成稿器” |

## 6. 把六段链再翻译成人话

### 第 1 站：Evidence

系统先去藏经阁找这次写作任务能用的证据。

你可以把它理解成：

- 先找材料
- 再过滤材料
- 再记录“为什么留下这些材料”

它现在已经不是纯黑箱，因为至少会返回取证追踪（`retrieval_trace`）。

但它也还不是完全白盒，因为：

- 证据排序逻辑
- 充分性判断
- 证据之间的筛选优先级

还没有完全讲透。

### 第 2 站：Planning

系统根据证据，先定“这篇文章要写什么”和“怎么写”。

它当前已经能给出：

- 论点（`thesis`）
- 大纲（`outline`）
- 策略编号（`play_id`）
- 叙事弧编号（`arc_id`）
- 风格说明（`style_notes`）

但你要保守理解：

这一步是“形成规划”，不是“形成已经很成熟的人类策划方案”。

### 第 3 站：Writing

系统把前面的规划，翻译成可以喂给模型的写作指令。

这一步最容易造成错觉。  
因为很多人看到“Writing”这个名字，会自然以为这里已经“写出了文章”。

但当前真实状态不是这样。

当前 Writing 的核心职责是：

- 组装 `system_prompt`
- 组装 `user_prompt`
- 记录 `writing_trace`

所以这里更像“写作指令编译器”，不是“正文生成器”。

### 第 4 站：Drafting

这一步才是真正把 Prompt 变成正文。

如果你想知道“系统到底哪一步写出了文章”，答案是：

**当前主链里，正文主要由 Drafting 负责生成。**

这一步对你最重要，因为：

- 它决定系统是不是已经真的出了正文
- 它也决定后面的 Quality 到底是在审什么

### 第 5 站：Quality

这一步是正文质检。

它现在已经不是完全黑箱，因为：

- 有规则引擎
- 有规则执行追踪（`rule_trace`）
- 有不同检查门

但它仍然是灰盒，因为不同入口现在跑的检查范围还没有完全统一。

所以正确说法应该是：

- **Quality 已经有真实审核能力**
- **但还不能吹成一套完全统一、完全收口的终审系统**

### 第 6 站：Delivery

这一步负责把正文做成交付物。

当前最重要的事实是：

- 正式交付物以 `docx` 为主
- `md` 主要是预览
- 标准主链里，Delivery 接收的是 Drafting 产出的正文

所以如果你问“最终给人拿走的文件在哪一步出现”，答案就是 Delivery。

## 7. 当前最重要的 3 个入口

这套系统前端现在已经不只是很早期的几个页面了。  
从当前路由文件直接看，页面入口已经比旧文档里记录的更多。

但如果只从“理解系统主链”出发，你最值得记住的是这 3 类入口。

### 7.1 文章工作流（article）

它是一条**轻量链**。

当前你应该把它理解成：

- Opinion
- Semantic Review

也就是：

**快速跑“观点生成 + 语义审核”这条短链。**

它不是完整写作主链，也不等于正式交付链。

### 7.2 标准六段链 API（standard-chain）

这是当前最接近“后端完整写作主链”的入口。

你可以把它理解成：

- 后端一把跑完整流程
- 从取证一直跑到交付

如果你想看主链真实长什么样，这个入口比轻量 article 更关键。

### 7.3 标准链工作台（前端）

当前前端里，标准链更像一个“步骤工作台”。

它的价值主要是：

- 让你逐步看每一站在干什么
- 方便你观察 Prompt、正文、质检、交付物的区别
- 方便调试，而不只是黑箱式一键运行

所以当前它更像：

- 调试工作台
- 观察面板
- 分步工作界面

还不应该直接吹成“完全成熟的一键写作产品页”。

## 8. 藏经阁应该怎么理解

藏经阁当前可以理解成 4 层。

| 层级 | 你可以怎么理解 | 当前主要作用 |
|---|---|---|
| L1 | 通用写作工艺层 | 放比较通用的表达规则、语体、禁用词、通用写作资源 |
| L2 | 医学写作剧本层 | 放医学写作更具体的剧本、叙事弧、句式模板、合规规则 |
| L3 | 疾病知识层 | 放疾病知识、论点推导规则、来源注册表 |
| L4 | 产品证据层 | 放具体产品的证据、产品档案、产品表达与合规配置 |

这 4 层的关系，不是并列摆设，而是从“更通用”到“更具体”逐层收窄。

你可以这样记：

- L1：所有文章都会碰到的通用规则
- L2：医学写作的通用打法
- L3：某个疾病方向的知识和论点规则
- L4：某个具体产品的资料和表达资产

当前直接核对到：

- 藏经阁消费者目录下有 `203` 个 JSON 文件
- L4 当前可见的主产品目录有 6 个：
  - `donanemab`
  - `furmonertinib`
  - `lecanemab`
  - `lemborexant`
  - `pluvicto`
  - `trastuzumab_deruxtecan_gastric`

但这里要特别注意：

**这些数字会变。**  
它们适合做“当前快照”，不适合被写成永远不变的制度性口径。

## 9. 白盒、灰盒、黑盒：你应该怎么看

### 9.1 白盒

白盒的意思是：

- 已经比较透明
- 逻辑边界比较清楚
- 你和后续 Agent 相对容易看懂和修改

当前更接近白盒的部分，可以保守理解为：

- Opinion
- Semantic Review
- Asset Bridge
- LLM Gateway
- Runtime Logging
- Contracts

### 9.2 灰盒

灰盒的意思是：

- 已经能跑
- 也已经有一部分结构暴露出来
- 但内部逻辑、边界或解释口径仍然不够透

当前最典型的灰盒，就是主链核心模块：

- Evidence
- Planning
- Writing
- Drafting
- Quality
- Delivery
- 主编排器
- 藏经阁 L1-L4 的“规则如何组合、如何选择、如何生效”

你可以把灰盒理解成：

**系统已经不是完全不知道它在干什么了，但也还没有白到“谁来都能一眼看透”。**

### 9.3 黑盒

黑盒的意思是：

- 你无法完全控制
- 它的输出不可能完全确定
- 你只能围着它做约束和校验

当前最典型的黑盒就是：

- 外部 LLM 能力

所以这套系统永远都还需要：

- Prompt 约束
- 审核机制
- 交付前判断

不能把 LLM 当成完全可控的流水线零件。

## 10. 当前最容易误解的 6 件事

### 10.1 “写作”不等于“正文已经写出来”

当前 Writing 主要产出的是 Prompt，不是最终文章。

### 10.2 “标准六段链能跑”不等于“已经是成熟成文产品”

它现在已经是一条真实可跑的主链，但前端和交付体验仍有明显产品化空间。

### 10.3 “Quality 有审核能力”不等于“已经统一终审口径”

不同入口的检查门还没有完全统一，所以不能把它说成终审体系已经彻底收口。

### 10.4 “藏经阁是知识层”不等于“它不能改”

恰恰相反，藏经阁是可直接维护的知识层，只是侠客岛默认对它只读。

### 10.5 “Delivery 能输出文件”不等于“成品感已经完全成熟”

当前已经能输出正式 `docx`，但成品版式、模板感、历史管理体验仍可继续优化。

### 10.6 “前端页面变多了”不等于“每个页面都是稳定主入口”

当前前端路由比旧稿记录的更丰富，但真正对系统理解最关键的，仍然是：

- Dashboard / 新建任务
- 标准链工作台
- 文章工作流
- Delivery Center
- Knowledge Assets

## 12. 如果后续 Agent 要接手，最先该看什么

如果后续是 Agent 接手，建议按这个顺序理解：

1. **先看主链，不要先沉迷模块目录。**
2. **先分清 Prompt / 正文 / 交付物。**

4. **先看灰盒，而不是先追白盒。**

- 当前真实主链是什么

## 附录 B：只给开发者或接手 Agent 看的入口
