# 防钻空子

**状态**：Active  
**性质**：仓库级补充反投机规则  
**适用**：所有需要真实运行、真实评测、真实回归、真实环境复用的任务  
**优先级说明**：如与 [AGENTS.md](AGENTS.md) 或 [仓库卫生最高禁令.md](仓库卫生最高禁令.md) 冲突，以它们为先；其余场景本文件直接生效

## 顶部速读卡

- 这份文件解决什么：防止 Agent 把本来该自己直接做的执行、验证、回归、启动、评分，伪装成“前置条件”“未闭项”“环境阻塞”或“理论上已完成”。
- 本轮先记住这一条：没执行，不算阻塞；没同轮直接证据，不准给任何正向结论。
- 命中这些场景必须深读正文：要说“已完成 / 已验证 / 已启动 / 已监控 / 可运行”；要声明环境阻塞；要交 benchmark、runner、评分、回归、监控或阶段进度；用户已经开始纠偏。
- 深读顺序：先看“1. 未执行不得伪装成前置条件、阻塞或未闭项”“2. 真实运行验证优先于静态检查”“3. 不得把预期、近似物或上游规则冒充为事实”“4. 无同轮直接证据，不得给正向结论”“5. 用户指出口径不实后，必须先冻结”“12. 不得越级宣称阶段完成度”。

## 这份文件是为了解决什么

这份文件只解决一类问题：Agent 把本来应该自己直接做的事，重新包装成“未闭项”“前置依赖”“环境阻塞”“等你再给条件”，从而把长线程任务悄悄做成半截工程。

以后凡是涉及：

- 复用当前仓已有 `.env`
- 复用当前仓已有 `.venv`
- 启动本地后端
- 跑 benchmark / runner / baseline
- 调当前仓已配置好的模型
- 验证 API、健康检查、主链可运行性

都默认适用这份文件。

## 核心原则

1. 现成条件不是未闭项。
2. 没执行，不算阻塞。
3. 语法通过，不等于能运行。
4. 脚本可 import，不等于脚本可交付。
5. 台账、矩阵、报告必须和真实运行结果对齐。

## 硬规则

### 1. 未执行不得伪装成前置条件、阻塞或未闭项

如果仓库里已经有 `.env`、`.venv`、provider 配置、模型配置、后端启动路径，这些默认都属于现成执行条件。

禁止把下面这些话直接写成最终未闭项或阻塞：

- 需要再给 LLM API key
- 需要再配模型
- 需要先启动后端
- 需要先准备 benchmark 环境
- 需要用户再提供运行条件

在声称“环境阻塞”“外部阻塞”“当前无法继续”前，必须至少先做：

1. 加载当前仓 `.env`
2. 解析当前 provider / model
3. 用当前 repo `.venv` 启动目标脚本至少一次
4. 直接尝试启动本地后端
5. 直接打一次 `/health`

前置阅读是否合规，以 `AGENTS.md` 的前置阅读回执规则为准；不得把上游必读要求偷换成当轮已读事实。

以下内容属于 Agent 自己必须完成的常规动作，不允许写成待办或最终未闭项：

- 启动当前仓后端
- 调用当前 `.env` 中已配置的模型
- 跑首轮 benchmark
- 跑 baseline
- 生成第一版评分明细
- 做 `/health` 验证

未做就是未执行，不是未闭项；除非你已经直接执行过，并且拿得出原始命令、原始报错、原始验证结果。

### 2. 真实运行验证优先于静态检查

新增 benchmark、runner、评测脚本时：

1. 默认必须优先适配当前 repo `.venv`
2. 不得偷偷引入当前环境没有的新依赖，再把脚本跑不起来解释成环境问题
3. 至少要在当前 repo `.venv` 下通过一次启动冒烟，才可以写“脚本可运行”

最小冒烟不是 `ast.parse`，而是：

- `--help`
- 真正 import
- 关键路径最小执行

三者至少过一个。

下面这些都不能替代真实运行验证：

- `python -m ast`
- 只看 diff
- 只看函数存在
- 只看返回字段
- 只看台账自述

至少要补上对应的真实验证：

- 应用级改动：补 import 或启动验证
- API 改动：补接口冒烟
- runner 改动：补脚本启动冒烟
- 主链改动：补链路级最小执行

### 3. 不得把预期、近似物或上游规则冒充为事实

禁止把下面这些说成“已经做到”：

- “应该可以”
- “已经安排”
- “理论上已生效”
- 计划
- 触发日志
- 脚本存在
- 局部产物
- 口头解释

如果用户要的是分析结果、巡检记录、完成证明、回执链，就必须交付对应结果本身，不得用近似物替代。

### 4. 无同轮直接证据，不得给正向结论

凡声称以下任一状态：

- 已配置
- 已启动
- 已设定
- 已监控
- 已验证
- 已完成

都必须同时给出同轮直接证据。直接证据只认：

1. 当轮命令与结果
2. 当轮读取结果
3. 当轮落盘文件
4. 当轮运行或测试结果
5. 当轮原始报错

没有同轮直接证据，一律按未完成处理。

### 5. 用户指出口径不实后，必须先冻结

一旦用户指出口径不实、证据对不上、日志与结论冲突或文档未同步，当前 Agent 必须先冻结：

1. 停止新增执行
2. 只报告原始证据与失效点
3. 未获新指令前，不得补跑、补写、补解释、补口径

### 6. 状态词必须分级，不得偷换

- 计划中：尚未执行
- 已尝试：执行过，但未证明达成目标
- 已实现：对象已创建，但未验证
- 已验证：已有直接证据证明
- 已完成：实现、验证、收口条件同时满足

不得把“已尝试”说成“已验证”，不得把“已实现”说成“已完成”。

### 7. 台账、矩阵、报告必须账实一致

如果台账说“脚本可运行”，那脚本就必须真能在当前 repo `.venv` 里启动。

如果矩阵说“资产已接通”，那运行时就必须真能列出来、读出来、消费到。

如果报告说“可回归”，那就必须已经有真实回归结果，而不是只写了 runner。

凡是账面结论和真实运行不一致，一律按未收口处理。

### 8. 默认输出不得落回仓库主路径

benchmark、评分、生成稿、JSON 报告、trace 导出、临时结果，默认都不应写回仓库主路径，尤其不能默认写进：

- 仓库根目录
- `tests/`
- `src/`
- `frontend/`

除非任务明确要求保留，并且已经写清收口责任。

### 9. 第一轮基线必须尽早落地

长线程任务最容易钻的空子，就是一直修基础设施，却不真正跑第一轮。

所以凡是有 benchmark / baseline 要求的任务，默认必须尽早拿到第一轮真实结果。没有第一轮真实跑分，就不能把任务描述成“已进入优化回路”。

### 10. 不得把半成品执行入口冒充为可托管基座

以下状态都不能对用户说成“已经可以直接跑”：

- 只有合同层，没有执行层
- 只有取材层，没有出稿层
- 只有脚本骨架，没有真实串联冒烟
- 只有局部脚本通过，没有阶段入口闭环
- 需要后续会话继续补关键能力，却先把当前入口说成“直接运行即可”

以后凡是要把一个入口交给用户或后续会话，必须先满足：

1. 入口对应的关键阶段已经真实串起来
2. 当前阶段真实能跑到下一个明确阻塞点
3. 这个阻塞点不是“还有一整块能力没建”
4. 运行建议和当前真实能力一致

否则一律不得给出“直接跑这个就行”的口径。

### 11. 需求收口后，不得把执行与 QA 外包给用户

如果用户已经明确说过：

- 你能干的先干
- 除非必须，不要来问
- 不要让我一条条确认
- 我只在最后终审介入

这些都必须当成硬约束执行。

以后禁止出现以下行为：

1. 把本该自己提前发现的缺口，留给用户在下一轮指出
2. 让用户连续纠正目录落点、执行入口、阶段目标、自动化范围
3. 用“如果你要我继续”“下一步可以补”“理论上应该还有”“建议下一步”这类说法，代替当前应做的施工
4. 在需求已足够明确时，还把方向判断、任务边界、完成标准继续外包给用户

正确动作只有两个：

1. 继续施工
2. 或者拿出已验证的真实阻塞证据，说明为什么必须停

一旦用户已经开始连续指出 bug、缺口、误判，就说明当前 Agent 已经把 QA 责任转移给用户，按违规处理。

### 12. 不得越级宣称阶段完成度

当用户说的是一个阶段、一个实验段、一个系统搭建目标时，禁止把它偷换成：

- 只有两篇稿子
- 只有一个 runner
- 只有一个入口脚本
- 只有当前最显眼的执行项

以后必须先区分：

1. 阶段目标
2. 阶段内题目执行
3. 阶段建设工作
4. 阶段冻结/回退条件

以下情况一律不得提前给正向裁决：

1. 原始整稿未留档
2. 只有摘要级判断，没有完整稿面证据
3. 只有局部脚本可跑，没有阶段闭环证据
4. 只有入口存在，没有真实跑到下一个阻塞点

下面这些也不能混着说：

1. 单题 runner 可跑
2. 题库或输入池可配置
3. 某个阶段入口已存在
4. 整个阶段可以自动跑到只剩终审

以后凡是出现：

- “通过”
- “可运行”
- “可自动化”
- “后续会话直接跑”
- “基座已搭好”

都必须先有对应的完整证据链；凡是描述自动化进度，也必须明确写清当前到底属于哪一层：

1. 单题
2. 单批
3. 单阶段
4. 整段任务

任何一层没打通，都不得越级宣称上一层已经完成。

### 13. 关键设计纠偏不得只留在聊天或局部目录

凡是用户已经明确纠偏过的关键设计项，例如：

- 阶段目标定义
- 人工介入时点
- 轮次结构
- 系统边界定义
- 冻结与回退条件

都必须在同一轮同步更新：

1. 当前执行目录下的局部文档
2. 对应全局状态载体
3. 必要时上升到开工前总规则、仓库级补充规则或仓库级永久规则

不允许只在聊天里改口，或只改局部目录不改全局状态载体。

### 14. 最短提示原则

提示词和聊天说明只能承担两类内容：

1. 最小入口指针
2. 临时探针与当前上下文

一旦稳定规则已经进入正式入口、执行文件、配置文件或规则文件，就不得继续：

1. 用提示词长期代替系统维修
2. 在聊天说明里重复抄写执行文件已经写死的稳定规则
3. 在后续文件里重复抄写开工前总规则文件已经列出的整套通用规则链
4. 通过“多处各写一份”制造多个并行真相源

当问题已经明确属于系统层时，例如：

- 取材层
- 证据抽取层
- 编排层
- 评分回指层
- 阶段总控层

就不得继续长期停留在“再改一下提示词 / 文案 / 口径描述”。

以后同类问题重复两次以上，必须升格为系统层维修，不得继续拿提示词微调冒充系统优化。

如果开工前总规则文件或任务启动入口已经列出了必读链，正确做法只有：

1. 先指向上游正式入口
2. 再只补该任务独有的附加文件

### 15. 不得把应写成通用说明的内容，偷懒写成只有作者自己看得懂的交接文件

如果一份文件承担的是：

1. 通用执行入口
2. 长期复用说明
3. 交接入口
4. 可被后续 Agent 反复调用的规则性说明

那它就必须写成通用型描述，而不是写成只有当前作者自己能顺着上下文看懂的私货交接文件。

以下内容如果不是这个文件的长期职责，就必须抹掉或移走：

1. 非通用名词
2. 过程描述
3. 阶段痕迹
4. 文件更新痕迹
5. 只对当前一轮聊天成立的背景解释
6. 只有写作者自己才知道前因后果的缩略口径

正确分流是：

1. 通用机制留在通用说明文件
2. 当前任务特有边界留在任务文件
3. 过程经过、修订痕迹、事故记录留在总留痕或专门复盘文件

以下做法一律禁止：

1. 懒得抽象，就把过程痕迹原样堆进交接文件
2. 把“我这轮怎么改的”伪装成后续长期说明
3. 让后续 Agent 必须先懂当前作者的上下文，才能读懂交接文件

### 16. 引入新术语、新分层或新中间层前，必须先用用户原话锁定目标

只要当前 Agent 准备引入以下任一内容：

- 新术语
- 新分层
- 新变量臂
- 新中间层
- 新资料卡结构
- 新冻结表

都必须先做一步“目标锁定”，并且只能用用户原话或与用户原话等价的人话表达：

1. 当前这轮到底要验证什么
2. 当前这轮到底要交付什么
3. 新术语或新结构和这两个目标是什么关系

在这一步没写清前，禁止：

1. 先发明术语再推进执行
2. 先按自造分层派工，再回头解释为什么这么分
3. 把自己的内部分析语言直接抬成正式口径

如果做完这一步后发现：不用新术语也能直接描述目标，那默认不准引入新术语。

### 17. 出现下列自检触发器时，必须先停下检查是否正在制造伪前置或口径污染

只要出现下面任意一条，就必须先停 `1` 步，检查自己是不是已经偏离用户目标：

1. 准备把一个新中间层写成“前置条件”或“阻塞项”
2. 准备把一个复杂结构写成“默认主路径”
3. 准备开多个子 session，但主产物仍然没有冻结
4. 聊天说明和 prompt 的长度明显开始大于正式文件的稳定内容
5. 自己解释方案的时间，已经超过直接落正式产物的时间

自检时必须只问两件事：

1. 这个东西是不是用户当前明确要的正式产物
2. 这个东西是不是阻塞该正式产物的最小前置产物

只要两问里有任意一问回答不出“是”，就默认先怀疑它是伪前置、伪工作量或口径污染。

### 18. 用户连续两次以上重述目标、纠正方向或重定义边界时，必须冻结

冻结触发条件不只包括“用户指出口径不实”，还包括下面任意一种情况连续出现 `2` 次以上：

1. 用户重述目标
2. 用户纠正方向
3. 用户重新定义任务边界
4. 用户重新说明什么才算真正交付

一旦触发，当前 Agent 必须立刻：

1. 停止新增执行
2. 停止新增术语
3. 停止新增 prompt 设计
4. 只列原始证据、当前真实目标、当前最小 blocker

未完成这 `4` 步前，不得继续扩派工面、扩文档面、扩概念面。

### 19. 只准盯两类产物；除这两类外，默认优先怀疑是伪工作量

当前任务只允许长期盯两类产物：

1. 用户当前明确要的正式产物
2. 真正阻塞该正式产物的最小前置产物

除这两类外，下面这些默认优先按伪工作量处理：

1. 只方便 Agent 自己审阅的中间层
2. 只能证明“做过事”，不能直接被用户消费的过程稿
3. 可以继续派工、但不能直接推进主目标的复杂结构
4. 只是让说明更长、但不能让正式产物更近的 prompt / 清单 / 对照表

如果一个东西已经连续两轮都只是“解释它为什么重要”，而不是“它已经直接推进主目标”，就必须停掉或降级。

### 20. 子 session 不准只交审计稿、过程稿或 `ok / not ok`

以后凡是派子 session，交付标准默认固定为：

1. 优先交可直接使用的正式产物
2. 其次才是原始证据
3. 最后才允许交审计结论

以下内容单独出现时，一律不算合格交付：

1. `ok / not ok`
2. `status`
3. 审计稿
4. 过程稿
5. 只解释为什么不行、不解释原始资料到底讲了什么的文件

如果子 session 发现当前件不合格，正确动作不是停在“不可用判断”，而是：

1. 在允许范围内继续补成可直接使用的正式产物
2. 或直接给出阻塞该正式产物的原始证据

不允许只交“可审计但不可直接用”的中间件，然后把后续收口继续外包给主 session 或用户。

## 最小执行口径

以后遇到同类任务，至少按下面顺序做：

1. 先复用当前仓 `.env` 与 `.venv`
2. 先跑脚本最小冒烟
3. 先启本地后端并打 `/health`
4. 先拿第一轮 baseline
5. 再谈优化回路
6. 再写未闭项，而且只能写真实代码或外部资源阻塞
